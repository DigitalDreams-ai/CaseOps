# Solution Plan — HEAL-33098

## 1. Problem Statement

SMS Magic managed package licenses have expired (or were never provisioned) in the lower Salesforce environments: DEV, QA, and UAT. Engineering and QA users in these orgs cannot access SMS Magic functionality, blocking QA sign-off on any release that involves SMS workflows. The ticket has been unworked for ~30 days. Primary stakeholder is Ona Brazwell; reporter and engineering/QA team are fully blocked.

---

## 2. Escalation Decision

**Support-Resolvable.**

This is a license provisioning and configuration issue, not a code or metadata defect. No Salesforce metadata changes are required. The fix path is:
1. Vendor coordination (SMS Magic admin portal) to renew or provision sandbox org licenses.
2. Salesforce Setup admin action to assign per-user licenses to affected users.

Engineering escalation is not warranted unless the vendor confirms that sandbox licensing requires a paid add-on that must go through a procurement/budget approval process, in which case the blocker becomes a business/procurement issue — not an engineering one.

---

## 3. Proposed Fix (Minimal Viable)

### Step 1 — Audit current license state (no changes yet)
- Log into the SMS Magic admin portal for each lower env org (DEV, QA, UAT).
- Confirm whether the org-level trial or sandbox license is expired, inactive, or simply unprovisioned.
- In Salesforce Setup > Installed Packages > SMS Magic > Manage Licenses, confirm how many licenses are available vs. assigned.
- Document findings before taking any action.

### Step 2 — Renew or provision the org-level SMS Magic license
- If trial has expired: contact SMS Magic vendor/account rep and request sandbox license renewal or a sandbox license add-on for the three orgs (DEV, QA, UAT).
- If licenses are available but simply unassigned at the org level: activate them via the SMS Magic admin portal.
- Priority order: UAT first (it gates pre-release validation), then QA, then DEV.

### Step 3 — Assign per-user licenses in Salesforce Setup
- Navigate to Setup > Installed Packages > SMS Magic > Manage Licenses in each org.
- Add licenses to the specific engineering and QA users who are blocked (confirm list with Ona Brazwell).
- Do NOT bulk-assign to all users — assign only to those who require access to minimize license consumption.

### Step 4 — Validate
- Proxy-login (or request affected user to test) in UAT.
- Send a test SMS via SMS Magic UI.
- Confirm send/receive integration is functional in UAT.
- Repeat smoke test in QA and DEV as needed.

### Step 5 — Update ticket
- Document license status found, actions taken, and validation results.
- Confirm resolution with Ona Brazwell and close ticket.

**Why this is the minimal viable fix:** The only thing broken is license access. No Salesforce metadata (fields, flows, validation rules, layouts, code) is involved. The smallest possible intervention is: restore the license at the org level, assign it to the affected users, verify it works. Nothing else is required.

## 5. Dependencies

- **Vendor availability:** SMS Magic account rep or support must be reachable to renew/provision sandbox licenses if the org-level license is expired. If sandbox licensing is a paid add-on, procurement approval may be required — this is the only dependency that could extend the timeline.
- **User list confirmation:** Ona Brazwell or the reporter must confirm which specific users need licenses assigned (to avoid over-provisioning).
- **Admin access:** Sean (or another Salesforce admin) must have Setup access in DEV, QA, and UAT orgs to perform the license assignment step.
- **SMS Magic admin portal credentials:** Must be available for all three sandbox orgs — confirm these are not tied to the expired trial account.
- **No Production dependency:** This issue is scoped entirely to lower environments. Production is not affected and no Production changes are anticipated.

---

## 6. Sandbox Plan + Risk + Rollback

### Deployment Plan
This fix is entirely administrative — there is no metadata to deploy. Actions are performed directly in:
- SMS Magic admin portal (vendor-side): license renewal.
- Salesforce Setup in each lower env (DEV, QA, UAT): per-user license assignment.

Order of operations: UAT → QA → DEV (UAT is highest priority as it gates releases).

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Vendor requires paid sandbox license add-on (not self-serviceable) | Medium | High — blocks fix until procurement | Escalate to business/procurement immediately if confirmed; set expectation with Ona Brazwell |
| SMS Magic admin portal credentials unavailable for sandbox orgs | Low | Medium — delays audit step | Coordinate with original implementer or SMS Magic support to recover access |
| License assigned to wrong users, consuming limited sandbox license seats | Low | Low — easily corrected | Confirm user list with Ona Brazwell before assigning |
| UAT send/receive integration still broken after license restored (e.g., SMS Magic webhook/channel config also expired) | Low-Medium | Medium — requires additional vendor troubleshooting | Validate immediately after license restore; if broken, open separate SMS Magic support ticket |

### Rollback
There is no rollback risk for this fix. License assignment in Salesforce Setup is reversible: remove a user from the Installed Package license list at any time via Setup > Installed Packages > SMS Magic > Manage Licenses. No data is deleted or modified. Vendor-side license renewal is additive and does not affect any existing configuration.

### Justification — Why Not a Bigger Fix
Alternatives considered and rejected:
- **Replacing SMS Magic with a different SMS vendor in lower envs:** Unnecessary — the existing package is installed and configured; it just needs a valid license.
- **Rebuilding SMS workflows to bypass SMS Magic in sandbox:** Over-engineering. The goal is to unblock QA testing of existing SMS workflows, not to redesign them.
- **Provisioning a full sandbox refresh with a Production license copy:** Disproportionate effort. The issue is a missing license, not a missing configuration.

The minimal viable fix is license restoration only. Everything else in the SMS Magic configuration is presumed intact.
