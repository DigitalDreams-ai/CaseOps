# Solution Plan — HEAL-33098
## SMS Magic Lower Environment Licenses

---

## 1. Problem Statement

The SMS Magic (Converse) managed package trial license has expired in the lower Salesforce environments (DEV, QA, UAT). This causes a "trial expired" banner and prevents outbound SMS messages from being sent or delivered in those orgs. Additionally, the reporter's Salesforce user account does not have an SMS Magic user license assigned, meaning even if the org-level license were reinstated, they personally could not use the feature. Engineering and QA team members are blocked from testing SMS-dependent workflows in lower environments. UAT is the highest-priority environment as it gates release validation.

**Affected users:** Reporter (Ona Brazwell), engineering team, QA team — all users of SMS Magic in DEV, QA, and UAT orgs.

---

## 2. Escalation Decision

**Escalated — Vendor + Internal Salesforce Admin. Not resolvable by code or metadata pipeline.**

This is a third-party managed package licensing issue. The SMS Magic package is installed but its sandbox/lower-environment license tier has expired. No Apex code, Flow, custom metadata, or declarative Salesforce configuration change can activate a vendor-controlled license. Resolution requires:
- Action with the SMS Magic vendor to reactivate sandbox licenses.
- In-org admin action to assign user-level licenses once the org-level license is active.

Engineeringinvolvement is not required — this is a vendor engagement and Salesforce Setup UI task owned by the assignee (Sean Bingham).

---

## 3. Proposed Fix (Minimal Viable)

No code or metadata changes. All steps are administrative actions in Salesforce Setup and the SMS Magic vendor portal.

### Step 1 — Contact SMS Magic Support (Sean Bingham, immediate)
1. Open a support ticket at the SMS Magic support portal (https://support.smsmagic.com or via their in-app help) referencing your production org ID and requesting sandbox license activation for three orgs: DEV, QA, UAT.
2. Provide the Salesforce Org IDs for each lower environment (retrievable from Setup → Company Information → Salesforce.com Organization ID).
3. Request confirmation of sandbox API credentials or a sandbox sending number for each org — sandbox envs often require a separate API key from production.
4. **Prioritize UAT** in the request as it is blocking release regression testing.

### Step 2 — Assign SMS Magic User Licenses (Sean Bingham, once org-level license is active)
In each lower environment (DEV, QA, UAT), perform the following:
1. Navigate to **Setup → Apps → Packaging → Installed Packages**.
2. Find **SMS Magic Converse** and click **Manage Licenses**.
3. Add licenses for all engineering and QA team members who require access, including the reporter (Ona Brazwell).
4. Save.

### Step 3 — Confirm Sandbox API Credential / Sending Number Configuration
1. In each lower env, navigate to the SMS Magic admin console (typically accessible via the App Launcher → SMS Magic Admin or Converse Settings).
2. Confirm a valid sandbox API key and sending phone number are configured. If not, use the credentials provided by SMS Magic support in Step 1.
3. Save and test the connection within the SMS Magic admin panel.

### Step 4 — Validate End-to-End in UAT
1. Log in to UAT as a licensed user.
2. Send a test SMS from a Case or Contact record using SMS Magic.
3. Confirm the message is delivered (no "trial expired" banner, no send failure).
4. Have the reporter (Ona Brazwell) or a QA team member co-validate.

### Step 5 — Update and Close Ticket
1. Document the resolution (license activation date, users licensed, any vendor ticket reference number) in HEAL-33098 comments.
2. Close the ticket once UAT integration is confirmed working.

**Why this is the minimal viable fix:** The entire problem is a license expiration — there is no Salesforce configuration defect, no code bug, and no metadata gap. Reactivating the vendor license and assigning user seats is the smallest and only action that resolves the stated issue. Any other change (reconfiguring flows, modifying permission sets, deploying metadata) would be irrelevant to the root cause and would not restore SMS functionality.

## 5. Dependencies

- **Vendor response time:** Resolution depends on SMS Magic support activating the sandbox license. This is outside internal control — follow up if no response within 1 business day.
- **Budget approval:** SMS Magic may charge separately for sandbox license seats. Confirm with the budget owner before the vendor invoices. Do not let this block the UAT license — escalate internally if approval is slow.
- **Org IDs:** Sean Bingham must collect the Salesforce Org IDs for DEV, QA, and UAT before contacting SMS Magic support (required for their provisioning process).
- **Sandbox API key:** If SMS Magic provisions a separate sandbox API key, it must be entered into the SMS Magic admin console in each lower env before end-to-end testing will pass. This is a configuration step, not a metadata deployment.
- **User list:** A list of engineering and QA team members who need SMS Magic access in lower envs must be compiled before Step 2 (user license assignment). Coordinate with the team lead.

## 6. Sandbox Plan + Risk + Rollback

### How to Deploy to Sandbox
Not applicable — there is no metadata deployment. All actions are:
- External (vendor portal / support ticket).
- In-org Setup UI actions (Manage Licenses, SMS Magic admin console).

No CI/CD pipeline, no changeset, no SFDX deploy command is involved.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SMS Magic charges extra for sandbox seats — budget not pre-approved | Medium | Medium — delays activation | Get budget pre-approval before vendor call; frame as regression testing blocker |
| Vendor takes >2 business days to respond | Medium | High — blocks UAT release validation | Escalate to SMS Magic account manager if support ticket is not acknowledged within 24 hours |
| Sandbox API key is different from production — messages fail after license activation | Medium | Medium — UAT still broken after license fix | Request sandbox-specific API credentials explicitly in the vendor support ticket (Step 1) |
| User license count is insufficient — not enough seats for full team | Low | Low — affects only excess users | Request enough seats for all expected users upfront; verify seat count before assigning |
| DEV environment has no requirement for full integration — effort wasted | Low | Low | Clarify with reporter: Step 3/4 validation only required in UAT per reporter's stated minimum requirement |

### Rollback Steps
Not applicable — no metadata was deployed, so there is nothing to roll back. If the vendor activates a license and it causes unexpected behavior (extremely unlikely), the SMS Magic managed package settings can be reverted within the SMS Magic admin console. No Salesforce metadata would be affected.

### Recommended Priority Order
1. **UAT** — activate first; blocks release regression testing.
2. **QA** — activate second; needed for feature testing cycles.
3. **DEV** — activate last; clarify with reporter whether full integration or UI-only access is sufficient before spending license seats on DEV.
