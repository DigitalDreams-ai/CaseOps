# Solution Plan — HEAL-33505

## 1. Problem Statement

The reporter is asking whether Salesforce **Opportunity Field History** can be configured to track edits and updates to the **"Order Notes"** field on the Opportunity object. Currently, changes to this field are not captured in the Opportunity Field History related list, meaning there is no audit trail of who changed the field, when, or what the previous value was.

**Who is affected:** Sales/ops users who need to audit changes to Order Notes on Opportunity records. Reporter-provided example record: `https://10xhealth.lightning.force.com/lightning/r/Opportunity/006Ql00000dY7TdIAK/view`

**What is broken:** No history tracking exists for Order Notes. This is almost certainly a configuration gap — Field History Tracking has simply not been enabled for the field. However, if Order Notes is a **Long Text Area** field type, it is a **platform limitation**: Salesforce natively does not support Field History Tracking for Long Text Area fields.

**Critical unknown:** The field type of "Order Notes" on the Opportunity object has not yet been confirmed in Production.

---

## 2. Escalation Decision

**Decision: Conditional — Support-Resolvable OR Escalate to Engineering depending on field type.**

| Scenario | Field Type | Path | Owner |
|---|---|---|---|
| A | Text, Picklist, Number, Date, or other trackable type | Enable native Field History Tracking in Setup | Support (5-minute config change) |
| B | Long Text Area | Native tracking not supported by Salesforce platform | Escalate to Engineering for custom solution |

**Why this split:** Salesforce explicitly excludes Long Text Area fields from the Field History Tracking feature. Given that the field is named "Order Notes," it is likely a Long Text Area, but this must be confirmed before any action is taken. Do not assume.

**Required operator action before proceeding:** Log into Production → Setup → Object Manager → Opportunity → Fields & Relationships → locate "Order Notes" → confirm the **Field Type** column. Also confirm the current count of fields already enabled for Field History Tracking on Opportunity (limit is 20 fields).

---

## 3. Proposed Fix

### Scenario A — Field Type IS trackable (Text, Picklist, etc.)

**This is the minimal viable fix: a single Setup toggle. No code, no deployment pipeline, no Sandbox required for a metadata-only field history enable.**

**Steps:**
1. Log into Production Salesforce as an administrator.
2. Navigate to **Setup → Object Manager → Opportunity → Fields & Relationships**.
3. Confirm "Order Notes" field type is a trackable type (not Long Text Area).
4. Navigate to **Setup → Object Manager → Opportunity → Field History Tracking**.
5. Ensure **"Enable Opportunity Field History"** is toggled ON (it likely already is).
6. Check the checkbox next to **"Order Notes"** in the field list.
7. Click **Save**.
8. Navigate to the reporter's example Opportunity record and confirm the **Opportunity Field History** related list is visible.
9. Make a test edit to Order Notes and confirm a new history entry is captured.
10. Inform the reporter: history tracking is now active **going forward only** — no retroactive history will be populated for changes made before this enable date.

**Why this is the minimal viable fix:** The ask is purely "can history be captured for this field." The answer is yes, via a native checkbox in Setup. No flow, trigger, custom object, or code is needed. Enabling a single field for tracking is the smallest possible change that fully satisfies the requirement.

---

### Scenario B — Field Type is Long Text Area

**Escalate to Engineering.** Support cannot resolve this via configuration.

**Engineering handoff summary:**
- **Problem:** Salesforce native Field History Tracking does not support Long Text Area fields. Order Notes on Opportunity is a Long Text Area and cannot be tracked natively.
- **Proposed approach (minimal viable):** Implement an **Apex trigger** on the Opportunity object (before/after update) that compares the old and new values of Order Notes and writes a record to a **custom history object** (e.g., `Opportunity_Notes_History__c`) storing: Opportunity lookup, changed date/time, changed-by user, old value, new value.
- **Why trigger over Flow:** Flow cannot reliably capture "old value" of a Long Text Area field for comparison in all scenarios. An Apex trigger with `Trigger.oldMap` is the standard pattern for this use case.
- **Why custom object over Big Objects:** Big Objects add architectural complexity and are immutable (no deletes). A custom object is simpler, queryable, and sufficient for this audit use case at typical Opportunity volumes.
- **Why not redesign around a different field type:** Changing Order Notes from Long Text Area to a shorter field type would break existing data entry UX and likely truncate existing values. That is a disproportionate change relative to the ask.
- **Minimal scope:** One trigger, one custom object, one related list added to Opportunity page layout. No approval process, no validation rules, no permission set changes beyond read access to the new history object.

## 5. Dependencies

### Scenario A
- **20-field limit:** Salesforce allows a maximum of 20 fields tracked per object. Confirm the current count on Opportunity before enabling. If at limit, a currently-tracked field of lower business value must be removed first — this requires stakeholder sign-off before removing any existing tracked field.
- **No retroactive history:** Reporter must be informed that only changes made after the enable date will appear in Field History. This is a Salesforce platform constraint, not a product decision.
- **Related list visibility:** Opportunity Field History related list must be present on the relevant Opportunity page layouts for users to see the data. Confirm this is already on the layout; if not, add it as a secondary step.

### Scenario B
- **Sandbox build and deploy:** Custom object and Apex trigger must be built in Sandbox, tested, and deployed via changeset or CI/CD pipeline. Cannot be done directly in Production.
- **FLS and permissions:** Users who need to view history must have at least Read access to the new custom object and its fields. Profile or Permission Set update required.
- **Trigger test coverage:** Apex trigger requires minimum 75% code coverage for deployment. A test class must be written.
- **No retroactive history:** Same constraint as Scenario A — only captures changes after deployment.
- **Governor limits:** At very high Opportunity update volumes, trigger must be bulkified. Standard practice, but must be confirmed in implementation.

## 6. Sandbox Plan + Risk + Rollback

### Scenario A — Sandbox Plan

**Sandbox is not strictly required** for enabling native Field History Tracking (it is a metadata toggle with no code risk). However, best practice is:
1. Enable Field History Tracking for Order Notes in **Sandbox** first.
2. Make a test edit to an Opportunity Order Notes field in Sandbox.
3. Confirm history entry appears in the Opportunity Field History related list.
4. Promote the same change to **Production** once validated.

**Risk:**
- **Low.** Enabling Field History Tracking is a read-only audit feature. It does not change field behavior, layout, or any automation. The only risk is consuming one of the 20 tracked-field slots.
- If Opportunity is already at the 20-field limit, enabling Order Notes requires disabling another field — that is the only meaningful risk, and it requires stakeholder input.

**Rollback:**
- Navigate back to Setup → Object Manager → Opportunity → Field History Tracking → uncheck Order Notes → Save.
- Takes effect immediately. No data loss — existing history entries already written are retained in the OpportunityFieldHistory object but no new entries will be created.
- Rollback is instantaneous and risk-free.

---

### Scenario B — Sandbox Plan

**Sandbox is required.** Custom Apex and custom objects must never be created directly in Production.

1. **Build in Sandbox:**
   - Create `Opportunity_Notes_History__c` custom object with all fields listed in Section 4.
   - Write `OpportunityOrderNotesHistoryTrigger` Apex trigger (after update, bulkified, comparing `Trigger.oldMap` vs `Trigger.newMap` for Order Notes field).
   - Write Apex test class with coverage of insert, update (with and without Order Notes change), and bulk scenarios.
   - Add related list to Opportunity page layout in Sandbox.
2. **Test in Sandbox:**
   - Edit Order Notes on a test Opportunity → confirm history record created.
   - Edit other Opportunity fields without changing Order Notes → confirm no spurious history records created.
   - Bulk update test (200+ records) → confirm no governor limit errors.
3. **Deploy to Production:**
   - Package custom object, fields, trigger, test class, and layout change in a changeset.
   - Deploy via standard change management process.
   - Validate in Production with reporter's example record.
4. **Inform reporter** of go-live date and confirm history is now being captured.

**Risk:**
- **Medium.** Apex triggers introduce code complexity. Poorly written triggers can cause DML failures on Opportunity updates if the trigger itself throws an uncaught exception — this would block all Opportunity saves. Proper exception handling must be implemented.
- Custom object storage consumes org data storage. At typical Opportunity update volumes this is negligible, but should be noted.
- Deployment requires code coverage threshold to be met across the entire org's Apex — if the org is near coverage thresholds, a new test class may be needed for other components (unlikely but possible).

**Rollback:**
- **Trigger:** Deactivate the Apex trigger in Production Setup → Apex Triggers → deactivate `OpportunityOrderNotesHistoryTrigger`. Immediate effect. No Opportunity behavior is affected.
- **Custom object:** Can be deleted from Setup if no other components depend on it (and if data storage needs to be reclaimed). History records already written are deleted with the object.
- **Layout:** Remove the related list from Opportunity page layout. No data impact.
- Full rollback is clean and reversible with no downstream risk to existing Opportunity data or automation.

---

## Minimal Viable Fix Justification

**Scenario A:** We are not proposing a custom trigger, custom object, Flow, or any code change. The reporter asked if history tracking *can* be enabled — if the field type supports it, the answer is a single checkbox in Setup. Any additional engineering would be over-engineering a solved platform problem.

**Scenario B:** We are not proposing a full audit framework, Big Objects, platform events, or a third-party change data capture tool. The requirement is narrow: capture who changed Order Notes, when, and what the value was. A single Apex trigger writing to a single custom object satisfies this completely. We reuse Salesforce's existing related list pattern rather than building a custom Lightning component, because the standard related list on a custom object provides the same display functionality at zero additional development cost.
