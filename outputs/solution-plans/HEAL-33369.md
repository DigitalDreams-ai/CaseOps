# Solution Plan — HEAL-33369

## 1. Problem Statement

Mya's **Time to Schedule (TTS)** reporting relies on three Salesforce reports built on the **Opportunity Field History** report type. This report type is non-joinable and cannot reference parent Account fields directly. Two fields needed for segmentation — **Billing State** and **Main Practitioner** — exist on the Opportunity object but are blank because no automation currently copies them from the parent Account. As a result, Mya cannot filter or group TTS data by geography or practitioner within Salesforce, and has been forced to maintain a fragile Excel cross-reference with EHR data as a workaround.

**Affected users:** Mya (report consumer). Potentially any ops or analytics user relying on the same three reports:
- Lab Review (`00OQl000006nwH7MAI`)
- Genetic Test Clarity Calls (`00OQl000006nyxFMAQ`)
- Peptide Express Consultations (`00OQl000006nz0TMAQ`)

---

## 2. Escalation Decision

**Support-Resolvable** — Confidence: Medium.

The fix is a Salesforce configuration task: populate existing Opportunity-level mirror fields from Account using a Flow, then add those fields as columns in three existing reports. No Apex code, managed package changes, or data model alterations are required. Engineering is not involved.

Confidence is medium rather than high because Sean identified "problematic existing processes" during investigation. These must be documented and resolved before deploying new automation to avoid trigger loops or data overwrites. If the conflicting processes turn out to be locked managed-package automation or involve cross-org integrations, the escalation decision will be revisited.

---

## 3. Proposed Fix (Minimal Viable)

### Why this fix and not a bigger one

- **We are NOT changing the report type.** Switching from Opportunity Field History to a joined or custom report type would require rebuilding all three reports from scratch, re-validating all existing columns, and risking loss of field history granularity. The mirror-field approach preserves the existing report type and adds only two columns.
- **We are NOT creating new fields.** Billing State and Main Practitioner fields already exist on the Opportunity object. Creating duplicates would introduce confusion and maintenance overhead.
- **We are NOT writing a batch job to backfill history.** Historical TTS data is acknowledged as out of scope for this fix. The Excel workaround covers past periods; the Flow covers net-new and updated records going forward.
- **We are using a single Record-Triggered Flow** rather than a Process Builder (legacy) or Apex trigger because it is the current Salesforce-recommended, low-code approach and is the least likely to conflict with other automation if scoped correctly.

### Step-by-Step Fix

**Phase 1 — Audit Existing Processes (pre-requisite)**

1. In Sandbox, open Setup > Flows. Filter by Object = Opportunity. Document every active Flow that references `BillingState` (Account) or the Opportunity mirror fields for Billing State and Main Practitioner.
2. Open Setup > Process Builder. Repeat the search.
3. For each conflicting process: determine whether it writes to the same Opportunity fields. If yes, either deactivate the conflicting process (if stale/redundant) or add an entry condition to the new Flow to avoid collision.
4. Document findings before proceeding. Do not deploy the new Flow until conflicts are resolved.

**Phase 2 — Build Record-Triggered Flow in Sandbox**

Flow name: `Opportunity_Populate_Account_Mirror_Fields`

- **Object:** Opportunity
- **Trigger:** A record is created or updated
- **Entry condition:** Run only when `Billing_State__c` (Opportunity) is blank OR `Account.BillingState` has changed OR `Main_Practitioner__c` (Opportunity) is blank OR the related Account's Main Practitioner field has changed. (Exact field API names to be confirmed against org schema.)
- **Optimization:** Run in fast-field-update mode if both fields are on the Opportunity itself and no related-record writes are needed; otherwise use "Actions and Related Records" mode.
- **Action — Get Records:** Query the parent Account (`AccountId`) to retrieve `BillingState` and the Account-level Main Practitioner field API name.
- **Action — Update Records:** Write Account `BillingState` → Opportunity `Billing_State__c`; write Account Main Practitioner → Opportunity `Main_Practitioner__c`.
- **Fault path:** Log fault to a custom object or send email alert to SF Admin so silent failures are visible.
- **Save as inactive first.** Activate only after sandbox validation.

**Phase 3 — Validate in Sandbox**

1. Create a test Opportunity linked to an Account with known Billing State and Main Practitioner values.
2. Confirm Opportunity mirror fields populate on save.
3. Update the Account Billing State; confirm the Opportunity field updates on next Opportunity save (or immediately if trigger is set to fire on Account change — confirm desired behavior with Mya).
4. Cross-reference output against Mya's Excel reference file (provided 2026-05-07) to confirm field values match expected format.
5. Confirm no duplicate-write errors or loop conditions from previously identified conflicting processes.

**Phase 4 — Deploy Flow to Production**

1. Export Flow from Sandbox via change set or Salesforce CLI (`sf project deploy start`).
2. Deploy to Production during low-traffic window.
3. Activate Flow in Production.
4. Spot-check 3–5 existing Opportunities: open a record, make a minor edit (e.g., description), save, and confirm mirror fields populate.

**Phase 5 — Update the Three Reports**

All report edits can be made directly in Production Reports (no deployment needed):

1. Open Lab Review report (`00OQl000006nwH7MAI`) in Edit mode.
2. Add columns: `Billing_State__c` and `Main_Practitioner__c` from the Opportunity object.
3. Position columns consistently with Mya's Excel reference layout.
4. Save and run. Validate with Mya.
5. Repeat for Genetic Test Clarity Calls (`00OQl000006nyxFMAQ`) and Peptide Express Consultations (`00OQl000006nz0TMAQ`).
6. Use identical column API names and placement across all three reports to ensure consistency.

## 5. Dependencies

- **Existing processes must be audited and conflicts resolved first.** Deploying the new Flow before resolving conflicts risks data corruption or infinite loops.
- **Field API names must be confirmed.** The exact API names for the Opportunity mirror fields and the Account Main Practitioner field must be verified in the org schema before building the Flow. Do not assume from display labels.
- **FLS must permit Flow writes.** The running user context for the Flow (System Context vs. User Context) must have write access to both Opportunity mirror fields. Confirm FLS for relevant profiles if running in User Context.
- **Account Billing State must be populated on the parent Account.** If the Account record itself has a blank Billing State, the Flow will write a blank value — this is correct behavior, but Mya should be aware that Account data quality affects report output.
- **Backfill is out of scope.** Historical Opportunities will not auto-populate. If Mya needs historical data in Salesforce, a separate one-time data load (Data Loader) would be required — this is a separate work item and is not part of this fix.
- **Report FLS.** The two Opportunity mirror fields must be readable by Mya's profile/permission set. Verify before adding to reports.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan

1. Target sandbox: Full or Partial sandbox with representative Opportunity and Account data.
2. Build Flow in Sandbox as described in Phase 2 above.
3. Execute Phase 3 validation steps.
4. If validation passes, create a change set or use Salesforce CLI to move Flow metadata to Production.
5. Report changes (Phase 5) are made directly in Production — no sandbox step needed for reports.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Conflicting existing automation causes field overwrite loop | Medium | High | Audit all existing Flows/PBs before activating new Flow; add entry conditions to prevent re-firing |
| Flow writes blank values because Account fields are empty | Medium | Low | Expected behavior; document for Mya; Account data quality is a separate concern |
| Incorrect field API names used in Flow | Low | Medium | Confirm API names in Schema Builder before building Flow |
| Mirror fields not visible to Mya's profile in reports | Low | Low | Check FLS for Mya's profile before adding columns to reports |
| Report column additions break existing report subscriptions or dashboards | Low | Low | Check if any dashboards reference these reports before editing; note existing column order |
| Historical data remains blank in reports | High (certain) | Low | Acknowledged and accepted; Excel covers historical periods; communicated to Mya |

### Rollback Steps

**If the Flow causes unintended data writes or errors:**
1. Deactivate `Opportunity_Populate_Account_Mirror_Fields` Flow immediately in Production (Setup > Flows > Deactivate). This stops all further writes within seconds.
2. No field definition was changed, so no schema rollback is needed.
3. If mirror fields were incorrectly overwritten on Opportunity records: use Data Loader to restore previous values from a pre-deployment data export. (Export `Billing_State__c` and `Main_Practitioner__c` from Opportunities before activating the Flow as a backup snapshot.)

**If report column additions cause display issues:**
1. Open the affected report in Edit mode.
2. Remove the two added columns.
3. Save. No data is affected by report column changes.

**Recovery time estimate:** Flow deactivation is immediate. Data restore from export snapshot is 30–60 minutes depending on record volume.
