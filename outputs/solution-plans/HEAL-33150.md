# Solution Plan — HEAL-33150

## 1. Problem Statement

Inbound email replies from external mail clients (e.g., Outlook) are not being logged to the originating Salesforce Case for the **Cx Case Record type**. When an agent sends an outbound email from a Cx Case, the reply from the customer does not attach to the case activity feed. The thread ID token (`{!Case.Thread_Id}`) was confirmed present in the outbound email, meaning the break is on the **inbound routing side**, not the outbound template side. Affected scope: multiple teams and departments — this is an ongoing data integrity risk, as replies may be silently discarded or generating orphaned cases.

---

## 2. Escalation Decision

**Escalated to Engineering.**

Reason: The fix requires direct inspection and modification of Email-to-Case routing address configuration, email template thread token verification, and potentially Apex/Flow email handler review — all of which require Production org access and scope beyond Support sandbox resolution. The defect is isolated to the Cx record type, but the routing infrastructure is shared, so changes must be scoped carefully to avoid regressions on other record types.

---

## 3. Proposed Fix (Minimal Viable)

Engineering should perform the following **audit-first, change-minimal** investigation and fix sequence:

### Step 1 — Audit Email-to-Case Routing Addresses (Production, read-only)
- Navigate to: **Setup → Email-to-Case → Routing Addresses**
- Identify the routing address(es) associated with the Cx Case Record type
- Confirm:
  - A routing address exists and is **Active**
  - The routing address is mapped to the **Cx record type** (check "Default Record Type" or routing rule assignment)
  - The **"Thread ID" matching** option is enabled (not "From Address" or "None")
  - The routing email address is not shared with another record type in a conflicting way

### Step 2 — Verify Outbound Email Template Thread Token
- Identify the email template assigned to Cx Case outbound emails
- Confirm `{!Case.Thread_Id}` is present in the template body (typically in the footer as `ref:{!Case.Thread_Id}:ref`)
- If missing: add the merge field to the template footer — this is the **smallest possible template change**
- Note: reporter confirmed thread ID is present in outbound, so this step is verification only and likely a no-op

### Step 3 — Apply Minimal Fix Based on Audit Findings

Expected finding is one of three scenarios — fix only the identified gap:

| Finding | Minimal Fix |
|---|---|
| No routing address exists for Cx record type | Create a new Email-to-Case routing address, assign to Cx record type, set matching to "Thread ID" |
| Routing address exists but Thread ID matching is off | Edit routing address: set "Email Matching" to "Thread ID" |
| Routing address exists and is correct but inactive | Set routing address to Active |

**Do not** restructure routing for other record types. **Do not** modify shared email handlers unless the audit confirms they are the defect source.

### Step 4 — Confirm No Apex/Flow Handler Interference (if Steps 1–3 yield no finding)
- Search for `OnEmailMessage` Apex triggers or Flows on the `EmailMessage` object
- Check for any record-type-specific filtering logic that may be discarding inbound messages for Cx cases
- This step is only necessary if routing address configuration is confirmed correct

---

## 4. Affected Components

| Component | Type | Change Expected |
|---|---|---|
| Email-to-Case Routing Address (Cx) | Setup Configuration | Likely: activate, correct matching mode, or create |
| Cx Case email template | Email Template (Classic or Lightning) | Verify only — add `{!Case.Thread_Id}` if missing |
| EmailMessage object | Standard Object | No change expected |
| Case Activity Feed | Standard Feature | No change — resolves automatically once routing is fixed |
| OnEmailMessage Apex Trigger / Flow | Apex / Flow | Inspect only; change only if confirmed as defect source |
| Cx Record Type | Record Type | No change expected — record type itself is not the defect |

---

## 5. Dependencies

- **Email-to-Case must be enabled** in the org — assumed true since other record types work
- **Routing address inbox** (the email address configured in the routing address) must be reachable and forwarding correctly to Salesforce — Engineering should confirm the mailbox is live and not bouncing
- **Thread ID format** in the outbound template must match the format Email-to-Case expects (`ref:{!Case.Thread_Id}:ref`) — stripping or reformatting by an email relay would break matching even with correct routing config
- If a **new routing address is created**, the corresponding email inbox must be provisioned and forwarding rules configured before activation
- **No changes to shared routing addresses** that serve other record types — scope any fix to Cx-specific configuration only

---

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan
1. Replicate the Cx routing address configuration in a **full Sandbox** (or Partial if full unavailable)
2. Send a test outbound email from a Cx Case in Sandbox
3. Reply from an external email client (e.g., Gmail or Outlook test account) to the routing address
4. Confirm reply appears in Case Activity Feed on the originating Cx Case
5. Confirm no new/orphaned case is created by the reply
6. If Apex/Flow changes are required: deploy via change set or SFDX to Sandbox first, run regression tests on EmailMessage triggers for all record types
7. Only after Sandbox validation: deploy to Production

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Fix to routing address affects other record types sharing the same address | Medium | High | Audit sharing before any change; create Cx-specific address if needed |
| New routing address inbox not provisioned correctly — replies still lost | Low | High | Verify mailbox and forwarding in Sandbox before Production |
| Apex/Flow change introduces regression on other EmailMessage handling | Low | Medium | Scope change narrowly; run full EmailMessage trigger regression in Sandbox |
| Thread token stripped by customer email relay — fix does not resolve issue | Low | Medium | If confirmed, document as external dependency outside Salesforce control |

### Rollback Steps

- **Routing address change:** Revert matching mode setting in Setup → Email-to-Case → Routing Addresses (no deployment required — UI change only)
- **Routing address creation:** Deactivate the new routing address (do not delete — preserves audit trail)
- **Email template change:** Restore prior template version from Salesforce version history or re-remove the `{!Case.Thread_Id}` merge field
- **Apex/Flow change:** Redeploy prior version via change set or SFDX rollback; deactivate Flow version and reactivate prior version

---

## Minimal Viable Fix Justification

The investigation confirms the outbound thread token is present, which rules out a full template redesign. The defect is isolated to the **inbound routing path for the Cx record type only** — other record types work. Therefore:

- We are **not** redesigning the Email-to-Case architecture
- We are **not** modifying templates beyond verifying the thread token footer
- We are **not** touching Apex or Flow handlers unless the routing address audit yields no finding
- The fix is: **correct or create the Cx routing address with Thread ID matching enabled** — a single Setup configuration change requiring no code, no deployment package, and no structural changes to the org

This is the smallest possible change that closes the inbound reply loop for Cx cases without touching anything that serves other record types.
