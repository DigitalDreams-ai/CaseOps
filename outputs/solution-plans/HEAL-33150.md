# Solution Plan — HEAL-33150: Cx Case Record Email Reply Not Captured

---

## 1. Problem Statement

When a Salesforce agent sends an email from a **Cx Case Record** (Cases object, 10X Health Production org), the outbound email is delivered successfully and includes the Email-to-Case thread token. However, when the recipient replies from an external mail client (Outlook confirmed), the reply is **not ingested back into Salesforce** — no EmailMessage record is created, and the case Activity timeline shows only the original outbound message.

**Who is affected:** Customer Support department, multiple teams, large numbers of records (per reporter Noah Scott). The specific observed example involves a user named Tyler replying to a case email regarding a refund approval.

**Impact:** Incomplete communication paper trail on Cx Case Records. Agents cannot see reply context, causing fragmented case history and potential customer service failures.

---

## 2. Escalation Decision

**Escalated to Engineering.**

This is NOT support-resolvable without direct Production org access because:

- The root cause requires inspecting live **Email-to-Case routing address configuration** in Salesforce Setup — specifically whether the Cx Case Record type has an active routing address and whether the `Reply-To` header on outbound emails resolves to that address.
- Confirming the `Reply-To` header requires either sending a live test email and inspecting raw SMTP headers, or reading Setup records that are not accessible via metadata export alone.
- The fix may involve **SMTP forwarding configuration** at the email server level (if On-Demand Email-to-Case is in use), which is outside Salesforce declarative configuration.
- Broad scope (multiple teams, large record volume) means a misconfigured fix could break case routing org-wide — Engineering must validate before touching Production.

---

## 3. Proposed Fix (Minimal Viable)

### Why this fix, not a bigger one

The thread token is already present in outbound emails (confirmed by reporter). This eliminates the need to rebuild the email action, redesign the case page layout, or modify EmailMessage triggers. The token generation is working. The **only broken link is reply routing** — replies are not landing in a Salesforce-monitored mailbox. Therefore the minimal fix is: **ensure an active Email-to-Case routing address exists for the Cx Case Record type and that the outbound email `Reply-To` header points to it.**

We do NOT need to:
- Modify the EmailMessage object or FLS
- Change Case record type definitions
- Create new flows or Apex
- Redesign the Send Email quick action (unless inspection reveals a custom action is overriding the Reply-To)

### Engineering Investigation Steps (ordered by priority)

**Step 1 — Audit Email-to-Case routing addresses (read-only, Production)**
- Go to: Setup → Email-to-Case → Routing Addresses
- Document every routing address: name, email address, active/inactive status, assigned record type, case origin value
- Identify whether any routing address is assigned to or intended for the Cx Case Record type
- **Expected finding:** Either no routing address exists for Cx Case Record type, OR one exists but is inactive, OR one exists but is not used as the Reply-To on outbound emails

**Step 2 — Inspect outbound email Reply-To header**
- Send a test email from a Cx Case Record (Sandbox preferred, or a non-critical Production case)
- In the receiving mail client, view raw message source/headers
- Confirm: What address appears in the `Reply-To` header?
- **Expected finding:** Reply-To resolves to an OWA or user address, NOT the Email-to-Case routing address

**Step 3 — Identify a working baseline (read-only, Production)**
- Find at least one other Case record type in the org where email threading works correctly
- Document its full Email-to-Case routing address config (address, active status, record type assignment, SSL/TLS settings, case origin)
- This becomes the template for the Cx Case Record fix

**Step 4 — Apply the minimal fix (Sandbox first)**

*Scenario A (most likely): No active routing address for Cx Case Record type*
- In Sandbox Setup → Email-to-Case → Routing Addresses → New
- Configure mirroring the working baseline: set routing email address, set Status = Active, assign record type = Cx Case Record, set Case Origin appropriately
- If On-Demand Email-to-Case: configure SMTP forwarding from the routing mailbox to the Salesforce email service endpoint (requires IT/email server access)
- If standard Email-to-Case: confirm the routing address mailbox is being polled

*Scenario B: Routing address exists but Reply-To is wrong*
- Inspect the Send Email quick action (`Case.SendEmail`) on the Cx Case Record type page layout
- If a custom email quick action is overriding the Reply-To: update it to use the Email-to-Case routing address as the Reply-To
- If the standard action is used but the From/Reply-To is set to an OWA: update the Email-to-Case routing address to be the designated Reply-To, or remove the OWA override

*Scenario C: Routing address inactive*
- Setup → Email-to-Case → Routing Addresses → find the Cx Case Record address → Edit → set Active = true
- This is the absolute smallest possible fix — a single toggle

**Step 5 — Validate end-to-end in Sandbox**
- Send email from a Cx Case Record in Sandbox
- Reply from an external mail client (Outlook)
- Confirm: EmailMessage record created on the case, reply visible in Activity timeline
- Confirm no regression on other Case record types

**Step 6 — Deploy to Production**
- If fix is purely a Setup configuration toggle (activate routing address): apply directly in Production Setup — no Gearset deploy required
- If fix involves metadata changes (custom email action, page layout update): deploy via Gearset from validated Sandbox changeset
- Notify Noah Scott and Sean Bingham of completion; provide reproduction test case for UAT sign-off

## 4. Affected Components

| Component | Type | Change Expected |
|---|---|---|
| Email-to-Case Routing Addresses | Setup Configuration | Primary change target — activate or create routing address for Cx Case Record type |
| Case Record Type: Cx Case Record | Record Type | Read-only audit only — no change expected unless custom action is found |
| Case Page Layout (Cx Case Record) | Page Layout | Audit only — change only if custom Send Email action is overriding Reply-To |
| Send Email Quick Action (`Case.SendEmail` or custom) | Quick Action | Change only if a custom variant is found that overrides the Reply-To header |
| Org-Wide Email Addresses | OWA Config | Audit only — confirm whether OWA is being used as Reply-To and intercepting replies |
| EmailMessage Object | Standard Object | No change expected — system-managed; inbound processing should work once routing address is correct |
| Email Services (On-Demand E2C endpoint) | Email Service | Audit only — SMTP forwarding config may need IT involvement if On-Demand is in use |

---

## 5. Dependencies

1. **Active Email-to-Case must be enabled org-wide** — confirm Setup → Email-to-Case → Settings → Enable Email-to-Case is checked. (Almost certainly already true given other record types work, but must be verified.)
2. **Routing address mailbox must be accessible** — if creating a new routing address, the associated email inbox must exist, be reachable, and (for On-Demand) have SMTP forwarding configured to the Salesforce email service endpoint. This may require IT/infrastructure involvement outside of Salesforce Setup.
3. **No conflicting case assignment rules** — confirm that inbound emails matched to Cx Case Records are not being rerouted by assignment rules in a way that drops the thread match. Assignment rules should run after thread matching, not before.
4. **Sandbox must mirror Production Email-to-Case config** for validation to be meaningful — if Sandbox does not have equivalent routing addresses or mailbox forwarding, end-to-end testing must be done carefully or in a Full/Partial Sandbox with email service configured.
5. **Thread token preservation through Outlook reply** — if Outlook is stripping the thread token from the reply body/headers, fixing the routing address alone will not resolve the issue. Engineering should confirm the token survives the Outlook reply (inspect raw headers of Tyler's reply if available). If token stripping is occurring, additional investigation into email template format or token placement (body vs. header) is needed — but this is a secondary hypothesis, not the primary fix.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan

1. Use a Sandbox that has Email-to-Case enabled and, if possible, mirrors Production routing address configuration (Full or Partial Sandbox preferred)
2. In Sandbox Setup → Email-to-Case → Routing Addresses: apply the minimal fix identified in Step 4 above (activate existing address, or create new one mirroring working baseline)
3. If metadata changes are needed (custom quick action, page layout): export from Sandbox via Gearset, review changeset, deploy to Sandbox, validate
4. Execute end-to-end test: agent sends email from Cx Case Record → external recipient replies from Outlook → confirm EmailMessage created on case, visible in Activity timeline
5. Execute regression test on at least one other Case record type: confirm existing email threading still works
6. If all tests pass: promote to Production via Gearset (metadata changes) or apply Setup change directly (routing address config)

### Risk Assessment

| Risk | Likelihood | Severity | Mitigation |
|---|---|---|---|
| Fixing routing address causes inbound emails to trigger unwanted Case assignment rule reassignments | Medium | Medium | Review assignment rules before activating; test with a non-critical case in Sandbox |
| New routing address receives a flood of previously undelivered replies that were queued | Low | Low | Replies sent before the fix will have been discarded by Outlook or the sender's mail server; no retroactive ingestion expected |
| SMTP forwarding setup (On-Demand E2C) requires IT involvement and delays fix | Medium | Medium | Identify early whether On-Demand or standard E2C is in use; engage IT in parallel if forwarding is needed |
| Fix inadvertently affects other Case record types sharing the same routing address | Low | High | Confirm routing address scope is limited to Cx Case Record type; do not modify shared routing addresses |
| Outlook strips thread token from replies regardless of routing address fix | Low | High | Inspect raw headers of a test reply before closing; if token stripping is confirmed, escalate as a separate sub-issue |

### Rollback Steps

- **If fix was a routing address activation toggle:** Deactivate the routing address in Setup → Email-to-Case → Routing Addresses → Edit → Active = false. Immediate effect; no deploy required.
- **If fix involved creating a new routing address:** Delete the routing address record in Setup. No downstream metadata impact.
- **If fix involved metadata changes (quick action, page layout) deployed via Gearset:** Redeploy the previous version of the affected metadata components from the prior Sandbox or Production backup in Gearset. Confirm rollback by sending a test email from a Cx Case Record and verifying behavior returns to pre-fix state.
- **In all cases:** Notify Noah Scott and Sean Bingham immediately if rollback is executed; document the rollback reason in HEAL-33150.
