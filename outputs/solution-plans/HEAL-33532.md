# Solution Plan — HEAL-33532

## 1. Problem Statement

Five users at the customer org requested SMS texting access via Salesforce (SMS Magic). Two of those users — **Alejandra Molina** and **Meghan Langbehn** — could not send or receive SMS. The other three (Angelica Del Toro, Rosemary Lawler, Tiffany Ortiz) already had full access and were unaffected. Root cause: Alejandra and Meghan were missing one or more required SMS Magic permission sets in Production.

---

## 2. Escalation Decision

**Support-Resolvable — High Confidence.**

The fix is a standard permission set assignment — a routine administrative action within Support scope. No metadata changes, no code, no flows, no automation involved. The org-wide SMS Magic configuration was confirmed correct; only per-user permission assignments were missing.

---

## 3. Proposed Fix (Minimal Viable)

### Actions Already Taken (2026-05-11, Sean Bingham)

**Alejandra Molina:**
- Assigned: `SMS App Permission Set`

**Meghan Langbehn:**
- Assigned: `SMS App Permission Set`
- Assigned: `SMS Converse Conversation User`
- Assigned: `SMS Magic Additional Permissions`

### Remaining Step

1. Await confirmation from **Tyler Kaplan** that Alejandra Molina and Meghan Langbehn can successfully send and receive SMS in Salesforce.
2. If confirmed → resolve and close HEAL-33532.
3. If not confirmed → investigate whether an **SMS Magic user license** (distinct from a permission set) is missing for either user. Run a license audit in Production.

### Why This Is the Minimal Viable Fix

The three other users (Angelica, Rosemary, Tiffany) already held the full SMS access bundle and needed nothing. Rather than auditing or modifying org-wide SMS Magic configuration, profiles, or permission set groups, we assigned only the specific missing permission sets to the two affected users. This is the smallest change that closes the access gap. No sandbox work, no deployment, no broader permission redesign — just targeted permission set assignments directly in Production.

## 5. Dependencies

- **SMS Magic managed package** must already be installed and active in Production — confirmed true (three other users already have working access).
- **SMS Magic user licenses** must be available and assigned to Alejandra and Meghan. If license seats are exhausted, permission sets alone will not grant access. This is the primary residual risk.
- No profile changes are required — permission sets layer on top of existing profiles without modification.
- No connected app, named credential, or org-wide SMS setting changes are needed.

---

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

No Sandbox work is required or appropriate for this fix. Permission set assignments are:
- Non-metadata changes (they live at the user record level, not in deployable metadata).
- Environment-specific (a Sandbox assignment would not transfer to Production and would not validate anything meaningful).
- Already applied directly in Production, which is standard practice for permission set assignments.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Missing SMS Magic user license for Alejandra or Meghan | Low–Medium | Medium — texting still blocked | Run license audit; request additional seat from SMS Magic if needed |
| Permission set assignment causes unintended access | Very Low | Low — permission sets only add access, scoped to SMS Magic features | Review permission set contents if customer reports unexpected behavior |
| No risk to other users or org-wide configuration | N/A | None | No broader changes were made |

### Rollback Steps

If the permission set assignments need to be reversed (e.g., wrong user, customer request):

1. Navigate to **Production → Setup → Users**.
2. Open the affected user record (Alejandra Molina or Meghan Langbehn).
3. Scroll to **Permission Set Assignments** related list.
4. Click **Remove** next to each permission set assigned as part of this fix.
5. Confirm removal.

Rollback is immediate, non-destructive to other users, and requires no deployment.
