## 2. Escalation Decision

**Decision: Insufficient Data — Investigation Must Be Completed Before Any Fix Is Proposed**

Neither a Support-Resolvable fix nor an Engineering escalation can be justified at this time. The internal notes confirm no diagnosis has been performed. The root cause is entirely unknown and spans a wide possibility space:

- Expired or single-use consent form link (most likely, lowest effort to fix)

No hypothesis has been confirmed. Proposing a fix now would be speculative and risks wasted effort or unintended side effects.

## 3. Proposed Fix

**Phase 0 — Required Before Any Fix (Blocking)**

The following information-gathering steps must be completed first. These are zero-risk, read-only actions:

1. Open Salesforce Account 001Rh00000GWgxNIAT in Production (read-only). Locate all consent form records associated with this patient. Check: link URL, expiration date/status field, sent date, completion status.
2. Copy the exact consent form URL the patient is clicking. Open it in an incognito browser window and observe the exact error message rendered.
3. Ask Steph Silva to obtain and share a screenshot of the error the patient sees.
4. Confirm which consent tool is in use (DocuSign, Motive, native Salesforce Experience Cloud form, or other).

**Phase 1 — Most Likely Minimal Fix (Pending Confirmation from Phase 0)**

If Phase 0 confirms the link is expired or single-use and already consumed:

- **Fix:** Resend or regenerate the consent form link for this patient. This is a data/record action, not a metadata change.
- **Steps:**
  1. Locate the consent record on Account 001Rh00000GWgxNIAT.
  2. Use the existing "Resend" or "Regenerate Link" action (or equivalent button) already present in the org — do not build anything new.
  3. Confirm with the patient or reporter that the new link resolves the error.
- **Why this is minimal:** A link resend requires zero metadata changes, zero deployment, and zero sandbox work. It is a one-record operational action. If this resolves the issue, no further work is needed.

**Phase 2 — Fallback Investigation (If Phase 1 Does Not Apply)**

If the link is valid and not expired, escalate based on the error source:

- **If Salesforce Experience Cloud / guest user error:** Escalate to Engineering to inspect guest user profile permissions, sharing rules, and site configuration for the consent form page.
- **If third-party app error (DocuSign, Motive, etc.):** Escalate to the relevant app vendor or internal integration owner with the exact error text and envelope/record ID.
- **If device/network issue:** Advise patient to retry on a different device or network; no Salesforce action required.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

**For Phase 1 (link resend — most likely fix):**
No sandbox work required. A record-level resend action in Production carries no deployment risk. Validate by confirming the patient can access the new link.

**For any metadata fix (Phase 2, if required):**
- Reproduce the confirmed error in Sandbox using a test patient record and the same consent form configuration.
- Apply the minimal metadata change identified (e.g., guest user profile permission, field value, flow adjustment).
- Validate in Sandbox that a test user can complete the consent form end-to-end.
- Deploy to Production via Change Set or preferred deployment method after Sandbox validation passes.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Resending link does not fix the issue | Medium | Low — no harm done, investigation continues | Proceed to Phase 2 |
| Metadata change breaks consent flow for other patients | Low (no change proposed yet) | High | Test in Sandbox with multiple record types before deploying |

| Patient blocked from clinical workflow during investigation delay | Medium | High — patient impact | Prioritize Phase 0 same business day |

### Rollback

**For Phase 1 (link resend):** No rollback needed. Resending a link does not alter existing records destructively. The original (broken) link state is already the current state.

**For any metadata change (Phase 2):** Document the exact before-state of every component changed (screenshot or export). If the change causes regression, revert via Change Set or manual field-level undo in Sandbox first, then Production. No data migration or record updates are expected, so rollback scope is limited to configuration only.

## Minimal Viable Fix Justification

The single most important reason no larger fix is proposed: **the root cause is unconfirmed.** Building or changing anything before Phase 0 is complete would be over-engineering a problem we have not yet defined. The most statistically common cause of "patient can't complete consent form" in SMS-delivered consent workflows is an expired or already-used single-use link — a problem solved by a one-click resend with zero metadata impact. Every alternative (rebuilding the consent flow, modifying Experience Cloud permissions, reconfiguring a third-party integration) is larger, riskier, and not yet indicated by evidence. We do the smallest possible thing first: look at the record, try the resend, confirm with the patient. Only if that fails do we escalate scope.
