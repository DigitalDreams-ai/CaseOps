# Solution Plan — HEAL-32826
## PAM Informed Consent Cases for Shopify Orders

---

## 1. Problem Statement

During the window **3/18–3/27**, hundreds of PAM (Program Activation Manager) cases created from Shopify MGT/GHT orders were not assigned Informed Consent 2026 records in Salesforce. The automation responsible for consent assignment (a Salesforce flow or trigger fired on account/opportunity creation from the Shopify integration) silently failed. A fix was deployed on 3/27 that appears to have restored forward-looking consent assignment — new orders post-3/27 now have Informed Consent 2026 on the SF account. However, the ~9-day backfill window remains unaddressed. Affected patients are missing required consent records, creating compliance and regulatory exposure. The Shopify→SF→Chrono sync chain and the safety of the "Assign patient agreements" bulk remediation action are unverified.

**Who is affected:** Patients who placed Shopify MGT/GHT orders between 3/18 and 3/27 whose SF accounts lack an Informed Consent 2026 record. Scope: hundreds of records.

---

## 2. Escalation Decision

**Escalated to Engineering.** Support cannot safely resolve this.

**Why:**
- Scope is hundreds of records across a 9-day window — exceeds manual Support remediation threshold.
- Root cause spans multiple systems: Shopify order creation → SF account/opportunity automation → Chrono sync. Support does not have visibility or authority across all three.
- The remediation action ("Assign patient agreements") has unknown behavior at scale: risk of duplicate consents, incorrect consent types, and unknown downstream Chrono sync side-effects.
- Sean Bingham (SF) and Adam Botma (eComm, Spike ticket) are already engaged. Coordination must happen at Engineering level before any Production action.
- The investigation record is empty — no SF configuration has been verified by Support, so no safe independent Support action exists.

---

## 3. Proposed Fix (Minimal Viable)

This is an **Engineering investigation and coordinated backfill**, not a configuration change. The minimal viable fix has two phases:

### Phase 1 — Confirm Root Cause and Safe Remediation Path (Engineering)

1. **Sean Bingham** reviews the SF-side flow or trigger responsible for assigning Informed Consent 2026 on account/opportunity creation for Shopify orders. Confirms: (a) what broke on or before 3/18, (b) what the 3/27 fix changed, (c) whether the "Assign patient agreements" action is the correct backfill mechanism.
2. **Adam Botma** confirms the Shopify-side consent assignment flow and whether Shopify is the source of truth for consent state or whether SF owns it.
3. **Jointly confirm:** Does "Assign patient agreements" correctly backfill Informed Consent 2026 on affected accounts without (a) creating duplicates, (b) assigning the wrong consent version, or (c) triggering Chrono sync issues?
4. **Identify affected record set:** Query SF for accounts/opportunities created from Shopify MGT/GHT orders between 3/18 00:00 and 3/27 23:59 that lack an Informed Consent 2026 record. Use example case 500Ql00000tN0qrIAC as a known-affected anchor.

### Phase 2 — Batch Backfill (Engineering, Production)

5. Once Phase 1 confirms safety: execute "Assign patient agreements" action on the identified affected record set in Production. Run in batches (e.g., 50 records at a time) with spot-checks between batches to confirm correct consent record creation and no Chrono sync errors.
6. Post-backfill: spot-check 10–15 previously affected accounts to confirm Informed Consent 2026 now exists and is correct.
7. Open an ADO bug ticket documenting the systemic defect if root cause confirms a code/flow defect in the Shopify→SF consent assignment path.

**Why this is the minimal viable fix:** The 3/27 fix already resolved the forward-looking problem. The only remaining work is the backfill for the 3/18–3/27 window. We are NOT proposing a flow redesign, a new object, or new automation — because the existing "Assign patient agreements" action may already be the correct remediation tool. The minimal fix is: confirm it is safe, then run it on the affected records. Nothing more.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

There is no Salesforce configuration change to deploy. The fix is a data remediation action ("Assign patient agreements") against existing Production records.

**Pre-Production validation steps (use Sandbox):**
1. In a **full Sandbox** (with a copy of Production data including affected accounts), identify 3–5 accounts that replicate the affected state (missing Informed Consent 2026, created in the 3/18–3/27 window).
2. Manually run "Assign patient agreements" on these accounts in Sandbox.
3. Confirm: (a) Informed Consent 2026 record is created, (b) no duplicate consent records exist, (c) correct consent version/type is assigned, (d) SF→Chrono sync fires correctly (or does not fire unexpectedly).
4. If Sandbox does not have a Chrono integration active, confirm with Engineering what the Chrono sync trigger condition is and whether it will fire differently in Production.
5. Only after Sandbox validation passes should the Production batch backfill proceed.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| "Assign patient agreements" creates duplicate consent records | Medium | High — compliance, data quality | Validate in Sandbox first; spot-check after each batch in Production |
| Wrong consent version assigned (e.g., 2025 instead of 2026) | Medium | High — regulatory/compliance | Confirm consent version logic in Phase 1; spot-check output records |
| Chrono sync errors or duplicate clinical notifications triggered | Medium | High — patient safety, clinical workflow | Confirm Chrono sync behavior with Engineering before Production run |
| Affected record query misses some accounts or overcaptures | Medium | Medium — incomplete backfill or unintended changes | Anchor query validation to known-affected case 500Ql00000tN0qrIAC |
| 3/27 fix is incomplete — new orders still affected | Low | High — scope grows | Monitor post-3/27 new orders for a 48h window before running backfill |

### Rollback

There is no configuration change to roll back. Rollback applies only to the data action:

- **If duplicate or incorrect consent records are created:** Engineering manually deletes the incorrectly created Informed Consent records using SF Data Loader or a targeted DELETE operation. Scope is limited to records created by the backfill action (identifiable by CreatedDate = backfill execution date).
- **If Chrono sync issues occur:** Immediately pause the batch backfill. Engage the Chrono integration team to identify and reverse any erroneous downstream records before resuming.
- **Rollback window:** Because this is a data backfill (not a metadata deployment), rollback is always possible by deleting the created consent records. Running in batches of 50 limits blast radius at each step.
- **Recommendation:** Run first batch of 10 records in Production, wait 30 minutes, manually inspect all 10 output records and Chrono state before proceeding with remaining batches.

## Justification: Why This Is the Minimal Viable Fix

The forward-looking problem is already solved by the 3/27 fix. The only remaining gap is the backfill for 3/18–3/27. We could propose redesigning the Shopify→SF consent assignment automation entirely, adding monitoring/alerting, or building a new backfill flow — but none of that is necessary to solve the stated problem. The "Assign patient agreements" action already exists and is the documented remediation path referenced in the Jira evidence. The minimal fix is: **verify it is safe, then run it on the affected records.** No new metadata. No new automation. No redesign. Just a coordinated, confirmed, batched data action on a known finite record set.
