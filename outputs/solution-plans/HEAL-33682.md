# Solution Plan — HEAL-33682

## 1. Problem Statement

Three Lab Order records in Salesforce Production (IDs: a47Ql000000JI2rIAG, a47Ql000000Lo9tIAC, a47Rh0000001zeEIAQ) triggered physical PGT Booklet fulfillment via the Novogenia API (product code `#VRSF` — Report Print & Shipment Fee), but no corresponding Opportunity or Lab Order record exists in Salesforce to represent the booklet order. This creates an orphaned fulfillment state: Novogenia produced and shipped physical booklets, but Salesforce has no record of the transaction.

Affected parties: patients/customers associated with those 3 Lab Orders, operations staff unable to reconcile fulfillment, and potentially billing if `#VRSF` carries a cost. Two of the three orders date to 2024, placing them outside the 30-entry API log retention window available to Support. Confirmed by Courtney Marlin via Novogenia portal review (2026-05-18T10:36).

---

## 2. Escalation Decision

**Escalated to Engineering.**

Support cannot resolve this issue because:
- The API submission logs for the 2024 Lab Orders are beyond the 30-entry Support visibility window.
- Root cause is unknown: it could be (a) the API payload erroneously included `#VRSF`, (b) a trigger or Flow that should create the SF booklet order record failed silently, or (c) records were created and subsequently deleted.
- Without confirming root cause, any manual record creation in Production risks duplicate fulfillment or data misattribution.
- If `#VRSF` is being included in error across additional orders, a systemic fix is required — not a one-off data patch.

Engineering-level log access is required before any corrective action is taken.

---

## 3. Proposed Fix (Minimal Viable)

**This is an Engineering investigation task, not a configuration change.** The minimal viable fix is scoped to two phases:

### Phase 1 — Engineering Investigation (required before any fix)

1. Pull historical API call logs for the 3 affected Lab Order IDs from the Novogenia integration layer.
2. Confirm whether `#VRSF` was intentionally or erroneously included in each payload.
3. Identify the Apex trigger, Flow, or integration process responsible for creating the SF booklet order/Opportunity record at submission time.
4. Determine the failure point: did the automation not fire, fire and fail, or was a record created and later deleted?
5. Check whether additional Lab Orders outside these 3 also have `#VRSF` in API logs without corresponding SF records (scope check for systemic recurrence).

### Phase 2 — Minimal Corrective Action (after root cause confirmed)

**If root cause is a bug in automation (trigger/Flow failed to create SF record):**
- Fix the automation bug in Sandbox. This is the narrowest possible fix — patch the specific failure point, do not redesign the integration.
- After fix is validated in Sandbox and deployed to Production, manually create the 3 missing SF booklet Lab Order records with correct linkage to parent Lab Orders/Opportunities, using confirmed data from API logs to avoid misattribution.

**If root cause is erroneous `#VRSF` inclusion in the payload:**
- Identify and patch the payload construction logic (Apex or Flow) to conditionally include `#VRSF` only when a booklet order record exists in SF.
- Do not create SF records for the 3 affected orders until it is confirmed the booklets were legitimately intended.

**Interim workaround (approved by reporter):** Manual reconciliation is acceptable per Courtney Marlin's own assessment. No action required in Production until Engineering confirms root cause.

- **Engineering log access:** Investigation cannot proceed without access to API submission logs beyond the 30-entry Support window.
- **Root cause confirmation before data creation:** Manual SF record creation for the 3 orphaned orders must not happen until Engineering confirms the intended state of each order — risk of duplicate fulfillment or billing errors.
- **Scope check for systemic recurrence:** If additional orphaned orders are found during log review, the fix scope expands. The current fix is scoped to 3 records; Engineering must confirm no broader data integrity issue exists before closing.
- **Courtney Marlin / Novogenia portal access:** Required to cross-reference any additional orders Engineering identifies in logs against Novogenia fulfillment records.
- **Sandbox environment parity:** If an automation bug is confirmed, the fix must be testable in a Sandbox that mirrors the Production integration configuration for the Novogenia API.

## 6. Sandbox Plan + Risk + Rollback

### Deployment to Sandbox

1. Engineering identifies the failing automation component in Production logs.
2. Reproduce the failure condition in Sandbox using a test Lab Order with `#VRSF` product code.
3. Apply the narrowest targeted fix (patch the specific trigger/Flow node that failed — do not refactor the broader integration).
4. Validate in Sandbox: confirm that submitting a Lab Order with `#VRSF` now creates the corresponding SF booklet order record, AND that orders without `#VRSF` are unaffected.
5. Deploy to Production via standard change management process.
6. Post-deploy: manually create the 3 missing SF booklet records in Production using API log data, with confirmation from Courtney Marlin.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Manual record creation causes duplicate Novogenia fulfillment | Medium | High | Do not create SF records until Engineering confirms booklets were legitimately intended and Novogenia will not re-trigger |
| Additional orphaned orders discovered during log review | Medium | Medium | Scope check in Phase 1; address as part of same Engineering task |
| Automation fix inadvertently affects other Lab Order types | Low | High | Sandbox regression test on non-booklet Lab Orders before Production deploy |
| 2024 orders have billing/revenue recognition implications | Unknown | High | Flag to Finance/Billing team once root cause confirmed; do not close ticket until billing impact assessed |
| `#VRSF` erroneously included going forward (ongoing leak) | Unknown | High | Resolved by Phase 2 payload fix; monitor for new orphans post-deploy |

### Rollback

- **Automation fix:** If the deployed fix causes regressions, revert the Apex class or Flow version via Salesforce deployment rollback or reactivate the prior Flow version. No data changes are made by the fix itself, so rollback is low-risk.
- **Manual record creation:** If the 3 SF booklet records are created incorrectly, delete them in Production (no downstream automation should trigger if root cause is fixed). Confirm with Courtney Marlin that Novogenia portal status is unchanged.
- **No rollback needed for investigation phase** — read-only log review, no Production changes.

---

## Justification: Why This Is the Minimal Viable Fix

The temptation here is to redesign the Novogenia integration to add validation guards, order record existence checks, and alerting for orphaned fulfillments. We are explicitly not doing that.

**Why not:** The problem affects 3 records. The root cause is unconfirmed. Over-engineering a guard system before confirming whether this is a payload bug or a record-creation failure would mean building the wrong thing.

**What we are doing instead:** The minimal fix is (1) read the logs to find the actual break point, then (2) patch exactly that break point — one automation fix, scoped to the confirmed failure mode, followed by a targeted 3-record data correction. Nothing more.

If the scope check in Phase 1 reveals systemic recurrence across many orders, the fix scope will need to expand — but that decision should be made with evidence, not preemptively.
