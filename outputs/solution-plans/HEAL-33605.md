# Solution Plan — HEAL-33605

## 1. Problem Statement

Amy Stolze does not have access to Email and SMS templates in Salesforce. The request is to mirror the access already held by Veronica B. The exact mechanism of the access gap is unconfirmed — it may be a Profile permission, a missing Permission Set assignment, or a template folder sharing/visibility setting. SMS templates may be managed via a third-party package (e.g., Mogli, Twilio Flex), which could introduce additional licensing or package-permission requirements.

**Who is affected:** Amy Stolze (end user, role unconfirmed).
**What is broken:** Amy cannot view or use Email and/or SMS templates in Salesforce.

## 3. Proposed Fix (Minimal Viable)

**Approach: Mirror Veronica B's Permission Set assignments and folder sharing to Amy Stolze — no new metadata created.**

The minimal fix reuses whatever access mechanism already grants Veronica B her template visibility. Creating new Permission Sets or modifying Profiles is unnecessary if a Permission Set or folder share already exists that can simply be assigned/extended to Amy.

### Step-by-Step (to be executed after prerequisites in Section 2 are confirmed)

**Step 1 — Compare Permission Sets**
- In Production (read-only audit): navigate to Setup > Users > Amy Stolze > Permission Set Assignments.
- Do the same for Veronica B.
- Identify any Permission Sets Veronica holds that Amy does not.

**Step 2A — If a Permission Set gap exists:**
- In Sandbox: assign the identified Permission Set(s) to a test user mirroring Amy's Profile.
- Validate template access is restored.
- Deploy to Production: Setup > Users > Amy Stolze > Permission Set Assignments > Edit > add the missing Permission Set(s).
- No metadata deployment required — this is a data-level user assignment.

**Step 2B — If the gap is folder sharing (Classic Email Templates):**
- In Production: navigate to the Email Template folder(s) Veronica can access.
- Edit folder sharing: add Amy Stolze (or her Role/Public Group) with at least Read access.
- No metadata deployment required — folder sharing is record-level configuration.

**Step 2C — If SMS templates are managed-package-controlled:**
- Identify the package (e.g., Mogli SMS, Twilio Flex).
- Check if Veronica holds a package-specific Permission Set (e.g., "Mogli User").
- If yes, assign the same package Permission Set to Amy (Step 2A applies).
- If a license seat is required, escalate to the package admin or account owner to provision a license before assigning the Permission Set.

**Step 3 — Validate**
- Log in as Amy Stolze (or use "Log In As" in Sandbox with equivalent user) and confirm Email and SMS templates are visible and usable.
- Confirm Amy's access scope matches Veronica's — not broader.

## 5. Dependencies

- **Veronica B's access must be formally documented first** — "mirror Veronica's access" is an informal instruction. Replicating her access without audit risks propagating over-permissioning.
- **If a managed-package license is required for SMS templates**, a license seat must be provisioned before a Permission Set can be assigned. This may require involvement from the package vendor or a contract/billing owner.
- **Amy Stolze's Profile must support the templates in scope** — if her Profile has "View All" or "Manage" restrictions at the object or app level that Veronica's Profile does not, a Profile-level change may be needed. However, this should be avoided in favor of a targeted Permission Set if possible, to minimize blast radius.
- **Folder sharing model** — if the org uses role-based folder sharing, Amy may need to be assigned to the correct Role rather than given individual folder access, to ensure consistency with how other users in her role are managed.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

- **For Permission Set assignment changes:** No deployment pipeline needed. Assignments are data-level and applied directly to users. Validate in a Full or Partial sandbox with a user record matching Amy's Profile by temporarily assigning the Permission Set and testing template visibility.
- **For folder sharing changes:** Test in sandbox by sharing the relevant folder with a test user and confirming access. Apply in Production directly — no Change Set required.
- **For managed-package Permission Set:** Test in sandbox if the package is installed in sandbox. If not, validate in a lower-risk Production window with Amy's consent.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Over-permissioning Amy beyond Veronica's scope | Medium | Medium — exposes unintended templates | Audit Veronica's exact folder/PS access before copying; do not grant broader than confirmed |
| Managed-package license not available | Medium | Low-Medium — blocks SMS fix, requires escalation | Confirm license count before assigning package PS |
| Replicating Veronica's over-permissioning | Low | Medium | Formally review Veronica's access before treating it as the gold standard |
| Profile-level restriction blocks PS-granted access | Low | Medium — fix appears to work but breaks later | Test end-to-end in sandbox before Production assignment |

### Rollback

- **Permission Set assignment:** Remove the added Permission Set from Amy Stolze's user record. Instant, no deployment required.
- **Folder sharing:** Remove Amy (or her Role/Group) from the folder's sharing settings. Instant.
- **Managed-package Permission Set:** Remove the package Permission Set assignment from Amy's user record. Instant.

**No Change Sets were created. No metadata was modified. All changes are data-level user/folder assignments and are fully reversible in under 5 minutes.**

---

## Why This Is the Minimal Viable Fix

The request is a single user access issue. The correct resolution is the smallest change that brings Amy's access in line with Veronica's — nothing more. We are **not** modifying Profiles (which would affect all users on that Profile), **not** creating new Permission Sets (one likely already exists that grants Veronica this access), and **not** redesigning folder structure or template ownership. Every alternative approach is larger in scope and blast radius than a targeted Permission Set assignment or folder share. The fix is bounded, reversible, and requires no deployment pipeline.
