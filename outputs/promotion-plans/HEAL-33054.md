### Step 3.1 — Deploy Fields and FLS (PR #9062)

**What it does:** Deploys the two new custom fields (`Account.Cardone_Ventures_Employee__c` and `Opportunity.Cardone_Ventures_Employee__c`) and the five permission set FLS configurations to Production. No layout changes in this step.

```bash
sf project deploy start \
  --metadata "CustomField:Account.Cardone_Ventures_Employee__c" \
  --metadata "CustomField:Opportunity.Cardone_Ventures_Employee__c" \
  --metadata "PermissionSet:Admin_Team" \
  --metadata "PermissionSet:Medical_Coordinator" \
  --metadata "PermissionSet:Nursing_Staff" \
  --metadata "PermissionSet:Operations_Manager" \
  --metadata "PermissionSet:Operations_Team" \
  --target-org prod \
  --wait 30 \
  --json
```

**Expected success indicators:**
- `"status": "Succeeded"` in JSON output
- `"numberComponentsDeployed"` equals 7 (2 fields + 5 permission sets)
- `"numberComponentErrors": 0`
- No `"componentFailures"` array entries

**If this step fails:** Do NOT proceed to Step 3.2. Review the `componentFailures` array in the JSON output. Common failure causes: field API name collision (check 2A failed to catch existing field), permission set name mismatch (verify exact API names in Sandbox), or metadata format issue. Address root cause before retrying.

**Pause after Step 3.1:** Wait for `"status": "Succeeded"` confirmation before executing Step 3.2. Do not run both commands simultaneously.
