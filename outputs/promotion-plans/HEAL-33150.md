# Promotion Plan - HEAL-33150

## Status

Blocked. No Production change is authorized or ready.

## Corrected Promotion Scope

The likely Production change is not a blind Cx routing-address deployment. The fix must first be proven in Sandbox and should target the actual reply path for internal approval emails:

- Email action/template/default sender configuration, and/or
- Org-Wide Email Address / Reply-To behavior, and/or
- Shared approval/refund mailbox forwarding into Email-to-Case.

## Prerequisites

1. Engineering/Admin confirms the current Outlook From and Reply-To values for the affected Cx internal approval workflow.
2. Engineering/Admin identifies whether a shared mailbox receives the missing replies.
3. Sandbox fix is implemented.
4. Sandbox Outlook reply test creates an `Incoming = true` `EmailMessage` on the original Cx Case.
5. Regression test confirms existing routed replies to `support@10xhealthsystem.com` / `info@10xhealthsystem.com` still attach.
6. Sean explicitly authorizes any Production change.

## Production Readiness Criteria

Promotion can proceed only when all are true:

| Criterion | Required Result |
| --- | --- |
| Root cause confirmed | Reply path bypasses Salesforce or mailbox forwarding missing |
| Sandbox proof | Internal approver Outlook reply attaches to original Cx Case |
| Regression proof | Existing routed customer replies still attach |
| Change scope known | Exact Setup/mailbox/template/action change documented |
| Rollback path known | Previous settings/forwarding captured before Production change |
| Authorization | Explicit Production approval recorded |

## Post-Production Validation

After an approved Production change:

1. Send a controlled internal approval email from a test Cx Case.
2. Confirm Outlook receives the intended From and Reply-To.
3. Reply from Outlook without changing the subject/thread token.
4. Verify the reply appears on the original Case as `Incoming = true`.
5. Confirm no duplicate/orphan Case is created.
6. Confirm existing routed customer replies still work.

## Rollback

Rollback depends on the final change type:

- Reply-To / email action / template setting: restore the previous setting or metadata version.
- Org-Wide Email Address behavior: restore previous allowed/default sender configuration.
- Shared mailbox forwarding: remove or revert the forwarding rule.

No rollback or deployment command should be executed without explicit authorization.
