## Section 1: Summary — What Will Change in Production

### Metadata Change Matrix (Conditional by Scenario)

| Component | Type | Scenario A | Scenario B | Scenario C | Deploy Method |
|---|---|---|---|---|---|
| Email-to-Case Routing Address (new) | Setup Configuration | ✅ CREATE | ❌ | ❌ | Direct Setup (no Gearset) |

### Objects / Fields / Flows / Validation Rules
- **No CustomObject, CustomField, Flow, or ValidationRule changes are expected for any scenario.**
- **No Apex code changes are expected for any scenario.**
- **No permission set or profile changes are expected for any scenario.**
- The only Salesforce metadata that may change is a Quick Action definition and/or a Page Layout assignment — and only under Scenario B.
- All other changes are declarative Setup configuration (routing address records) applied directly in the Production org UI or via IT infrastructure — not deployable via sf project deploy.

### Scenario Descriptions
- **Scenario A:** No active Email-to-Case routing address exists for the Cx Case Record type. A new routing address must be created in Setup and (if On-Demand E2C) SMTP forwarding configured.
- **Scenario B:** A routing address exists but the outbound email Reply-To is wrong — a custom Send Email quick action or OWA override is intercepting the Reply-To header. Metadata fix required.
- **Scenario C:** A routing address exists but is set to Inactive. A single toggle in Setup activates it. No metadata deploy required.

### SCENARIO A — Create New Email-to-Case Routing Address for Cx Case Record Type

**Use when:** No routing address exists for Cx Case Record type (confirmed by Section 2.2 returning zero rows).

**Step A-1: Confirm the working baseline routing address (from a working Case record type)**
```
sf data query --query "SELECT Id, Name, EmailAddress, IsActive, CaseRecordTypeId, CaseOrigin, IsVerified FROM EmailServicesAddress WHERE IsActive = true ORDER BY Name ASC" --target-org prod --json
```
**What it does:** Lists all active routing addresses in Production. Identify the routing address of a confirmed-working Case record type. Note its `CaseOrigin` value and email domain pattern — mirror these when creating the Cx Case Record routing address.
**Expected output:** At least one active routing address row. If zero, Email-to-Case is not functioning for any record type — escalate immediately; do not proceed.

**Step A-2: Create the new routing address in Production Setup (manual — no CLI)**
```
Navigation: Production Setup → Email-to-Case → Routing Addresses → New
Field values to set:
  Routing Name: Cx Case Record Email Routing (or match org naming convention)
  Email Address: <routing-address>@<your-domain>.com (coordinate with IT for mailbox creation)
  Case Record Type: Cx Case Record
  Case Origin: Email (or match working baseline value from Step A-1)
  Active: CHECKED
  Save the record
  Note the generated routing address ID from the URL bar after save
```

**What it does:** Creates a new Email-to-Case routing address record that Salesforce will monitor for inbound emails to log against Cx Case Record cases.
**Expected output:** New routing address appears in the Routing Addresses list with Active = true.

**Step A-3 (On-Demand E2C only): Confirm email service endpoint for SMTP forwarding**

**Expected output:** Returns the Email Services function. The inbound email address for SMTP forwarding will be visible in Setup → Email Services → [function name] → Email Addresses. Provide this address to IT for SMTP forwarding configuration from the new routing mailbox.

**Step A-4 (On-Demand E2C only): IT action — configure SMTP forwarding**
