# Escalation Gate Decision — HEAL-33150

## Decision
Engineering-Required

## Confidence
High — 91% — Root cause is isolated to Email-to-Case routing infrastructure (routing addresses, thread token configuration, and potentially Apex/Flow email handlers) specific to the Cx Case Record type. These components require direct org access at an Engineering level; Support cannot safely audit or modify Email-to-Case routing addresses, email templates with merge field validation, or inbound routing handler logic within sandbox scope.

## Key Evidence
- The defect is record-type-specific: inbound email replies are failing to attach to existing Cx Case Records, while other record types are presumed functional — isolating the fault to Cx-specific Email-to-Case routing configuration rather than a global platform issue
- The reporter confirmed the thread ID token was present in the outbound email, ruling out a simple template omission and pointing instead to a misconfigured or absent inbound routing address, or a routing address not set to Thread ID matching mode — both requiring Engineering-level org inspection
- Fixing the issue requires auditing Email-to-Case routing addresses, verifying `{!Case.Thread_Id}` merge field rendering in Cx templates, reviewing routing mode settings, and potentially inspecting OnEmailMessage Apex triggers or Flows — scope that exceeds standard Support sandbox resolution
- The investigation record remains template-incomplete (reproduction steps, confirmed facts, and hypotheses unpopulated), meaning no hands-on Salesforce org verification has been performed — Engineering must conduct the baseline audit before any fix is scoped or applied
- Issue affects multiple teams and departments per reporter confirmation, raising the blast radius and making unsupervised Support-level experimentation with routing configuration an unacceptable risk

## Red Flags
- The investigation record is largely unfilled — no confirmed reproduction steps, no documented org inspection, and no validated hypotheses. The root cause stated in internal notes is analytical inference, not verified against the live org. Engineering should treat the proposed root cause as a strong hypothesis, not a confirmed finding, and conduct a read-only Production audit before acting.
- Inbound replies may currently be silently discarded or generating orphaned cases. If this has been ongoing, there is an active data integrity risk that should be assessed for scope and duration before any fix is deployed.
- No workaround is documented for affected agents beyond manual copy-paste. Engineering handoff should include a interim workaround recommendation to prevent continued data loss during the investigation window.

## Next Step
Approve and dispatch Engineering Handoff for HEAL-33150. Engineering to perform a read-only audit of the Production org covering: (1) Email-to-Case routing addresses assigned to or used by the Cx Case Record type — confirm existence, active status, and matching mode (Thread ID vs. new case); (2) outbound email templates associated with Cx cases — confirm `{!Case.Thread_Id}` merge field is present and rendering correctly; (3) any OnEmailMessage Apex triggers or Flows that may intercept inbound replies before case matching occurs; (4) record type routing rules to confirm Cx record type is correctly mapped to its routing address. After audit, Engineering to document confirmed root cause, scope impact to other record types if routing addresses are shared, and propose a sandboxed fix prior to Production deployment. Operator should also request an interim workaround be communicated to affected agent teams immediately.
