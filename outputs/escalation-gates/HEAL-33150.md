# Escalation Gate Decision — HEAL-33150

## Decision
Engineering-Required

## Confidence
High — 94% — The root cause implicates Email-to-Case routing address configuration and outbound email header inspection, both of which require direct Production org access. No Support-side action can resolve a misconfigured or missing routing address, nor can Support confirm what address appears in the Reply-To header of live outbound emails without Engineering-level org access. The fix is likely declarative once root cause is confirmed, but diagnosis and validation cannot be performed without direct Setup access and live email testing.

## Key Evidence
- The outbound email thread token is confirmed present by the reporter, meaning Email-to-Case mechanics are partially functional — the failure is in the reply routing path, not token generation. This narrows the root cause specifically to routing address configuration or Reply-To header resolution, neither of which Support can inspect or modify.
- Email-to-Case routing address configuration (active/inactive status, record type assignment, Reply-To address injection) lives in Salesforce Setup and potentially involves SMTP forwarding at the email server level — both are outside the scope of Support-resolvable actions.
- Scope is explicitly broad: multiple teams, departments, and large numbers of records are affected. Any configuration change carries regression risk across open and future Cx Case Records, requiring Engineering-led testing in a mirrored Sandbox before Production changes are made.
- Five reporter screenshots and a Production Salesforce report link have not been reviewed by any analyst — these are required to confirm scope and may reveal additional affected record types or edge cases that could change the fix strategy.
- The investigation surfaces at least six plausible hypotheses (missing routing address, thread token not in Reply-To header, OWA used as sender, custom email action override, On-Demand Email-to-Case SMTP misconfiguration, record-type-specific routing rule gap) — none can be confirmed or eliminated without direct Production org inspection, which is an Engineering function.

## Red Flags
- No direct review of the five attached screenshots or the reporter-provided Salesforce report has been performed. The actual scope of affected records is unconfirmed and could be larger or structurally different than described.
- It is unknown whether this issue also affects external client communications or is limited to the internal user (Tyler) scenario described. If external client emails are also not routing back, business impact is significantly higher than currently assessed.
- On-Demand Email-to-Case SMTP forwarding dependency is unconfirmed. If the org uses On-Demand Email-to-Case, correcting a routing address will also require IT/infrastructure changes outside of Salesforce Setup, which could extend resolution timeline and requires a separate workstream.
- It has not been confirmed whether the Cx Case Record type has ever had working email threading. If this is a net-new record type that was never fully configured for Email-to-Case, the fix scope may be broader than a single routing address correction.

## Next Step
Engineering handoff is required. Engineering should execute the following in order:

1. Review all five screenshots attached to HEAL-33150 and the reporter's Salesforce report link to confirm scope and identify specific affected records.
2. Audit all active Email-to-Case routing addresses in Production Setup — document routing address, active/inactive status, record type assignment, and case origin for each.
3. Inspect the Cx Case Record type page layout and Lightning page — identify whether the Send Email action is the standard Case.SendEmail quick action or a custom variant, and confirm Reply-To address configuration.
4. Send a test email from a Cx Case Record (Sandbox mirroring Production config) and inspect raw email headers to confirm what address appears in Reply-To and whether the thread token is present in both body and headers.
5. Identify at least one working Case record type in the org, document its full Email-to-Case configuration as a verified baseline, and use it as the template for the Cx Case Record type fix.
6. Apply the confirmed fix in Sandbox, validate end-to-end (send from case → reply from Outlook → confirm EmailMessage created on case Activity timeline), then promote to Production via Gearset or org-standard tooling.
7. Support assignee Sean Bingham should notify reporter Noah Scott that the issue has been escalated to Engineering and provide a status update with estimated timeline once Engineering completes the initial audit.
