# Escalation Gate Decision — HEAL-33562

## Decision
Engineering-Required

## Confidence
High — 87% — The root cause is embedded in Salesforce Approval Process metadata configuration and/or permission set assignments, neither of which Support can safely audit or modify. The issue has persisted for months, affects a role class (Clinical/Nurse users) rather than a single record, and requires Admin-level org inspection. The internal investigation explicitly confirms no support-tier remediation path exists.

## Key Evidence
- The Recall button absence is tied to either the Approval Process "submitter" setting or a permission set assignment change — both are org-level metadata components outside Support's safe modification scope
- Affected record `006Ql00000eMscDIAS` confirmed in Production; Tyler Kaplan (RN, Clinical) cannot see the Recall button, and Courtney's internal note acknowledges a permission set investigation is required
- The OP team is actively performing recalls as a manual workaround on behalf of Nurses, confirming the issue is not isolated to one record or one moment in time — it is a systemic permission gap
- A critical risk flag exists: the access restriction may have been intentional, meaning restoring the button without stakeholder sign-off (Courtney, Clinical leadership, Order Processing) could violate a deliberate policy change
- If automation (Flow or Process Builder) is submitting approvals on behalf of users, re-granting a permission set alone will not restore the button — a deeper process-level fix would be required, which is strictly Engineering territory

## Red Flags
- The change may have been deliberate — re-enabling recall without stakeholder confirmation from Clinical leadership and Order Processing risks reversing an intentional access control decision
- Scope is unconfirmed: all Nurse/Clinical users may be affected, not just Tyler Kaplan, which elevates the blast radius of any fix applied without full audit
- The investigation record contains incomplete reproduction steps and no formal Acceptance Criteria, which may slow Engineering triage — handoff doc should be verified complete before routing
- Timing of the original change is unknown ("months" per Tyler), meaning audit trail in the org may be limited or require Setup Audit Trail review

## Next Step
Route to Engineering and Salesforce Admin for the following scoped audit:

1. Identify the Approval Process governing Opportunity records and inspect the "Allow submitters to recall approval requests" setting and submitter qualification criteria
2. Review Setup Audit Trail for permission set assignment changes affecting Clinical/Nurse user profiles over the relevant timeframe
3. Determine whether a Flow or Process Builder is initiating the approval submission on behalf of users, which would strip submitter status and block the Recall button regardless of permission settings
4. Hold all remediation pending explicit stakeholder sign-off from Courtney, Clinical leadership, and Order Processing confirming the access restriction was not intentional
5. Validate any proposed fix in Sandbox against a Nurse-profile test user before deploying to Production
