# Escalation Gate Decision — HEAL-33040

## Decision
Engineering-Required

## Confidence
High — 95% — Two independent sync paths have both failed, ruling out a configuration or operator error. The broken behavior spans automatic trigger logic and a manual fallback field, both of which require code or Flow inspection that Support cannot perform.

## Key Evidence
- Both the automated Shopify sync on Account creation and the manual fallback trigger (`Trig Shopify Sync = True`) are non-functional on the same account (`001Ql00000x8VCkIAM`), indicating the shared underlying sync logic is broken rather than a single misconfigured entry point.
- No Shopify Customer ID was written and no error appeared in `Shopify Sync Message`, confirming a silent failure — this type of failure requires Engineering to inspect Apex debug logs, Flow fault paths, or external API response handling.
- Danielle Cress escalated internally to the SF team on 2026-04-13 and manually set `Trig Shopify Sync = True` in Production with no result — a Support-level intervention was already attempted and had no effect.
- The issue is confirmed in Production on a live Account record; no sandbox fix has been developed or tested, meaning no deployment path is available to Support.
- The failure is scoped to accounts created via the new patient link entry point, suggesting a code-level or Flow-level conditional that does not handle this entry point correctly — a root cause only Engineering can identify and correct.

## Red Flags
- Ticket was auto-closed after 24 hours of no customer response, which may give the reporter a false impression the issue is resolved. The Jira ticket should be reopened and linked to an active engineering work item to prevent the bug from being lost.
- The scope of impact is unknown. Other accounts created via the new patient link during the affected window may also be missing from Shopify, but no audit has been performed yet. Engineering should assess blast radius before deploying a fix.
- Multiple `Trig Shopify Sync = True` attempts have already been made in Production. Once the fix is deployed, unintended duplicate sync events may fire if the field retains a truthy state. Engineering should account for this in the fix design.
