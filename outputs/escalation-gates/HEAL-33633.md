# Escalation Gate Decision — HEAL-33633

## Decision
Engineering-Required

## Confidence
High — 97% — The root cause is unambiguously in Salesforce integration code (Apex or Flow) that constructs the Wellvi API payload. Shipping address data is confirmed present on the Order record in Salesforce but is absent in the outbound payload, which is a field mapping defect in the integration layer. Support has no tooling or access to inspect or modify Apex classes, Flows, or API callout construction logic. There is no data entry fix, configuration toggle, or manual workaround available to Support that would resolve this.

## Key Evidence
- Wellvi's API returned HTTP 400 explicitly stating AddressLine1, City, StateProvince, and PostalCode are missing from the submitted payload — confirming the Salesforce integration is sending null values for all four address fields.
- Sarah Phonthaphanh confirmed shipping information is visibly present on the Order record in Salesforce — ruling out a simple data-entry omission and establishing that the defect is in the mapping/reading logic of the integration component, not in the data itself.
- Danielle Cress confirmed Wellvi never received the order and explicitly re-routed the issue from Wellvi back to the Salesforce/Engineering team — confirming this is an internal Salesforce-side defect, not a Wellvi platform issue.
- The fix requires identifying the specific Apex class or Flow responsible for constructing the Wellvi eSubmit callout, running SOQL against Order 801Ql00000zolrtIAA to inspect actual field values, correcting the field mapping, validating in Sandbox, and deploying to Production via Gearset — all actions outside Support's scope.
- No metadata or code changes have been made by this pipeline; the bug is live in Production and actively blocking the Operations user from submitting Joshua Brown's order.

## Red Flags
- It is unknown whether this mapping defect is isolated to Order 801Ql00000zolrtIAA or is systemic across all Wellvi eSubmit orders — Engineering must determine blast radius before scoping the fix.
- The address fields the integration reads may differ from the fields visible on the Order page layout; Engineering must run SOQL before assuming a code-only fix is sufficient — a data correction on this specific record may also be required.
- The Jira record is currently in "Waiting for customer" status, which is incorrect given the defect is internal to Salesforce — the status should be updated to reflect active Engineering investigation to avoid the ticket aging out or being closed prematurely.

## Next Step
Engineering handoff is required. Engineering should: (1) identify the Apex class or Flow that handles the Wellvi eSubmit callout on Order records; (2) run SOQL against Order 801Ql00000zolrtIAA to confirm which shipping address fields are null versus populated at the record level; (3) trace the field mapping in the integration component to determine whether it references the wrong field, the wrong related object, or a custom field set that is unpopulated on this record; (4) correct the mapping so AddressLine1, City, StateProvince, and PostalCode in the Wellvi request body draw from the correct populated Salesforce field(s); (5) validate the fix in Sandbox against this Order record or a data clone; and (6) deploy to Production via Gearset.
