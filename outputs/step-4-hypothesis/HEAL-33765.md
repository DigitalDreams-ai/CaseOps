# Step 4 — Problem Hypothesis and Solution

**Issue:** HEAL-33765  
**Date:** 2026-05-21

---

## Problem Hypothesis

**Confirmed facts (from Step 3):**
- Three Pre-charting users (adelavega, cfernandez, jgallano) cannot update the Main Practitioner field on Account records
- Users are transitioning from MIST (Medication Instruction Support Team) role to Pre-charting role
- Requestor reports they are "completely blocked" from this functionality
- Record type is Patient Account (001Rh000001CVZ8IAO example provided)
- Impact is immediate and blocks daily Pre-charting workflow

**Symptoms (what the user reported):**
- Users cannot edit Main Practitioner field on Account records
- No error message provided, but functionality is blocked
- Symptom affects all three named users uniformly (likely a role/permission set issue, not user-specific)

**Root cause hypothesis:**
The Pre-charting role's Permission Set or Field-Level Security (FLS) configuration does not grant Read/Write access to the Main Practitioner field on Account records, causing the field to be inaccessible or read-only for these users.

---

## Smallest Viable Fix

**What to fix:**
- **Metadata/code artifact:** Permission Set assigned to Pre-charting role OR Field-Level Security rule for Main Practitioner field on Account
- **Change scope:** Add Read/Write Field-Level Security permission on Account.Main Practitioner field for the Pre-charting Permission Set
- **Why it solves the problem:** Granting explicit FLS Read/Write access on this field will make it editable for all users with the Pre-charting role

---

## Sandbox Validation Plan

**What to test in Sandbox:**
1. Log in as a Pre-charting user (or user with Pre-charting permission set)
2. Navigate to a Patient Account record
3. Attempt to edit the Main Practitioner field
4. Verify the field is editable and accepts input

**Expected outcome:**
- Main Practitioner field is visible and editable on Account records
- No "insufficient privileges" error appears
- Field value can be changed and saved

**Success criteria:**
- All three named users (adelavega, cfernandez, jgallano) can successfully edit the Main Practitioner field in Sandbox
- Field-level security confirms Read/Write permission is granted
- Change can be deployed to Production without errors

---

## Rollback Plan

**If Sandbox test fails:**
1. Investigate which Permission Set is actually assigned to Pre-charting users
2. Check FLS configuration for Main Practitioner field across all Permission Sets
3. If field is read-only or missing from FLS entirely, escalate to determine the intended access model

**If Production deployment fails:**
1. Revert FLS changes via Gearset or org backup
2. Notify reporter that deployment was rolled back pending further investigation
3. Investigate: Compare Sandbox FLS state vs Production state (may have permission set differences)

---

## Risks and Constraints

**Implementation risks:**
- Risk: If Pre-charting role should NOT have Main Practitioner write access for security/compliance reasons, this fix could introduce unauthorized access
- Constraint: Cannot modify the Main Practitioner field definition itself (field must remain as-is for other users)

**Mitigation:**
- Verify with stakeholder (Scarlett Garcia / Clinical department) that Pre-charting users should indeed have write access to Main Practitioner
- Confirm via Production metadata inspection that other similar clinical roles have this same access level
- Audit FLS after deployment to ensure only Pre-charting (and expected roles) have write access
