## Salesforce Problem

### Confirmed Facts

- The issue is systemic — it affects **all** cases in the CX case function, not a single record.
- The affected department is **Customer Support**.
- The error occurs specifically when attempting to **close a case** (status change or close action).
- At least two specific Case record IDs are confirmed affected: `500Ql00000vqcl7IAA` and `500Ql00000w0TS3IAM`.
- The issue was reported on 2026-04-30 and is categorized under **Salesforce - Errors**.
- The impact level is **"Multiple teams, departments, or large numbers of records"** and the reporter is **completely blocked**.
- A screenshot of the error exists but its content has not yet been parsed in this investigation step.
- The system affected is identified as **Salesforce - Errors**, indicating a runtime error rather than a UI/UX or access issue.

### Hypotheses

1. **Flow fault (most likely):** A record-triggered Flow or Screen Flow associated with Case closure is encountering a fault — e.g., a null reference on a required field, a failed lookup, a DML error, or an uncaught fault path. Because the error is 100% consistent across all cases, the fault likely hits a code path that always executes on closure (e.g., an element that requires a field value that is now null or missing, or references a deleted/inactive record such as a Queue, User, or related record).
2. **Validation rule:** A validation rule on the Case object was recently added or modified and is now preventing the Status from being set to "Closed." If the rule evaluates a condition that is always true (or references a field that has changed), it would block all closures.
3. **Process Builder fault:** If a legacy Process Builder process is still active on the Case object and fires on case closure, a broken action (e.g., referencing a deleted field, inactive user, or failed cross-object update) could cause the error.
4. **Apex trigger error:** An Apex before/after trigger on the Case object could be throwing an unhandled exception when the Status is changed to Closed, surfacing as a user-visible error.
5. **Required field or missing data:** A required field was recently made mandatory (via field-level settings, flow assignment, or validation rule) that is not populated on existing cases, causing closure to fail universally.
6. **Recent deployment regression:** A recent metadata deployment introduced a breaking change to the case closure automation (flow version activation, field deletion, etc.).

### Likely Affected Metadata

- **Object:** `Case` (standard Salesforce object)
- **Flow(s):** One or more record-triggered Flows or Screen Flows associated with the CX Case function — likely activated on Case Status change or Case closure. API name(s) unknown; must be inspected in the org (look for active Flows on the Case object with entry criteria or fault paths related to status = "Closed").
- **Process Builder:** Any active Process Builder process on the Case object firing on record update (Status change).
- **Validation Rules:** Any active Validation Rules on the Case object, particularly those referencing Status = "Closed."
- **Apex Triggers:** Any Apex triggers on the `Case` SObject (before update / after update) that execute on status change.
- **Case Status Picklist:** The Case Status field and its picklist values (specifically the "Closed" category values) — if a value was removed or recategorized, flows/automations referencing it may fail.
- **Related Records:** Any Queue, User, or related object referenced by the flow/automation that may have been deleted, deactivated, or renamed.
