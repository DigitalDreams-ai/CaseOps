## Issue Understanding

### Observed Behavior

When a user edits the **Order Notes** field on an Opportunity record, that change is **not recorded** in the **Opportunity Field History** related list. The field history tracking for "Order Notes" is either not enabled or not configured, so edits to that field go untracked and are invisible in the audit/history view.

### Expected Behavior

Edits and updates to the **Order Notes** field on Opportunity records should be captured in the **Opportunity Field History** related list, providing a full audit trail of who changed the field, when, and what the previous and new values were — consistent with how other tracked fields appear in that related list.

### Acceptance Criteria

1. The **Order Notes** field on the Opportunity object appears as a tracked field in the Opportunity Field History configuration.
2. After any edit to **Order Notes** on an Opportunity, a new row is created in the **OpportunityFieldHistory** (Field History related list) showing: changed-by user, date/time, old value, and new value.
3. Existing tracked history behavior for other Opportunity fields is not disrupted.
4. The fix is visible to Clinical department users (and other affected teams) without requiring additional permissions.

## Salesforce Problem

### Confirmed Facts

- The **Opportunity Field History** feature (standard Salesforce capability) is in use in this org — the related list is present and rendering on at least the example Opportunity record.
- The **Order Notes** field on the Opportunity object is **not currently enabled for Field History Tracking**, which is why edits to it do not appear in the history related list.
- Salesforce Field History Tracking is configured per-object in **Object Manager → Opportunity → Fields & Relationships → Set History Tracking** (or via `Setup → Object Manager`).
- Salesforce platform **limitation**: Field History Tracking supports a **maximum of 20 fields per object**.
- Salesforce platform **limitation**: **Long Text Area** and **Rich Text Area** field types **cannot** be tracked via standard Field History Tracking. If "Order Notes" is one of these types, the standard feature will not work.
- The reporter's department is Clinical; multiple teams and departments are affected, indicating this is a shared gap — not a user-specific permissions issue.

### Hypotheses

1. **(Most likely)** The **Order Notes** field was never added to the Opportunity Field History Tracking configuration. The fix is simply to enable tracking for that field in Object Manager — a straightforward admin configuration change.
2. **(Blocking edge case)** The **Order Notes** field is a **Long Text Area or Rich Text Area** type, which Salesforce does not support for standard Field History Tracking. In this case, a workaround would be needed (e.g., a Flow that logs changes to a custom object or uses a separate audit mechanism).
3. **(Secondary constraint)** The Opportunity object may already have **20 fields tracked**, which is the Salesforce platform maximum. Adding "Order Notes" would require removing another field from tracking first.

### Why This Should Fix It

Salesforce Field History Tracking works at the field level — each field must be individually opted in. Once "Order Notes" is checked in the Set History Tracking configuration, Salesforce's platform automatically begins capturing before/after values on every save that modifies that field, writing rows to the `OpportunityFieldHistory` object. This is exactly the audit behavior the reporter is requesting.

### Risks

1. **20-field tracking limit:** If the Opportunity object is already at 20 tracked fields, adding "Order Notes" requires removing another — which could silently drop history tracking for a field that other teams rely on. Must verify current count before making changes.
2. **Long Text Area / Rich Text Area limitation:** If the field is this type, standard tracking is not possible. Attempting to enable it will either fail silently or not appear as an option. This would require the Engineering escalation path.
3. **Historical data gap:** Field History Tracking is **not retroactive**. Only changes made *after* the tracking is enabled will be captured. Changes made before enablement are permanently unrecorded.
4. **Storage:** Field history entries count against org data storage. For high-volume orgs, enabling an additional field may incrementally increase storage consumption over time.

### Rollback Plan

If enabling tracking causes unintended issues (unlikely for this type of change): return to **Setup → Object Manager → Opportunity → Set History Tracking** and uncheck "Order Notes." No data loss occurs from disabling — existing history rows are retained but no new ones are written.
