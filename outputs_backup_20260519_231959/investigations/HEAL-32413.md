After an HRT Labs Opportunity is approved, nurses should be able to update the Lab Vendor field (and Lab Name field) to reflect the appropriate vendor used during the lab draw. This is a required step in the standard clinical lab draw workflow. All nurses in the Clinical department should have this capability.

### Acceptance Criteria

1. Nurses with a Clinical/nurse-level Salesforce profile or permission set can edit the Lab Vendor field on a Labs Opportunity after it reaches Final Approval status.
2. The record-locked error no longer appears for nurse-level users attempting to update Lab Vendor on approved Labs Opportunities.
3. Other Opportunity record types (non-Labs) remain unaffected — their approval lock behavior is unchanged.
4. The fix applies to all nurses on the clinical team, not just the original reporter.

## Salesforce Problem

### Confirmed Facts

1. The root cause is the **Opportunity Approval Process** (`Opportunity_Approval_Process`) with `recordEditability = AdminOrCurrentApprover` and `finalApprovalRecordLock = true`.
2. The approval process entry criteria includes `Opportunity.RecordType = Labs`.
3. Once a Labs Opportunity reaches Final Approval, Salesforce locks the record. Nurse-level users who are not admins or designated approvers receive the standard record-locked error when attempting any edit.
4. This is **expected Salesforce platform behavior** when `finalApprovalRecordLock = true` — it is not a bug in the traditional sense, but a configuration that is too restrictive for the clinical workflow.
5. Sean Bingham confirmed on 2026-03-24 that the lock is originating from the approval process, not from a flow or validation rule.
6. Sean Bingham proposed splitting Labs into its own approval process (`Opportunity_Approval_Process_Labs`) with `finalApprovalRecordLock = false`, while keeping the existing process (with `finalApprovalRecordLock = true`) for all other Opportunity record types.
7. The proposed new process order: Labs-specific process (processOrder 1, `finalApprovalRecordLock = false`), updated existing process with Labs removed from entry criteria (processOrder 2), Genetic Approval Process (processOrder 3).
8. An Engineering ticket (Bug #20941) has been created: "Bug: Nurses cannot update lab vendor on approved Labs Opps (HEAL-32413)".
9. As of 2026-04-07 (last update), no confirmed resolution has been communicated to the customer.

### Hypotheses

- **Primary (confirmed):** The `finalApprovalRecordLock = true` setting on `Opportunity_Approval_Process` locks Labs Opportunities upon final approval, preventing any non-admin, non-approver edits. This is the sole cause of the issue.
- **Secondary (ruled out):** A validation rule or flow was initially suspected by Danielle Cress (2026-03-19). Sean Bingham confirmed on 2026-03-24 that no validation rule or flow is causing this — the lock is entirely from the approval process.
- **Proposed fix hypothesis:** Splitting Labs into a dedicated approval process with `finalApprovalRecordLock = false` will allow nurses to edit Lab Vendor and Lab Name post-approval without affecting other Opportunity types. This was Sean's proposed solution (2026-03-25) and has been escalated to Engineering for implementation.

## Solution Plan

### Classification

- **Engineering escalation**
- **Escalation reason:** The fix requires creating and modifying Approval Process metadata in Salesforce, which is an Engineering-owned change. Approval Processes cannot be modified by Support-level configuration alone (no UI-only permission assignment will resolve a `finalApprovalRecordLock = true` restriction for non-admin users).

### Proposed Change

1. **Create** `Opportunity_Approval_Process_Labs` (new Approval Process):
   - Object: Opportunity
   - Record Type entry criteria: Labs only
   - processOrder: 1
   - `finalApprovalRecordLock`: **false**
   - Approval steps, approvers, email alerts, and field updates: mirror the existing Labs step (`Labs_Approval_Process_Steps`) from `Opportunity_Approval_Process`
   - `recordEditability`: AdminOrCurrentApprover (during the approval process itself)

2. **Update** `Opportunity_Approval_Process`:
   - Remove Labs record type from entry criteria
   - processOrder: 2
   - `finalApprovalRecordLock`: remains true (for all non-Labs opps still entering this process)

3. **Verify** `Opportunity_Approval_Process` (Genetic variant) remains processOrder 3 and is untouched.

4. Deploy from Sandbox → Production via Gearset (or team standard).

### Why This Should Fix It

When `finalApprovalRecordLock = false` on the Labs-specific approval process, Salesforce will not lock the record upon final approval. Nurse-level users will be able to edit fields including Lab Vendor and Lab Name on approved Labs Opportunities without encountering the record-locked error. The change is scoped to Labs record type only, leaving all other Opportunity approval behavior intact.

### Risks

1. **Downstream automations depending on Labs lock:** Any flow, trigger, or process builder that assumes approved Labs Opps are locked could behave unexpectedly once the lock is removed. Engineering should audit for this before deploying.
2. **Data integrity on approved Labs Opps:** With the record no longer locked, nurses could inadvertently edit fields beyond Lab Vendor/Lab Name. Consider whether field-level security or a validation rule should protect other fields post-approval.
3. **Approval step replication fidelity:** If the new `Opportunity_Approval_Process_Labs` does not perfectly replicate approvers, email alerts, and field updates from the existing Labs step, the approval workflow for new Labs Opps could break.
4. **Process ordering conflicts:** Incorrect processOrder configuration could cause Labs Opps to match both the new Labs-specific process and the updated existing process, leading to double-approval routing.

### Rollback Plan

If the deployment causes issues:
1. Deactivate `Opportunity_Approval_Process_Labs`.
2. Restore the Labs record type to the entry criteria of `Opportunity_Approval_Process` (revert to pre-change state).
3. Confirm processOrder reverts to original configuration.
4. Nurse-level users will return to the pre-fix state (cannot edit after approval), but no data loss will occur.
