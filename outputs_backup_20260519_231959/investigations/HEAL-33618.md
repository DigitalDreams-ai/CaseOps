## Salesforce Problem

### Confirmed Facts
- The standard Salesforce Case object includes an `IsEscalated` boolean field and a standard Escalation Rules engine, but **neither records a timestamp** of when escalation occurred.
- The standard `CreatedDate` and `ClosedDate` (via `IsClosed` status) fields are available on Case and can be used as the start/end anchors for elapsed-time calculations.
- Salesforce standard reports do not offer a formula-based elapsed time between two date/time fields natively without a custom formula field or summary formula.
- Capturing an escalation timestamp requires one of: (a) a custom Date/Time field on Case populated via Flow/trigger when escalation occurs, (b) leveraging the Case History related list (field history tracking) to query when `IsEscalated` changed to `true`, or (c) a Process/Flow that stamps a custom field on escalation.
- The affected org is `10xhealth.lightning.force.com` (Production).
- Impact scope: Multiple teams and departments.
- The customer has a workaround currently (nature of workaround not described).

### Hypotheses
1. **Most likely:** No custom `Escalation_Date__c` (or equivalent) field exists on the Case object, so there is no timestamp to report on. A custom field + automation (Flow) needs to be created to stamp the date/time when a Case is escalated.
2. **Alternative:** A custom escalation date field may already exist but is not being populated consistently (automation gap).
3. **Alternative:** Field History Tracking on `IsEscalated` may be enabled, allowing a SOQL query of `CaseHistory` to retrieve the timestamp — but this is not surfaceable in standard reports without a custom report type or formula.
4. The standard Salesforce Escalation Rules may fire but do not write a timestamp to any reportable field.

## Solution Plan

### Classification
- **Engineering escalation**
- **Escalation reason:** Requires new custom field creation, Flow automation, and report configuration — metadata changes that must be built and deployed via Sandbox → Production. Cannot be resolved through data, access, or config changes alone.

### Proposed Change
1. **Custom Field:** Create `Case.Escalation_DateTime__c` (Date/Time) to store the moment of escalation.
2. **Flow:** Create a Record-Triggered Flow on the Case object that fires when the escalation condition is met (to be confirmed: `IsEscalated` changed to `true`, or Status changed to an escalation value, or assignment to Tier 2 queue). Set `Escalation_DateTime__c = {!$Flow.CurrentDateTime}` if the field is currently blank (preserve first escalation only).
3. **Formula Fields (optional but recommended):**
   - `Time_To_Escalate_Hours__c`: `(Escalation_DateTime__c - CreatedDate) * 24` — returns hours.
   - `Time_Escalated_To_Close_Hours__c`: `(ClosedDate - Escalation_DateTime__c) * 24` — returns hours (only meaningful when case is closed).
4. **Report:** Build a Cases report surfacing `CreatedDate`, `Escalation_DateTime__c`, `ClosedDate`, and the two elapsed-time formula fields. Scope to the `Customer_Success_Team_Case` filter / relevant Case record type or queue.
5. **Backfill (if needed):** Confirm with stakeholder whether historical Cases need a retroactive escalation date (likely not possible without audit log review).

### Why This Should Fix It
Stamping a Date/Time value at the moment of escalation gives the reporting layer a concrete anchor point. With `CreatedDate`, `Escalation_DateTime__c`, and `ClosedDate` all present on the Case record, elapsed-time calculations become straightforward formula fields or report summary formulas — no external tooling required.

### Risks
1. **Escalation trigger ambiguity:** If escalation is not consistently driven by a single, automatable condition (e.g., agents sometimes skip the escalation step), the field may be inconsistently populated.
2. **Historical data gap:** Cases escalated before this solution is deployed will have no `Escalation_DateTime__c` value. Historical reporting will be incomplete.
3. **Business hours vs. calendar time:** Formula-field elapsed time uses calendar hours. If SLA reporting requires business-hours elapsed time, a more complex solution (Apex or Milestones/Entitlements) is needed.
4. **Multiple escalations:** If a Case can be escalated more than once, the Flow logic must decide whether to record only the first escalation or overwrite. Recommend first-escalation-only (null check).
5. **Flow conflicts:** Existing automation on the Case object must be reviewed to prevent conflicts.

### Rollback Plan
- Deactivate the Record-Triggered Flow in Production (no data loss; existing stamps remain).
- The custom field can be hidden from layouts/reports; deletion requires field removal from all references first.
- No destructive data changes from this solution — purely additive.
