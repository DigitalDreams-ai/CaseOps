## Salesforce Problem

### Confirmed Facts

- Salesforce provides a native "Recall Approval" action in the Approval History related list. This button is controlled by the Approval Process configuration ("Allow submitters to recall approval requests") AND by user-level permissions.
- The Recall option is only available to the original submitter of the approval request **or** to users with the "Modify All Data" permission, unless the approval process is configured to allow all submitters to recall.
- Tyler Kaplan previously had access to this Recall capability and no longer does, confirming a change occurred in either the approval process settings or the user's profile/permission set.
- Order Processing team members can still perform the recall, indicating that their profile or permission set grants either "Modify All Data," a broader object permission, or they are configured as delegates/approvers with recall rights.
- The affected Opportunity is: 006Ql00000eMscDIAS (Lightning URL confirmed in the ticket).
- The issue is scoped to the Clinical department; Tyler's role is RN (Registered Nurse).
- The immediate business impact is low for this specific record (OP already recalled it), but the systemic issue remains unresolved.

### Hypotheses

1. **Hypothesis 1 (Most Likely) — Profile or Permission Set change removed "Modify All" or a related permission:** A recent admin change to Tyler's profile or an assigned permission set removed a permission (e.g., "Modify All Data," "Modify All" on Opportunity, or a custom permission) that previously enabled the Recall button to appear. This would be consistent with the button disappearing without Tyler being explicitly told about a change.
2. **Hypothesis 2 — Approval Process "Allow Recall" setting was disabled:** The Approval Process on the Opportunity object may have had its "Allow submitters to recall approval requests" checkbox unchecked during a recent approval process edit. This would hide the Recall button for all non-admin users universally.
3. **Hypothesis 3 — Page Layout or Lightning App Builder change:** The Approval History related list component or a related action button may have been removed from the page layout or Lightning record page assigned to Tyler's profile. Less likely since OP users on the same object can still recall.
4. **Hypothesis 4 — Role hierarchy or sharing change:** A change to the role hierarchy or sharing settings may have reduced Tyler's visibility/edit access on Opportunity records to a level that no longer triggers the Recall button rendering. Less likely but possible.

### Likely Affected Metadata

- **Object:** Opportunity (`Opportunity`)
- **Feature:** Salesforce Approval Process — native "Recall" action in Approval History related list
- **Approval Process:** The active Approval Process on the Opportunity object (name unknown — needs retrieval via `ApprovalProcess` metadata or Setup UI). Key setting: `allowRecall` field on the `ApprovalProcess` metadata type.

### Why This Should Fix It

Salesforce's native Recall button in Approval History is governed by two controls: (1) the `allowRecall` flag on the Approval Process, and (2) the submitting user's permissions. Restoring whichever of these was changed to remove Tyler's access will re-expose the Recall option in the Approval History dropdown for Clinical/Nurse users, eliminating their dependency on OP for this task.

### Risks

1. **Over-permissioning:** If the permission removed was intentional (e.g., a deliberate tightening of access), restoring it without stakeholder sign-off could re-introduce a security or process control gap. **Mitigation:** Courtney has explicitly committed to stakeholder review before any change.
2. **Approval Process change is org-wide:** If the fix is to re-enable `allowRecall` on the Approval Process, it affects all submitters, not just Nurses. This could unintentionally restore recall access to roles/profiles where it was intentionally restricted. **Mitigation:** Prefer a permission set approach if possible to scope access more narrowly.
3. **Unknown scope of affected users:** Other Nurses may be silently affected and not have reported the issue. Any fix should be applied consistently across the affected role/profile group.

### Rollback Plan

- **If Permission Set change:** Remove the restored permission or un-assign the permission set from affected users. This is immediately reversible in Production Setup without a deployment.
- **If Approval Process `allowRecall` change:** Re-deactivate and re-edit the Approval Process to set `allowRecall = false`. Note: Approval Processes cannot be edited while active without deactivating first — factor this into change window planning.
