The reporter confirmed this is a regression — the same issue was previously resolved in ticket HEAL-30582 but has recurred.

These values should then appear correctly in the generated PDF document.

## Salesforce Problem

### Confirmed Facts

- The autolaunched flow **`Opportunity_to_Order`** is a subflow of the screen flow **`Screenflow - Confirmation to Create Pharmacy Order`**.
- The flow's **Get Opportunity Fields** element queries `DEA_Number__c`, `State_License_Number__c`, and `Patient_Driver_s_License__c` from the Opportunity.
- The flow's **Assign Order Fields** element does **not** assign those three queried values onto the new Order record — the assignment step is missing or incomplete for these specific fields.
- Because the fields are never written to the Order, the PDF template renders a blank DEA field (the template itself is not the problem).
- This is a regression from HEAL-30582, indicating a prior fix was either overwritten by a subsequent deployment or the fix was incomplete/not fully deployed to Production.
- The issue affects "a small team or multiple records" per the reporter's scoping.
- A workaround exists (reporter can continue working around it), but the issue is unresolved at the flow level.

### Hypotheses

1. **Primary (confirmed by Sean Bingham):** The `Assign Order Fields` element in the `Opportunity_to_Order` autolaunched flow is missing variable assignments for `DEA_Number__c`, `State_License_Number__c`, and `Patient_Driver_s_License__c`. The values are queried but never mapped to the Order record fields.
2. **Regression cause:** The fix applied in HEAL-30582 was likely overwritten by a subsequent flow deployment that did not include the corrected assignment mappings (e.g., a sandbox-to-production deployment that used an older version of the flow).
3. **Secondary (less likely):** The Opportunity records in question may not have these fields populated at order creation time — however, Sean's investigation confirms the flow logic itself is the gap, not the source data.

## Solution Plan

### Classification

- **Engineering escalation**
- **Escalation reason:** Requires modification of a Salesforce Flow (`Opportunity_to_Order`) — adding missing variable assignments in the `Assign Order Fields` element. This is a metadata/automation change that must be built, tested in Sandbox, and deployed to Production.

### Proposed Change

In the autolaunched flow **`Opportunity_to_Order`**:
1. Open the `Assign Order Fields` assignment element.
2. Add the following variable assignments (mapping from the queried Opportunity record variable to the new Order record variable):
   - `{!Order.DEA_Number__c}` ← `{!Get_Opportunity_Fields.DEA_Number__c}`
   - `{!Order.State_License_Number__c}` ← `{!Get_Opportunity_Fields.State_License_Number__c}`
   - `{!Order.Patient_Driver_s_License__c}` ← `{!Get_Opportunity_Fields.Patient_Driver_s_License__c}`
3. Save a new version of the flow and activate it.
4. Test by generating a pharmacy order from an Opportunity with all three fields populated and verifying the PDF output.
5. Deploy the updated flow to Production via Gearset (or org-standard deployment tooling).

### Why This Should Fix It

The root cause is a missing assignment step — the flow queries the correct fields but never writes them to the Order. Adding those three assignments ensures that when the Order record is created by the flow, it carries the DEA, State License, and Driver's License values from the source Opportunity. The PDF template will then have populated fields to render.

### Risks

- **Regression risk:** This issue has already regressed once from HEAL-30582. Engineering should document exactly which flow version contains the fix and ensure deployment procedures include a check to prevent an older flow version from overwriting the corrected one in future deployments.
- **Field API name mismatch:** If the Order object uses different API names for these fields than the Opportunity, the assignment will silently fail. Engineering must verify field API names exist on both objects.
- **Active flow version management:** Salesforce Flows require explicit activation of the new version; deploying without activating will not resolve the issue.
- **Partial data:** If an Opportunity does not have these fields populated at order creation time, the Order will still have blank fields — this is expected behavior, but should be documented to avoid future false bug reports.

### Rollback Plan

If the updated flow causes unintended behavior:
1. In Flow Builder, deactivate the new version of `Opportunity_to_Order`.
2. Activate the previous version.
3. No data loss risk — this flow creates Order records; rollback only affects future order creation, not existing records.
