## Salesforce Problem

### Confirmed Facts

- The "Send Patient Agreements" feature exists in the 10X Health Salesforce org and is used by the Clinical department.
- The feature sends an email to patients containing a link to sign consent documents on a patient-facing portal.
- Multiple patient Account records are affected (at least 3 confirmed, described as "numerous").
- Patients are able to receive the email (delivery is not the issue).
- The failure occurs at the step where portal displays consent documents after patient authentication — red error "cannot sign consents" appears.
- The Clinical team has a workaround available.
- The issue was reported 2026-05-13 and is categorized under "Salesforce - Customer Accounts."
- **ROOT CAUSE CONFIRMED:** Portal users lacked object-level READ permission on `Patient_Agreement__c` custom object. When portal attempted to retrieve and display agreement records after authentication, access was denied due to insufficient profile permissions.

### Root Cause Identified

**Exact location of error:** `Portal - Authenticated User` profile is missing READ permission on `Patient_Agreement__c` custom object.

When a patient logs into the consent portal, the portal code attempts to query and display records from `Patient_Agreement__c` linked to that patient's Account. Because the profile lacked READ access, the query fails with a permission-denied error, manifesting as the red error message "cannot sign consents" visible to the end user.

This is **not** an authentication issue (login succeeds), **not** a Flow/automation issue (records are generated correctly), and **not** an Experience Cloud user provisioning issue. The specific failure point is the permission check on the `Patient_Agreement__c` object when the portal UI renders the consent document list.

## Solution Plan

### Classification

- Support-resolvable or Engineering escalation: **Support-resolvable**
- Reason: Root cause is a configuration issue (missing READ permission on custom object) in a user profile — no Flow changes, Apex changes, or third-party integration work required.

### Proposed Change

**Grant READ permission on `Patient_Agreement__c` and `Patient_Agreement_Configuration__c` to the `Portal - Authenticated User` profile.**

The fix is a simple profile permission update:
1. Open profile `Portal - Authenticated User` in Setup
2. Add object-level READ permission for `Patient_Agreement__c`
3. Add object-level READ permission for `Patient_Agreement_Configuration__c` (required for consent type references)
4. Deploy to Production

This allows portal users to query and display agreement records when the consent portal UI renders the signing form.
