## Issue Understanding

### Observed Behavior

Mya has built three Salesforce reports (Lab Review, Genetic Test Clarity Calls, Peptide Express Consultations) using the **Opportunity Field History** report type. These reports track the number of days between an opportunity stage changing from "New" or "Needs Scheduling" to "Scheduled," effectively measuring Time to Schedule (TTS) for consultation opportunities.

The reports are functioning correctly for the core TTS metric after QA validation. However, two fields are missing:
1. **Billing State** (account-level field) — not available on the Opportunity Field History report type.
2. **Main Practitioner** (account-level field) — not available on the Opportunity Field History report type.

There are corresponding fields at the **Opportunity level** (Main Practitioner and State), but those fields are **blank** — no data is being populated into them from the related Account record.

As a workaround, Mya is manually cross-referencing Salesforce opportunity data with EHR appointment data in an Excel workbook to match records and analyze TTS by state and provider. This introduces margin for error in match formulas.

### Expected Behavior

The three TTS reports should each include columns for:
- **Billing State** (from the patient's Account)
- **Main Practitioner** (from the patient's Account)

These columns would allow the Clinical department to filter and analyze average TTS by state and by provider directly within Salesforce, eliminating the need for the Excel cross-reference workbook. Priority order: Lab Review → Genetic Test Clarity Calls → Peptide Express Consultations.

### Acceptance Criteria

1. The Lab Review TTS report (`00OQl000006nwH7MAI`) includes a **Billing State** column with populated data.
2. The Lab Review TTS report includes a **Main Practitioner** column with populated data.
3. The same columns are added to the Genetic Test Clarity Calls report (`00OQl000006nyxFMAQ`) and the Peptide Express Consultations report (`00OQl000006nz0TMAQ`).

5. The reports can be used to calculate average TTS segmented by Billing State and Main Practitioner without requiring an external Excel document.

## Salesforce Problem

### Confirmed Facts

1. The reports use the **Opportunity Field History** standard report type. This report type is scoped to the `OpportunityFieldHistory` object and does not natively expose Account-level fields (e.g., Billing State, Main Practitioner).
2. There are **Opportunity-level fields** for "Main Practitioner" and "State" (likely custom fields), but they are **currently blank** — no data is being copied into them from the related Account.
3. The **core TTS calculation** (days between stage = "New"/"Needs Scheduling" and stage = "Scheduled") is working correctly per Mya's QA validation.
4. Three separate reports exist for three product lines: Lab Review, Genetic Test Clarity Calls, and Peptide Express Consultations — all using the same report type pattern.
5. Sean confirmed (2026-05-08) that he found "existing processes for this that are problematic," suggesting there is already some automation touching these opportunity fields that is not functioning as intended.
6. The workaround currently in use is a manual Excel cross-reference between Salesforce opportunity data and EHR appointment data — functional but error-prone.

### Hypotheses

1. **Primary hypothesis — Blank Opportunity fields due to missing or broken automation:** The Opportunity-level "Main Practitioner" and "State" fields were likely intended to be populated via a Flow, Process Builder rule, or Apex trigger that copies values from the parent Account at opportunity creation or update. This automation is either absent, broken, or only partially firing (e.g., fires on manual creation but not on auto-creation of lab consultation opportunities).
2. **Secondary hypothesis — Report type limitation is the core blocker:** Even if Opportunity fields were populated, the Opportunity Field History report type may not expose all related Opportunity fields cleanly. The fix may require either (a) populating the Opportunity fields so they surface in the existing report type, or (b) creating a custom report type that joins Opportunity Field History with Opportunity (and by extension Account) fields.
3. **Tertiary hypothesis — "Existing problematic processes" reference:** Sean's comment suggests he discovered an existing Flow or Process that was meant to sync Account → Opportunity data but has a bug (e.g., null handling, trigger order issue, or fires before Account data is set on auto-created opportunities). This may need to be fixed or replaced rather than adding net-new automation.
4. **Custom report type as alternative path:** If fixing the blank Opportunity fields is complex, a custom Salesforce report type built on Opportunity Field History with a joined Opportunity relationship might expose account-lookup fields — though standard Salesforce report type builder has limits on multi-object joins from field history objects.

  - Peptide Express Consultations (`00OQl000006nz0TMAQ`)

### Why This Should Fix It

Populating the Opportunity-level Main Practitioner and State fields via a corrected or new Flow ensures that the data exists at the Opportunity level. Since the Opportunity Field History report type can surface Opportunity-level fields (even if not Account-level fields directly), this approach works within the existing report type constraint. Once fields are populated, adding them as columns to the three reports is a straightforward report edit.

### Risks

1. **Existing "problematic" automation:** Fixing or replacing the existing automation without fully understanding it could cause unintended side effects on other Opportunity workflows.
2. **Auto-created opportunity timing:** If Account data is set after the Opportunity is auto-created, a create-trigger Flow may fire before the Account relationship is established, resulting in still-blank fields. A scheduled or update-trigger path may be needed.
3. **Historical data gap:** Existing Opportunity records will still have blank fields unless a backfill is performed — historical TTS report rows will be missing state and practitioner data.
4. **Report type limitation edge case:** If the Opportunity Field History report type does not expose custom Opportunity fields, a Custom Report Type may be needed, which changes the report structure.
5. **Three reports to update:** Changes must be replicated across all three product line reports — risk of inconsistency if done manually.

### Rollback Plan

- If the new/modified Flow causes issues: deactivate the Flow in Production immediately. Opportunity fields will return to blank state (original condition) with no data loss.
- If a Custom Report Type is added and causes confusion: it can be deactivated or deleted without affecting existing reports (if reports remain on the original type).
- No destructive changes to existing records are proposed — rollback risk is low.
