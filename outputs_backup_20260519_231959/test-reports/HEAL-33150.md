## Test Cases

### Test Case 1: Outbound Email from Cx Case Record Includes Thread Token in Reply-To Header

**Objective:** Confirm that when an agent sends an email from a Cx Case Record, the outbound email's `Reply-To` header resolves to the active Email-to-Case routing address (not an OWA or user address), and the case thread token is present.

**Steps:**
1. Log in to a Sandbox org that mirrors Production Email-to-Case configuration.
2. Open an existing Cx Case Record.
3. Use the Send Email action to compose and send an email to a test recipient.
4. In the test recipient's inbox, inspect the raw email headers.
5. Observe: `Reply-To` header should contain the Email-to-Case routing address (e.g., `case@<routing-domain>.salesforce.com` or equivalent On-Demand endpoint), not a user address or OWA.
6. Observe: The case thread token (`ref:...`) should be present in both the email body and the `Reply-To` header parameters.

**Result:** ✗ Blocked

**Notes:** This test case cannot be executed. No Sandbox with a mirrored Email-to-Case configuration has been provisioned for this issue. Engineering must first confirm whether a suitable Sandbox exists and perform this test as part of their root cause audit. The reporter has confirmed the thread token is present in the outbound email body, but the `Reply-To` header destination has not been inspected.

### Test Case 2: Reply from Outlook Logs as EmailMessage on Cx Case Record Activity Timeline

**Objective:** Confirm that when a recipient replies to a case email from Outlook, the reply is ingested by Salesforce Email-to-Case and appears as a new EmailMessage record on the originating Cx Case Record's Activity timeline.

**Steps:**

2. From the test recipient's Outlook inbox, reply to the email sent in Test Case 1.
3. Wait up to 5 minutes for Salesforce Email-to-Case processing.
4. Navigate to the originating Cx Case Record in Salesforce.
5. Observe: The recipient's reply should appear as a new EmailMessage entry in the case Activity timeline, linked to the correct case.

**Result:** ✗ Blocked

**Notes:** Blocked by the same dependency as Test Case 1. This is the primary acceptance criterion for HEAL-33150 and represents the core end-to-end validation. Without a configured Sandbox and Engineering-confirmed routing fix in place, this test cannot be executed. This is also the exact failure scenario described by reporter Noah Scott.

**Steps:**

2. From the Cx Case Record, send an initial email to a test recipient.
3. Have the recipient reply from Outlook.

5. Have the recipient reply again from Outlook.
6. Navigate to the Cx Case Record Activity timeline.
7. Observe: All four messages (2 outbound agent emails, 2 inbound recipient replies) should appear in chronological order on the Activity timeline, all linked to the same case.

**Result:** ✗ Blocked

**Notes:** Blocked pending resolution of Test Cases 1–3. This test case maps directly to the reporter's complaint that "the full back-and-forth with Tyler is not reflected in Salesforce, and the case does not show the full communication history as a proper paper trail."
