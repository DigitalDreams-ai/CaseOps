### Test Case 2: Record-Triggered Flow — Escalation Timestamp Stamped on First Escalation Event

**Objective:** Confirm that the Record-Triggered Flow correctly stamps `Escalation_DateTime__c = NOW()` when the agreed escalation condition fires for the first time on a Case record.

**Steps:**
1. In Sandbox, create a new Case record with status/queue reflecting a Tier 1 (non-escalated) state.
2. Confirm `Escalation_DateTime__c` is null on the new record.
3. Trigger the escalation condition (exact action TBD — set `IsEscalated` = true, change Status to escalation value, or reassign to Tier 2 queue — **must be confirmed with Noah Scott before build**).
4. Save the record.
5. Observe: `Escalation_DateTime__c` is populated with a Date/Time value matching the current timestamp (within a reasonable tolerance of ±1 minute).

**Result:** ✗ BLOCKED

**Notes:** Escalation trigger mechanism is unconfirmed — this is the highest-priority unknown. Flow cannot be built or tested until Engineering confirms the trigger with Noah Scott. Null check logic must be included in the Flow to ensure re-triggering the escalation condition does not overwrite the original timestamp.

### Test Case 3: Record-Triggered Flow — Null Check Prevents Overwrite on Subsequent Escalation Events

**Objective:** Confirm that if the escalation condition fires more than once on a Case, `Escalation_DateTime__c` retains the original timestamp and is not overwritten.

**Steps:**
1. Using a Case record from Test Case 2 (which already has `Escalation_DateTime__c` populated), note the existing timestamp.
2. De-escalate the Case (reverse the escalation condition) and then re-trigger the escalation condition a second time.
3. Save the record.
4. Observe: `Escalation_DateTime__c` value remains unchanged from the first stamp.

**Result:** ✗ BLOCKED

**Notes:** Flow must include a decision element checking `{!Record.Escalation_DateTime__c} IS NULL` before stamping. If this null check is absent, the first escalation timestamp will be overwritten, corrupting elapsed-time calculations.

### Test Case 4: Formula Field — Time to Escalate Calculates Correctly

**Objective:** Confirm that `Time_To_Escalate_Hours__c` formula field returns the correct elapsed hours between `CreatedDate` and `Escalation_DateTime__c`.

**Steps:**
1. Locate a Case in Sandbox where `Escalation_DateTime__c` has been stamped (from Test Case 2).
2. Note the `CreatedDate` and `Escalation_DateTime__c` values.
3. Manually calculate the expected elapsed hours: `(Escalation_DateTime__c - CreatedDate) * 24`.
4. Open the Case record and observe the value in `Time_To_Escalate_Hours__c`.
5. Observe: formula field value matches the manually calculated expected value.

**Result:** ✗ BLOCKED

**Notes:** Optional field per internal notes, but required for the requester's stated reporting goal. If business-hours elapsed time is required instead of calendar time, this formula will be insufficient — Entitlements/Milestones or Apex would be required (significantly larger effort). This must be clarified with Noah Scott.

### Test Case 5: Formula Field — Time Escalated to Close Calculates Correctly

**Objective:** Confirm that `Time_Escalated_To_Close_Hours__c` formula field returns the correct elapsed hours between `Escalation_DateTime__c` and `ClosedDate`.

**Steps:**
1. Locate a Case in Sandbox where `Escalation_DateTime__c` is stamped and the Case has been closed (`ClosedDate` is populated).
2. Note the `Escalation_DateTime__c` and `ClosedDate` values.
3. Manually calculate the expected elapsed hours: `(ClosedDate - Escalation_DateTime__c) * 24`.
4. Open the Case record and observe the value in `Time_Escalated_To_Close_Hours__c`.
5. Observe: formula field value matches the manually calculated expected value.

**Result:** ✗ BLOCKED

**Notes:** Formula will return null or behave unexpectedly if `Escalation_DateTime__c` is null (non-escalated cases). Engineering should confirm null-handling behavior in the formula definition.

### Test Case 6: Formula Field — Null Handling for Non-Escalated Cases

**Objective:** Confirm that formula fields return null (blank) rather than an error or zero for Cases where `Escalation_DateTime__c` has not been stamped.

**Steps:**
1. Locate a Case in Sandbox where `Escalation_DateTime__c` is null (case was never escalated).
2. Observe the values of `Time_To_Escalate_Hours__c` and `Time_Escalated_To_Close_Hours__c`.
3. Observe: both fields display blank/null — no error, no zero, no negative value.

**Result:** ✗ BLOCKED

**Notes:** This is important for report integrity. Non-escalated cases should not appear to have a zero-hour escalation time, which could distort aggregate metrics.
