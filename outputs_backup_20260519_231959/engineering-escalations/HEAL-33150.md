# Engineering Handoff

## Engineering Message

**Issue:** HEAL-33150 — Cx Case Record emails do not capture replies from external mail clients (Outlook) back into Salesforce
**Problem:** When an agent sends an email from a Cx Case Record in Salesforce, the outbound email is delivered successfully and includes the expected case thread token. However, when the recipient replies from Outlook (or any external mail client), the reply is NOT routed back into Salesforce. No EmailMessage record is created on the case, the case Activity timeline remains incomplete, and there is no full communication paper trail. The issue affects multiple teams and large numbers of records in the Customer Support department. Other Case record types may not be affected, but this has not been confirmed.
**Potential Fix:** Identify and correct the Email-to-Case routing address configuration for the Cx Case Record type. Ensure the `Reply-To` header on outbound case emails from this record type resolves to an active, monitored Email-to-Case routing address (not an Org-Wide Email Address or user mailbox). If no routing address is configured for this record type, create and activate one mirroring the working configuration of other Case record types in the org.

## Root Cause

Not yet confirmed — requires direct org inspection. Most likely failure point is one or more of the following:

1. **Primary (most likely):** The outbound email from Cx Case Records has its `Reply-To` header set to an address that is NOT an active Email-to-Case routing address (e.g., it resolves to an Org-Wide Email Address or a user's personal/mailbox address). Outlook replies are delivered to that address, which is not monitored by Salesforce Email-to-Case, so the reply is never ingested and no EmailMessage is created.

2. **Secondary:** An Email-to-Case routing address exists but is inactive, misconfigured, or not assigned to the Cx Case Record type. The routing address may exist for other Case record types but was never set up or was deactivated for Cx Case Records.

3. **Tertiary:** The Cx Case Record type page layout uses a custom Send Email quick action that does not correctly set the `Reply-To` header to the Email-to-Case routing address, even though the thread token is embedded in the email body. Body-only token presence is insufficient if `Reply-To` does not point to the Email-to-Case endpoint.

## Affected Component

- Metadata/code component: Email-to-Case Setup (Routing Addresses), Case Record Type (`Cx Case Record`), Case Page Layout(s) assigned to Cx Case Record type, Send Email Quick Action (`Case.SendEmail` or custom variant on Case), Org-Wide Email Addresses, EmailMessage object, Email Services (if On-Demand Email-to-Case)
- Element/rule/process, if applicable: Email-to-Case routing address for Cx Case Record type; `Reply-To` header configuration on outbound case emails; possibly Case Assignment Rules affecting inbound email processing

## Potential Fix

1. **Audit all Email-to-Case routing addresses** in Production Setup (Setup > Email-to-Case > Routing Addresses). Document each: routing email address, active/inactive status, record type assignment, case origin, SSL/TLS settings, and whether it is On-Demand or standard.
2. **Identify the routing address (if any) assigned to the Cx Case Record type.** If none exists, create one. If one exists but is inactive, activate it.
3. **Confirm the `Reply-To` header** on outbound emails from a Cx Case Record — send a test email and inspect raw headers. The `Reply-To` must be the Email-to-Case routing address (e.g., `case-xxxxxx@[org].case.salesforce.com` for On-Demand, or the org's monitored routing inbox for standard Email-to-Case).
4. **Compare against a working Case record type** — identify at least one other Case record type where email threading works correctly and document its full Email-to-Case configuration as the baseline.
5. **If using On-Demand Email-to-Case:** Confirm the email service endpoint address is correctly configured and that SMTP forwarding (if applicable) is active for the Cx Case Record routing address.
6. **If using a custom Send Email quick action on the Cx Case Record page layout:** Compare it against the standard `Case.SendEmail` action and confirm it is not overriding or omitting the `Reply-To` Email-to-Case address.
7. **Validate fix in Sandbox** before applying to Production: send email from Cx Case Record → reply from Outlook → confirm EmailMessage record is created and appears in case Activity timeline.
