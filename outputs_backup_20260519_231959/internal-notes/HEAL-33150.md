## Root Cause

The most likely root cause is a **missing, inactive, or misconfigured Email-to-Case routing address for the Cx Case Record type**. When an agent sends an email from a Cx Case Record, the outbound email includes the case thread token (confirming standard Email-to-Case mechanics are partially in place), but when the recipient replies from Outlook, the reply is not ingested back into Salesforce.

This points to one or more of the following specific failure modes:

1. **Primary hypothesis:** The outbound email's `Reply-To` header is not set to an active Email-to-Case routing address. The thread token alone is not sufficient — replies must land in a mailbox that is actively monitored by Salesforce Email-to-Case. If the `Reply-To` or `From` address on outbound Cx Case Record emails resolves to an Org-Wide Email Address (OWA) or a user's personal address rather than the Email-to-Case routing address, Outlook replies will go to that address and never be ingested by Salesforce.

2. **Secondary hypothesis:** An Email-to-Case routing address exists but is not correctly assigned to or configured for the Cx Case Record type. Other Case record types in the org may have working routing addresses while the Cx Case Record type was either never configured or was configured with a routing address that has since become inactive.

3. **Tertiary hypothesis:** A record-type-specific page layout or custom email quick action on the Cx Case Record type overrides the standard Send Email behavior and does not correctly inject the thread token into the `Reply-To` header (only the body), causing Outlook's reply to target the wrong address.

The fact that the thread token *is* present in the outbound email (confirmed by reporter) narrows the issue: the token is being generated, but the reply routing is failing. The most actionable investigation step is to inspect the Email-to-Case routing address configuration and confirm what address appears in the `Reply-To` header of outbound Cx Case Record emails.

## Solution Or Escalation

**Decision: Escalate to Engineering.**

This issue requires direct access to the Salesforce org's Email-to-Case Setup configuration, routing address records, and outbound email headers — none of which can be fully diagnosed or remediated through a Sandbox metadata-only fix without first confirming the exact misconfiguration in Production. The fix itself (once root cause is confirmed) is likely a declarative configuration change (activating/correcting a routing address, updating the Reply-To on the email action, or assigning an existing routing address to the Cx Case Record type), but the following factors require Engineering involvement:

1. **Email-to-Case routing address configuration** is managed in Salesforce Setup and may involve email server/SMTP forwarding configuration outside of standard metadata deployment (especially for On-Demand Email-to-Case).
2. **Confirming the Reply-To header** on outbound emails requires either sending a live test email and inspecting raw headers, or reviewing Email-to-Case routing address settings in the Production org — neither is a Sandbox-only action.
3. **Scope is broad** — multiple teams and large numbers of records are affected. A misconfigured routing address fix could have unintended side effects on case assignment rules, record type assignment on inbound email, and existing open cases.
4. **Unknown org configuration** — we have not confirmed whether other Case record types are working correctly (to establish a baseline), whether On-Demand vs. standard Email-to-Case is in use, or whether the Cx Case Record type has ever had working email threading.
5. The reporter's provided report link (`https://10xhealth.lightning.force.com/lightning/r/Report/00ORh000009iDyrMAE/view?queryScope=userFolders`) and the five attached screenshots have not been reviewed by the analyst and must be reviewed by Engineering to confirm scope and identify specific affected records.

Engineering should begin with a read-only audit of: (a) all active Email-to-Case routing addresses and their record type assignments, (b) the Send Email quick action configuration on the Cx Case Record type page layout, (c) raw headers of a test outbound email from a Cx Case Record, and (d) a comparison against at least one working Case record type.

## Risks Or Follow-Up

1. **Risk — Broad scope impact:** The issue affects multiple teams and large numbers of records. Any change to Email-to-Case routing addresses or email action configuration could affect all open and future Cx Case Records. Engineering must test in a Sandbox that mirrors Production before deploying.
2. **Risk — Inbound email routing side effects:** Fixing the routing address may cause previously unlogged replies to resurface or trigger case assignment rules unexpectedly. Existing open cases with incomplete email threads will not be retroactively populated.
3. **Risk — On-Demand Email-to-Case SMTP dependency:** If the org uses On-Demand Email-to-Case, any new or corrected routing address requires SMTP forwarding to be configured at the email server level, which may involve IT or infrastructure work outside of Salesforce Setup.
4. **Follow-up:** Once fix is confirmed, consider auditing all other Case record types to ensure their Email-to-Case configurations are also correct and consistent.
5. **Follow-up:** Review reporter's provided Salesforce report and attached screenshots to confirm scope and identify any additional affected record types or edge cases.
6. **Follow-up:** Confirm with reporter whether the issue also affects external client emails (not just internal user Tyler) — the description mentions both internal and external messaging.
