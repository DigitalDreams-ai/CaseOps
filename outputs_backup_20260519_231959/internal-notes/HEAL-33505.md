# Internal Notes

## Jira Issue

- Key: HEAL-33505
- Summary: Is there a way for "Opportunity Field History" to capture activity on "Order Notes" edits/updates to that field?
- Reporter: Sarah Phonthaphanh

---

## Root Cause

The **Order Notes** field on the Opportunity object has not been enabled for **Field History Tracking** in Salesforce. Salesforce's native Field History Tracking must be explicitly opted in per field via **Setup → Object Manager → Opportunity → Set History Tracking**. Since "Order Notes" was never added to the tracked field list, no changes to it are written to the `OpportunityFieldHistory` object — resulting in a silent gap in the audit trail.

**Secondary risk factors to confirm before fixing:**
1. **Field type:** If "Order Notes" is a **Long Text Area or Rich Text Area**, Salesforce's platform does not support native Field History Tracking for that data type. This would require an Engineering escalation (custom Flow or Apex audit solution).
2. **20-field limit:** Salesforce allows a maximum of 20 fields tracked per object. If the Opportunity object is already at 20, a currently tracked field must be removed before "Order Notes" can be added.

---

## Solution Or Escalation

**Path A — Support-resolvable (most likely, pending field type confirmation):**
If "Order Notes" is a supported field type (Text, Text Area, Picklist, Number, Date, etc.) and the tracked field count is under 20:
- Navigate to **Setup → Object Manager → Opportunity → Fields & Relationships → Set History Tracking**
- Check the box next to **Order Notes**
- Save
- No code, no Flow, no deployment pipeline required — declarative admin configuration only

**Path B — Engineering escalation (if field is Long Text Area / Rich Text Area):**
- Native Field History Tracking is a platform-level limitation for these field types
- Engineering must build a Record-Triggered Flow (or Apex trigger) that fires on Opportunity Update, detects a change to the "Order Notes" field, and writes a record to a custom audit object capturing: Opportunity ID, user, timestamp, old value, new value
- This is not a Support-resolvable change

**Recommended immediate action for operator:**
Run the following SOQL in the org's Developer Console or Workbench to determine field type and current tracking status:

```sql
SELECT QualifiedApiName, DataType, IsFieldHistoryTracked
FROM FieldDefinition
WHERE EntityDefinition.QualifiedApiName = 'Opportunity'
AND Label = 'Order Notes'
```

And to check how many fields are currently tracked:

```sql
SELECT QualifiedApiName, DataType
FROM FieldDefinition
WHERE EntityDefinition.QualifiedApiName = 'Opportunity'
AND IsFieldHistoryTracked = true
```

## Risks Or Follow-Up

1. **Long Text Area / Rich Text Area limitation** — would require Engineering escalation and a custom solution; do not attempt to enable native tracking for these types
2. **20-field tracking limit** — if at limit, a tracked field must be removed; this requires coordination with affected stakeholders to avoid silently breaking an existing audit trail
3. **No retroactive history** — enabling tracking will only capture future changes; historical edits to "Order Notes" are permanently unrecorded and cannot be recovered
4. **Page layout visibility** — confirm the Opportunity Field History related list is present on all page layouts used by the Clinical department and other affected teams
5. **Storage impact** — minor but worth noting for high-volume orgs
