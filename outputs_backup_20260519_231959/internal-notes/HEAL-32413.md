## Root Cause

The issue is caused by `finalApprovalRecordLock = true` on the `Opportunity_Approval_Process` approval process in Salesforce. The approval process entry criteria includes Opportunities with Record Type = Labs, and the record editability is set to `AdminOrCurrentApprover`. Once a Labs Opportunity reaches final approval, Salesforce platform-enforces a full record lock. Nurse-level users (non-admin, non-approver) receive the standard record-locked error when attempting any edit — including updating the Lab Vendor field, which is a required step in the clinical lab draw workflow.

This is NOT caused by:
- A validation rule (confirmed by Sean Bingham, 2026-03-24)
- A flow or process builder (confirmed by Sean Bingham, 2026-03-24)
- A profile or permission set gap (no permission set change can override an approval process record lock)

---

## Solution Or Escalation

**Engineering escalation required.**

Sean Bingham designed a solution on 2026-03-25:
1. Create a new approval process `Opportunity_Approval_Process_Labs` scoped to Labs record type only, with `finalApprovalRecordLock = false` and processOrder = 1. This process should mirror the existing Labs approval step (`Labs_Approval_Process_Steps`) — same approvers, email alerts, and field updates.
2. Update `Opportunity_Approval_Process`: remove Labs from entry criteria, set processOrder = 2. `finalApprovalRecordLock` remains `true` for all other Opportunity types that still pass entry criteria.
3. Confirm `Opportunity_Approval_Process` (Genetic variant) remains processOrder 3 and is unchanged.
4. Deploy Sandbox → Production via Gearset.

This was escalated to Engineering via Bug #20941 (10X Health Engineering). As of 2026-04-07, Nathaniel Cooper tagged Ali Saeed on the bug — no confirmed resolution or Sandbox deployment has been communicated back to the customer.
