# Solution Plan: HEAL-33116 — Adding "Scheduled by" into Consult Opp

---

## 1. Problem Statement

The Operations team had no way to distinguish between Consultation Opportunities that were self-scheduled by patients via Calendly versus those that required manual intervention by an internal scheduler. No attribution field existed on the Opportunity object to capture this information. This created a data quality and operational visibility gap — schedulers could not be credited for their work, and self-scheduled bookings looked identical to manually scheduled ones in Salesforce reporting.

**Who is affected:** Operations team, scheduling staff (Kershon Stephens, Margo Michel, Monica Lewis, Jemimah Aquino, Stephannie Opeda, Mariella Medina), Operations Manager, and any reporting consumers relying on Consultation Opportunity data.

## 3. Proposed Fix (Minimal Viable)

### Why This Fix and Not a Larger One

The problem is purely a missing picklist field on Opportunity. We do not need a new object, a junction record, a complex automation, or a redesigned scheduling workflow. The fix is: add one picklist field, place it on the relevant layouts, and grant FLS to the roles that use it. Everything else (Calendly automation, default values, required-field validation) is either deferred by the stakeholder or out of scope for the stated problem.

We are NOT:
- Building Calendly automation now (explicitly deferred by Sarah).
- Creating a required field or validation rule (premature — auto-population is not yet available, forcing manual entry before automation exists would block record saves).
- Adding the field to all Opportunity layouts (only the six Consultation-related layouts are in scope — adding it everywhere would pollute unrelated record types).

- **Components to package:** `Opportunity.Scheduled_By__c` field, 6 page layouts, 4 permission set FLS entries.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Production promotion fails due to metadata conflict | Low | Medium | Gearset will surface errors pre-deploy; resolve in Sandbox before retrying |
| Incorrect permission sets — some users can't see field | Medium | Low | Field is additive; missing access means blank field, not a broken record. Fixable post-deploy with FLS update |
| Picklist value typos in Production | Low | Low | Verify against Sandbox post-deploy; correct via Setup > Picklist Value Sets |
| Scheduler names change, picklist becomes stale | Medium (over time) | Low | Document as maintenance item; no immediate risk |
| Calendly automation gap causes blank field accumulation | High | Medium | Accepted risk — deferred by stakeholder. Consider adding field to list views so blanks are visible to Operations as a data quality monitor |

### Rollback Steps

Because this is a purely additive change (new field, layout additions, FLS grants), rollback impact is minimal:

1. **To remove the field from layouts:** Edit each of the six Opportunity page layouts in Production Setup and drag `Scheduled_By__c` back to the available fields palette. No data is lost.
2. **To revoke FLS:** Edit each permission set, find `Scheduled_By__c` under Object Settings > Opportunity, and set to No Access. Users will no longer see the field.
3. **To delete the field entirely:** Setup > Object Manager > Opportunity > Fields & Relationships > `Scheduled_By__c` > Delete. WARNING: This permanently deletes any data entered in the field. Only do this if the feature is being abandoned entirely. All existing values will be lost and cannot be recovered from the field — ensure a data export is taken first if any records have been populated.
4. **Recommended rollback scope:** In most scenarios, rollback should only go as far as removing from layouts and revoking FLS — leave the field in place to preserve any data entered and to avoid a full delete/recreate cycle if the feature is reinstated.
