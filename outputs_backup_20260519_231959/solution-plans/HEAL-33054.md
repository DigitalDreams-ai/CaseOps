# Solution Plan: HEAL-33054 — Cardone Ventures Employee Checkbox

---

## 1. Problem Statement

The Wellness Department (reporter: Ashlee Edwards) had no way to flag whether an Account (Patient) record belongs to a Cardone Ventures employee, and no way to surface that flag on related Opportunity records for reporting. This was a net-new feature request — no existing field, automation, or workaround existed. Affected users: Wellness/Revenue Ops team, any staff running reports on CV employee patients.

---

## 2. Escalation Decision

**Support-Resolvable — No Engineering escalation required.**

The entire solution is within standard Salesforce Admin scope:
- Two new custom fields (one checkbox, one formula checkbox)
- FLS entries on existing Permission Sets
- Page layout additions

No Apex, Flow, trigger, or platform-event logic was needed. The formula field pattern (`Account.Cardone_Ventures_Employee__c` on Opportunity) is a native declarative capability. Sean Bingham (Salesforce Admin) resolved this fully in Sandbox.

## 3. Proposed Fix (Minimal Viable)

### Step 1 — Create Account Checkbox Field
- Object: `Account`
- Field Type: Checkbox
- API Name: `Cardone_Ventures_Employee__c`
- Label: `Cardone Ventures Employee`
- Default Value: `false`
- Description: "Indicates this Account belongs to a Cardone Ventures employee. Managed manually by Revenue Ops."

### Step 2 — Create Opportunity Formula Field
- Object: `Opportunity`
- Field Type: Formula (Checkbox)
- API Name: `Cardone_Ventures_Employee__c`
- Label: `Cardone Ventures Employee`
- Formula: `Account.Cardone_Ventures_Employee__c`
- Description: "Read-only. Derived from the related Account. Cannot be overridden at the Opportunity level."

### Step 3 — Apply FLS on Permission Sets

### Step 5 — Add to Opportunity Page Layouts

- Section: Opportunity Information
- Placement: After `AccountId`
- Render mode: Read-only (formula field)

### Why This Is the Minimal Viable Fix

- **Formula field instead of automation:** We could have used a Flow or trigger to copy the Account value onto the Opportunity as a stored field, but a formula field achieves the same result with zero automation overhead, no bulk-update risk, and no stale-data problem. The formula is always live.
- **No new object or junction:** The request is a single boolean attribute on an existing object hierarchy (Account → Opportunity). No new object, report type, or intermediate record is needed.
- **No automated backfill tooling:** Ivelisse Santos confirmed the team will manually identify CV employee Accounts. Building a data loader script or Flow to auto-identify them would require a source of truth that does not exist in Salesforce — manual is the correct and minimal approach here.
- **Read-only on Opportunity is by design:** Making the Opportunity field a formula (rather than editable) prevents data drift between Account and Opportunity values. This directly satisfies the stated requirement that "master is on the account level."
- **FLS scoped to existing Permission Sets only:** No new Permission Sets were created. The existing set covers all user groups who need visibility.

## 5. Dependencies

1. **Account must be the master record:** The Opportunity formula depends on `Account.Cardone_Ventures_Employee__c` existing on Account first. Account field must deploy before or simultaneously with the Opportunity formula field. Gearset handles this ordering if both are in the same deployment package.
2. **Opportunity must be linked to an Account:** The formula returns `false` (blank/unchecked) if an Opportunity has no associated Account. This is expected behavior and acceptable given the org's data model.
3. **Permission Sets must exist pre-deploy:** All five Permission Sets (Admin_Team, Medical_Coordinator, Nursing_Staff, Operations_Manager, Operations_Team) must already exist in Production. Confirmed — these are existing PS records.
4. **Page layout metadata must be current in Sandbox:** All 37 layout files must reflect current Production state before the Gearset diff is cut, to avoid overwriting unrelated layout changes.
5. **Manual backfill is a business dependency:** Until Revenue Ops manually checks the checkbox on CV employee Accounts, all historical and current Opportunity records for those Accounts will show `false`. Reports will undercount CV employees. This is a known and accepted gap — no technical dependency can resolve it without an external source of truth.
6. **Production deployment confirmation:** Fields, FLS, and layouts are live in Sandbox. Production state is unconfirmed until Gearset deployment `2ceeeedb-b80b-435f-9d4f-ada3979c553b` is verified as successful.

### Rollback Steps (if needed)

Because all changes are **additive** (new fields, new FLS, new layout placements — nothing modified or deleted), rollback risk is low. If rollback is required:

1. **Remove field from page layouts:** Open each affected Account (6) and Opportunity (31) layout in Setup → Object Manager → Page Layouts. Remove `Cardone Ventures Employee` from each. This can also be done via Gearset by reverting the layout XML to the pre-change state.
2. **Remove FLS from Permission Sets:** In each of the 5 Permission Sets, remove Read/Edit access to both `Cardone_Ventures_Employee__c` fields.
3. **Delete Opportunity formula field:** Setup → Object Manager → Opportunity → Fields & Relationships → delete `Cardone_Ventures_Employee__c`. No data loss (formula field stores nothing).
4. **Delete Account checkbox field:** Setup → Object Manager → Account → Fields & Relationships → delete `Cardone_Ventures_Employee__c`. If any Account records have been checked by the time rollback occurs, this data will be lost — confirm with stakeholders before deleting.
5. **Notify stakeholders** that the feature has been rolled back and HEAL-33054 will be re-opened.

**Full rollback time estimate: ~30 minutes if done via Gearset revert; ~90 minutes if done manually in Setup.**
