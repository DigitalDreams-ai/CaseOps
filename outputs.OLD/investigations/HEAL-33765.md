### Confirmed Findings

1. **Field exists and is correct**: [Account.Main_Practitioner__c](sf://00N0b00000B34Fc) is a valid custom lookup field on Account object with no inherent visibility restrictions.

2. **Permission Set FLS mismatch**: Users are assigned TWO permission sets with conflicting FLS rules:
   - **[Office_Manager](sf://0PS5a000000GZQ4GAO)**: Explicitly grants read/write on [Account.Main_Practitioner__c](sf://00N0b00000B34Fc)
   - **[Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4)**: Explicitly denies access via omission (no FLS entry = field is hidden/read-only in this permission set context)

3. **Silent denial (Salesforce FLS logic)**: In Salesforce, when a user is assigned multiple permission sets:
   - Field is readable/editable only if ALL assigned permission sets allow it
   - If ANY assigned permission set omits FLS entry for a field, that field is inaccessible
   - [Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4) has no FLS entry for Main_Practitioner, therefore **Lab_Providers_C_R_E silently blocks access** despite [Office_Manager](sf://0PS5a000000GZQ4GAO) granting it

4. **Root cause confirmed**: The users can see Main_Practitioner field via [Office_Manager](sf://0PS5a000000GZQ4GAO) permission set FLS, but [Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4) permission set assignment prevents edit/read due to missing FLS entry.

### Rejected Hypotheses

- Hypothesis: "Main_Practitioner field is not in any FLS rule" — REJECTED. Field IS in FLS rules; [Office_Manager](sf://0PS5a000000GZQ4GAO) grants it, Lab_Providers omits it (confirmed by metadata inspection).
- Hypothesis: "Profile-level FLS is blocking" — REJECTED. New_Operations profile has no Account FLS entries; FLS is entirely permission-set-driven.

### Recommended Implementation Surface

**Problem location**: [Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4) permission set is missing FLS entry for [Account.Main_Practitioner__c](sf://00N0b00000B34Fc)

**Fix**: Add FLS entry to [Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4) permission set:
```xml
<fieldPermissions>
    <editable>true</editable>
    <field>Account.Main_Practitioner__c</field>
    <readable>true</readable>
</fieldPermissions>
```

**Implementation location**: 
- Metadata artifact: `force-app/main/default/permissionsets/Lab_Providers_C_R_E.permissionset-meta.xml`
- Change type: FLS field permission addition (configuration-only, safe to deploy)
- Deployment method: Standard metadata deployment or Gearset

**Rationale**: [Lab_Providers_C_R_E](sf://0PS5a000000GZQYGA4) must grant the same read/write access as [Office_Manager](sf://0PS5a000000GZQ4GAO) and [Medical_Staff](sf://0PS5a000000GZQ2GAO) permission sets (reference implementations) to unblock Pre-charting workflow.
