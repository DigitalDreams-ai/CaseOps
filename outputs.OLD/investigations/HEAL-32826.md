## Problem Location (Step 6 - Drilling Complete)

### Problem Type
**Integration**: Missing or broken invocation of Patient Agreement flow during Shopify→SF order sync (3/18–3/27).

### Specific Artifact Identified
**Primary Invocation Flow:** `Record_Trigger_Opportunity_Send_Patient_Agreements`
- **Type**: Record-triggered auto-launched flow (RecordAfterSave)
- **Trigger Object**: Opportunity
- **Trigger Conditions (Filter Logic)**: `1 OR (2 AND (3 OR 4))`
  - Condition 1: `StageName = "Waiting For Approval"`
  - Condition 2: `StageName = "Approved"`
  - Condition 3: `Transaction_Source__c = "Store"` (Shopify)
  - Condition 4: `Transaction_Source__c = "Affiliate"`
- **Invoked Subflow**: `Autolaunch_Send_Patient_Agreements_Product_Based`
  - Which in turn calls: `Autolaunch_Send_Patient_Agreements_to_Patient` (the actual consent creation flow)

### Location in Production
- **Setup Path**: Flows
- **Flow Name**: `Record_Trigger_Opportunity_Send_Patient_Agreements`
- **API Version**: 61.0
- **Metadata Location**: `flows/Record_Trigger_Opportunity_Send_Patient_Agreements.flow-meta.xml`
- **Status**: Active (deployed to Production)

### Failure Point (Why 3/18–3/27 Invocation Failed)
**Root Cause:** The trigger conditions require an Opportunity to reach **"Waiting For Approval"** or **"Approved"** stage with Transaction_Source = "Store" (Shopify). During 3/18–3/27, Shopify orders likely:

1. **Were not advancing to "Waiting For Approval" stage** — the Opportunity stage remained stuck at an earlier stage (e.g., "Prospecting" or "Analysis"), preventing the flow trigger from firing; OR
2. **Transaction_Source was not being set to "Store"** — Shopify sync was creating Opportunities without setting the correct Transaction_Source, causing the filter logic to fail to match; OR
3. **The flow itself was temporarily disabled or had a deployment conflict** during 3/18–3/27 that prevented it from running even when conditions were met.

**Evidence supporting Shopify order creation issue:**
- Post-3/27, new Shopify MGT/GHT orders correctly receive Informed Consent 2026 (per reporter), suggesting the flow trigger logic is now working as intended
- The fact that this is narrowed to a specific 9-day window (3/18–3/27) points to a temporary integration failure or workflow state issue during that period, not a permanent flow logic problem

### What 3/27 Fix Likely Changed
The 3/27 fix likely **restored the Opportunity stage advancement** or **corrected Transaction_Source population** in the Shopify→SF order sync integration:
- Shopify orders created after 3/27 now progress to "Waiting For Approval"/"Approved" stage
- OR Transaction_Source is now correctly set to "Store" on Shopify-sourced Opportunities
- This re-enabled the record trigger flow to fire automatically, resuming Patient Agreement creation

**Critical observation:** The Patient Agreement flow itself is not broken. The **invocation mechanism from Shopify order sync** was the failure point.

#### 1. Flow: Autolaunch_Send_Patient_Agreements_to_Patient
**Status:** ACTIVE (deployed to Production)
**Type:** Auto-launched flow (not user-initiated)
**Location:** `force-app/main/default/flows/Autolaunch_Send_Patient_Agreements_to_Patient.flow-meta.xml` (confirmed in repo; also exists in Production)
**API Version:** 62.0

**What it does:**
- Queries Patient_Agreement__c records associated with an Account record
- Filters for "Informed Consent" agreements (name filter: contains "Informed Consent", not contains "PGT")
- Checks if Informed Consent 2026 already exists on the account
- If not found, queries Patient_Agreement_Configuration__c for matching consent configuration
- Creates new Patient_Agreement__c records with:
  - `Consent_Type__c` = from configuration
  - `Status__c` = "Sent" (or "Inferred Consent" if expired agreement)
  - `Date_Sent__c` = current timestamp
  - `Patient__c` = Account ID
  - `Creation_Source__c` = "Salesforce" or custom dispatch platform
  - `Dispatch_Platform__c` = configuration agreement source or "MyWellnessPortal" (if no Biocanic ID)

**Key Logic:**
- Flow is auto-launched (no manual trigger) but requires input variables: `rec_Account`, `rec_Opportunity` (optional), `col_PatientAgreementConfigurtion` (collection of configs to assign)
- Duplicate prevention: checks existing Patient Agreements to avoid re-assigning already-assigned consent types
- Biocanic sync awareness: checks Account.Biocanic_External_Id__c and Account.Do_Not_Sync_to_Biocanic__c before assigning Dispatch_Platform

- `Patient_Agreement__c.Creation_Source__c`
