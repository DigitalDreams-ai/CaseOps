# Solution Plan — HEAL-33116

## 1. Problem Statement

The Consultation Opportunity object lacked a `Scheduled_By__c` picklist field to distinguish between self-scheduled (Calendly) appointments and manually scheduled ones. Schedulers and operations staff had no structured way to record who initiated a consultation booking. This affected all Consultation Opportunity records and was reported by the clinical/operations team (stakeholder: Sarah Phonthaphanh).

This is a **net-new feature request**, not a defect. No data was corrupted; the gap was a missing metadata element.

---

## 2. Escalation Decision

**Split decision: Support-Resolvable (field + layout + FLS) + Escalated to Engineering (Calendly automation).**

**Support portion:** Creating the `Scheduled_By__c` picklist field, placing it on layouts, and granting FLS are standard Salesforce admin tasks fully within Support/Admin scope. These have been completed.

**Engineering portion:** Automatically populating `Scheduled_By__c = "Self-Scheduled"` when Calendly syncs a new booking to Salesforce requires integration-layer logic (e.g., a Flow triggered by the Calendly sync process, or a backend automation rule). This is outside Admin scope and requires Dev/Engineering involvement. Stakeholder explicitly agreed to defer this automation to a separate clinical projects discussion with Nate and team.

### Engineering Handoff (deferred)

- Open a new Engineering ticket for: "Auto-populate `Scheduled_By__c = Self-Scheduled` on Consultation Opportunity when record is created via Calendly sync"
- Engineering should investigate the Calendly-to-Salesforce sync mechanism (webhook, middleware, or native integration) to determine the correct trigger point for setting this field value
- No changes to the field itself are required — Engineering populates it via automation using the existing `Self-Scheduled` picklist value

### Why This Is the Minimal Viable Fix

The stakeholder need is: "know who scheduled a consultation." The smallest solution that satisfies this is a single picklist field on the Opportunity, visible to the right users, on the right layouts. We do NOT need:
- A new object or junction object (the Opportunity already contextualizes the consultation)
- A flow or validation rule for the manual-entry use case (staff can select from the picklist without enforcement)
- A full Calendly integration redesign (the existing sync already creates the record; we only need to set one field value, which Engineering can add minimally)
- A rollout to additional layouts beyond the 6 consultation-specific ones (other Opportunity types are out of scope)

The Calendly automation is deferred — not skipped — because the field is usable immediately without it (manual population), and building automation without stakeholder sign-off on the sync architecture would risk over-engineering or rework.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Field not deployed to Production (pipeline failure) | Low | Medium | Smoke test on record `006Ql00000bJEWbIAO` before closing ticket |
| Permission set coverage incomplete — some users cannot see field | Medium | Low | Confirm with Sarah; extend FLS to missing permission sets if needed |
| Picklist value staleness — staff names change | High (over time) | Low | Document a maintenance process; assign a field owner |
| Calendly auto-population never built — field stays blank on synced records | High (if deferred indefinitely) | Medium | Ensure Engineering ticket is created and prioritized before closing HEAL-33116 |
| Layout not assigned to all Consultation record types | Low | Medium | Verify record type → layout assignments in Production Setup |

### Rollback Steps

If the field or layout changes cause unexpected issues in Production:

1. **Remove field from layouts** (immediate, no data loss): Setup → Object Manager → Opportunity → Page Layouts → edit each of the 6 layouts → drag `Scheduled_By__c` back to the field palette → Save. This hides the field from users without deleting it or its data.
2. **Revoke FLS** (if needed): Setup → Permission Sets → each of the 4 sets → Object Settings → Opportunity → set `Scheduled_By__c` to No Access. Users lose visibility immediately.
3. **Delete the field** (only if absolutely necessary): Setup → Object Manager → Opportunity → Fields & Relationships → `Scheduled_By__c` → Delete. **Warning:** This permanently destroys all stored values. Only do this if the feature is fully abandoned and no data has been entered. Salesforce soft-deletes fields for 15 days before permanent deletion — use this window to recover if deletion was accidental.
4. **Gearset rollback:** If a specific deployment PR introduced a regression, revert via Gearset using the previous metadata snapshot. Coordinate with Sean Bingham as the pipeline owner.

Rollback of the Calendly automation (Engineering scope) is handled separately within that ticket's own deployment plan.
