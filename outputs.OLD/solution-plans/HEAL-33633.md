# Solution Plan — HEAL-33633

## 1. Problem Statement

Order `801Ql00000zolrtIAA` for patient Joshua Brown cannot be eSubmitted to Wellvi. The Wellvi API returns HTTP 400 citing four missing required fields in the submission payload: `City`, `PostalCode`, `AddressLine1`, and `StateProvince`. The address data is visibly present in the Salesforce Order UI, confirming this is not a data-entry problem. The Salesforce-to-Wellvi integration callout is failing to read and serialize the correct address fields into the API request body. Wellvi confirmed they never received the order. Clinical ops (Sarah Phonthaphanh, Danielle Cress) and the patient are blocked. The ticket was prematurely closed on 2026-05-14 and remains unresolved as of 2026-05-19.

---

## 2. Escalation Decision

**Escalated to Engineering.**

Reason: The failure is in integration code (Apex callout or Flow) that builds the Wellvi API request body. Support cannot modify Apex classes or integration callout logic. Evidence is unambiguous — the Wellvi 400 response explicitly lists the address fields as null in the payload, while the same fields are populated and visible in the Salesforce Order UI. This is a field-mapping defect in the integration layer, not a configuration or data issue within Support's scope.

---

## 3. Proposed Fix

**Minimal Viable Fix: Correct the address field mapping in the Wellvi eSubmit callout.**

Engineering should take the following steps:

1. **Identify the callout component.** Locate the Apex class (or Flow with HTTP callout action) responsible for building the Wellvi eSubmit API request body. Search codebase for references to the Wellvi endpoint and for the field names `City`, `AddressLine1`, `PostalCode`, `StateProvince` (or their Wellvi API equivalents).

2. **Audit the address field reads.** Determine which Salesforce field API names the callout currently reads for address. The likely bug is one of:
   - The callout reads standard Order shipping fields (`ShipToCity`, `ShipToStreet`, `ShipToPostalCode`, `ShipToState`) but the Order record stores address in custom fields (e.g., `Shipping_City__c`, `Shipping_Address__c`).
   - The callout reads from a related record (e.g., Contact or Account) using a relationship that is null at submission time.
   - The SOQL query that fetches the Order before building the payload does not include the address fields in the SELECT clause, so they arrive as null even though they exist on the record.

3. **Apply the targeted correction.** Once the mismatched field API name(s) are confirmed:
   - Update the SOQL SELECT clause to include the correct address field API names.
   - Update the payload serialization to map those fields to the Wellvi API keys (`AddressLine1`, `City`, `PostalCode`, `StateProvince`).
   - No schema changes, no new fields, no new objects — only fix the field references in the existing callout logic.

4. **Manual resubmission.** After the fix is deployed to Production, manually trigger the eSubmit action on Order `801Ql00000zolrtIAA` to confirm the previously blocked order completes successfully.

5. **Audit recent submissions.** Query Wellvi submission logs (or Salesforce integration logs) for any other Orders submitted in the same window (2026-05-14 onward) that may have silently failed for the same reason and require resubmission.

**Why this is the minimal fix:** The address data already exists on the Order record — this is confirmed by the UI showing it correctly. No data migration, no record updates, no new fields, and no process redesign are needed. The only broken thing is the field reference inside the callout. A one-to-four line code change (correcting field API names in the SOQL query and payload mapping) is sufficient to resolve the issue.

## 5. Dependencies

- **Correct field API names must be confirmed before the fix is written.** Engineering must query Order `801Ql00000zolrtIAA` directly (via Developer Console or Workbench) to confirm which field API names actually hold the address data (e.g., `ShipToStreet` vs. a custom field). The fix cannot be written correctly without this.
- **Sandbox must have a representative Order record with address populated** to validate the fix end-to-end before Production deployment. If Sandbox data is sparse, Engineering should create a test Order with address fields populated matching the Production pattern.
- **Wellvi Sandbox/test endpoint must be available** (or a mock) to validate the corrected payload is accepted (HTTP 200) before deploying to Production.
- **No dependent downstream changes** are expected — this is a read-only field mapping correction in the outbound callout. No inbound data, no UI, no other integrations should be affected.

## 6. Sandbox Plan + Risk + Rollback

### Deployment to Sandbox

1. Retrieve the current Wellvi callout Apex class from Production into version control.
2. Apply the address field mapping correction in a feature branch.
3. Deploy the corrected class to a Full or Partial Copy Sandbox that mirrors Production Order data.
4. Create or identify a test Order in Sandbox with all address fields populated.
5. Trigger the eSubmit action against the Wellvi test/sandbox endpoint.
6. Confirm: (a) HTTP 200 response from Wellvi, (b) all four address fields present and non-null in the outbound payload (log or debug the request body).
7. Obtain peer code review and QA sign-off.
8. Deploy to Production via standard change/release process.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |

---|

### Rollback

- The only change is to an Apex class. Rollback = redeploy the prior version of the class from version control.
- No data changes, no metadata changes, no field or object modifications — rollback is a single class redeployment with zero schema impact.
- Estimated rollback time: under 15 minutes via standard deployment tooling (SFDX or change set).
- If rollback is needed, affected Orders will continue to fail eSubmit at the same rate as before the fix — no regression beyond current broken state.
