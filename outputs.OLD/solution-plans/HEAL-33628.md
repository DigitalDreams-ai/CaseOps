# Solution Plan — HEAL-33628

## 1. Problem Statement

The "Send Patient Agreements" Salesforce feature is failing to open consent forms for multiple patients. Affected users (patients) see a red error when attempting to access their consent sign-in. Three specific Account IDs have been identified as affected: 001Rh000010no9eIAA, 001Rh000006rScJIAU, and 0015a00003GjtXVAAZ. The clinical workflow impact is active and ongoing. A workaround reportedly exists but has not been confirmed or communicated.

## 2. Escalation Decision

**Decision: Insufficient Data — Investigation Incomplete. Cannot escalate or resolve until investigation is completed.**

The investigation record contains only template placeholders. No reproduction steps, no error message text, no metadata inspection, and no root cause have been established. Escalating to Engineering at this stage would be premature and would likely be returned for more information. Support must complete the investigation first.

The problem could stem from any of the following — none confirmed:
- Expired or invalid agreement/consent links
- A broken or recently changed Flow or Process Builder automation in the "Send Patient Agreements" feature
- A permission or sharing issue on the consent records
- An integration failure with the external consent portal

## 3. Proposed Fix

**Phase 0 — Complete Investigation (Blocker: must happen before any fix is applied)**

Step 1: Log into Production in read-only mode. Navigate to each of the three affected Account records:
- 001Rh000010no9eIAA
- 001Rh000006rScJIAU
- 0015a00003GjtXVAAZ

Step 2: On each Account, locate the Patient Agreements related list or relevant child records. Note the status of any agreement or consent records (e.g., sent date, expiry date, status field value, any error fields).

Step 3: Attempt to reproduce the red error by triggering "Send Patient Agreements" on one of the affected records (in Sandbox if a matching record can be replicated, or by reviewing Flow debug logs in Production).

Step 4: Capture the exact red error message text.

Step 5: Open Setup > Flows. Locate the Flow associated with "Send Patient Agreements". Check:
- Last modified date (was it recently changed?)
- Run the Flow debug/interview log for one of the affected records
- Check for fault paths that may be swallowing errors silently

Step 6: Check Setup > Process Automation > Paused and Failed Flow Interviews for any failed interviews tied to the affected Accounts.

Step 7: Confirm and communicate the existing workaround to the affected clinical team immediately — do not wait for the fix.

**Phase 1 — Apply Minimal Fix (after root cause is confirmed)**

The fix will be determined by what Phase 0 reveals. The most likely candidate fixes, ranked by probability and minimal scope, are:

**Candidate A — Failed Flow Interview / Fault Path (Most Likely)**
If Flow debug logs show a fault on a specific element (e.g., a Create Record, Send Email, or callout action):
- Fix only the broken element in Sandbox
- Do not restructure the Flow
- Re-activate the Flow version in Sandbox, test against cloned records, deploy to Production

**Candidate B — Expired or Malformed Agreement Links**
If the consent records have an expired or null URL field:
- Identify the field holding the link (e.g., Agreement_URL__c or similar)
- Determine whether the link is generated by the Flow or by an integration
- If Flow-generated: correct the formula or action that builds the URL
- If integration-generated: flag to Engineering for the integration layer only

**Candidate C — Permission / Sharing Issue**
If the affected records are owned by a specific user or profile and sharing rules have changed:
- Check OWD and sharing rules for the Agreement/Consent object
- If a sharing rule was removed or a profile permission changed recently, restore the minimum required access
- Do not broaden permissions beyond what is necessary

**Candidate D — Integration / External Portal Failure**
If the error originates outside Salesforce (e.g., the consent portal returns an error):
- This is an Engineering escalation — provide Engineering with: the exact error, the three Account IDs, the Flow name, and the callout endpoint
- Do not attempt to fix integration layer in Support

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan

1. Identify the Sandbox that mirrors Production configuration (Full or Partial copy preferred).
2. Replicate the failure: create or clone an Agreement record matching the affected pattern.
3. Apply only the minimal fix identified in Phase 0 (e.g., correct the single broken Flow element).
4. Run Flow in debug mode in Sandbox against the test record — confirm the red error no longer appears.
5. If fix involves metadata changes (Flow, field, sharing rule), retrieve current Production metadata via VS Code / Salesforce CLI before deploying.
6. Deploy to Production using Change Set or CLI — include only the single changed component.
7. Verify on one of the three affected Account records in Production (with the clinical team if possible).

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|
| Fix is applied to wrong root cause | Medium (investigation incomplete) | High — error persists or new errors introduced | Complete investigation fully before any change |
| Flow change breaks other Agreement workflows | Low–Medium | High — other patients affected | Test in Sandbox with multiple record scenarios before deploy |
| Sharing rule change over-broadens access | Low | Medium — HIPAA/data exposure concern | Apply minimum necessary access only; review with compliance if needed |
| Integration is the real cause; fix is in wrong layer | Medium | Medium — wasted effort, delay | Identify callout errors in Flow debug before assuming SF-side fix |

### Rollback Steps

- **Flow change**: In Setup > Flows, open the Flow. Click "View All Versions". Activate the previous version. This immediately reverts runtime behavior with no deployment needed.
- **Sharing rule change**: In Setup > Sharing Settings, re-add or remove the sharing rule that was modified.
- **Field value correction (data fix)**: If a data fix was applied via Data Loader or anonymous Apex, retain the export of original values before update and reload using the same tool.
- **Full rollback trigger**: If Production behavior worsens after deployment, activate the previous Flow version immediately (< 2 minutes), then investigate in Sandbox before re-attempting.

### Why This Is The Minimal Viable Fix

The investigation record is entirely incomplete — no root cause, no error message, no metadata inspected. The minimal viable fix at this stage is therefore **not a code or configuration change** — it is the completion of the investigation. Proposing a specific Salesforce metadata change now (e.g., redesigning the Flow, modifying permissions, or rebuilding the consent template) would be over-engineering against an unknown problem and risks introducing new failures.

Once root cause is confirmed, the fix will be scoped to the single broken component only. For example: if one Flow element has a bad field reference, we fix that element — we do not rebuild the Flow. If one sharing rule was removed, we restore it — we do not redesign the sharing model. This approach avoids regression, minimizes deployment risk, and keeps the change reviewable and reversible.
