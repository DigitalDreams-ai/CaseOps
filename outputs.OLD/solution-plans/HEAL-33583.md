# Solution Plan — HEAL-33583

## 1. Problem Statement

When a patient replies to an SMS sent by scheduling coordinator Jemimah, the inbound reply notification email is routed to Stephanie Silva instead of Jemimah. This is a misdirected notification originating from the Vonage Integration user (`vcartaya@10xhealthsystem.com`). The affected SMS History record is `aADQl000001UJFx`, sender ID `17866404062`, patient number `15087261435`. This constitutes a HIPAA risk: patient PHI (lab result reply content) is being seen by the wrong staff member. A prior related issue (HEAL-33537) involved Stephanie's own number being misconfigured; Idman's fix for that ticket likely left a residual misconfiguration that now surfaces for Jemimah's records. Other coordinators may be similarly affected.

## 2. Escalation Decision

**Escalated to Engineering.** This cannot be safely resolved by Support. The fix requires auditing and modifying SMS-Magic notification flow logic, Vonage number-to-user mapping configuration, and/or SMS record ownership rules — all of which are automation/integration components that carry systemic risk if changed without full understanding of the prior fix applied by Idman for HEAL-33537. Robert (original flow configurator) is a required resource. Incorrect changes could break Stephanie's now-working configuration or cascade to other coordinators.

## 3. Proposed Fix (Minimal Viable)

Engineering should investigate and apply the smallest targeted correction in the following order:

**Step 1 — Audit Vonage number-to-user mapping**
Locate the SMS-Magic or Vonage configuration table that maps phone numbers (Sender IDs) to Salesforce users. Check whether sender ID `17866404062` (Jemimah's number) is correctly mapped to Jemimah's Salesforce User record. If it is mapped to Stephanie Silva or to a shared/integration user, correct it to point to Jemimah.

**Step 2 — Review what Idman changed for HEAL-33537**
Before touching anything, confirm with Idman the exact metadata modified for Stephanie's fix. Understand whether the fix was a number mapping correction, a flow recipient field update, or an ownership reassignment. This prevents re-breaking Stephanie's record and clarifies the pattern.

**Step 3 — Audit SMS-Magic notification flow recipient logic**
If the number mapping is correct, inspect the SMS-Magic notification flow (likely a Salesforce Flow or Workflow Rule triggered on inbound SMS creation). Identify the field used to resolve the notification email recipient — likely `OwnerId` on the SMS History record, a lookup to the originating agent, or a custom field populated at send time. Confirm this field is correctly populated on record `aADQl000001UJFx` and on Jemimah's other outbound SMS records.

**Step 4 — Spot-check other coordinators**
Query SMS History records for at least 3–5 other active scheduling coordinators to confirm their sender IDs and owner/recipient fields resolve correctly. If systemic misconfiguration is found, scope a batch correction before deploying.

**Step 5 — Apply targeted correction**
Apply only the minimal change identified: either (a) update the number-to-user mapping record for `17866404062` to Jemimah, or (b) correct the recipient field logic in the notification flow for a single misresolved field, or (c) reassign SMS record ownership if that is the root driver. Do not redesign the flow.

## 5. Dependencies

- **Idman's prior change (HEAL-33537) must be understood first** — do not modify shared flow logic or mapping tables until the delta from that fix is documented. Risk of regression is real.
- **Robert must be consulted** — he designed the original SMS-Magic/Vonage flow. His input is required before any flow logic changes.
- **Jemimah's Salesforce User record must have a correctly populated phone/sender ID field** — the mapping fix only works if her user record carries the right identifier.
- **SMS-Magic package version** — confirm the installed version of SMS-Magic in Production before making flow changes, as configuration screens and field names vary by version.
- **If systemic misconfiguration is found** — a broader fix (batch update of mapping records) becomes a dependency before the targeted fix is considered complete.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment
1. Reproduce the condition in Sandbox by locating or creating a Vonage number-to-user mapping record that mirrors the Production misconfiguration.
2. Apply the targeted fix (mapping correction or flow recipient field update) in Sandbox only.
3. Simulate an inbound SMS reply and confirm the notification email routes to the correct agent user.
4. Confirm Stephanie's equivalent mapping in Sandbox still resolves correctly (regression check for HEAL-33537).
5. Once validated, deploy to Production via Gearset (for metadata components) or direct config change (for mapping record data changes that are not deployable metadata).

### Risk Assessment
- **Medium risk:** If the notification flow uses a shared recipient-resolution field, changing it could affect all coordinators simultaneously. Mitigated by auditing Step 4 before deployment.
- **Low-Medium risk:** Idman's HEAL-33537 fix could be partially undone if the root cause is in shared flow logic rather than an isolated mapping record. Mitigated by reviewing Idman's change log first.
- **HIPAA risk (high urgency, low technical risk):** The misdirection of PHI warrants prompt action, but the technical fix itself is low complexity if root cause is a mapping record. Do not delay for process reasons.
- **Scope creep risk:** If systemic misconfiguration is found across multiple coordinators, the fix scope expands. Engineering should timebox the audit (Step 4) and file a separate ticket for bulk remediation if needed rather than blocking this fix.

### Rollback
- **If fix is a mapping record change:** Revert the number-to-user mapping record to its prior value (document the original value before changing it). No deployment reversal needed — this is a data/config change.
- **If fix is a Flow change:** Deactivate the updated Flow version and reactivate the prior version in Salesforce Flow Builder. Gearset can restore the prior metadata version if deployed via changeset.
- **If fix is a Workflow Rule change:** Deactivate the modified rule and reactivate the prior state. Document rule criteria and actions before any modification.
- In all cases: notify Danielle Cress and Stephanie Silva immediately if rollback is triggered, and confirm whether misdirection resumes so they can manually forward any misdirected messages.

## Minimal Viable Fix Justification

The investigation points to a single misconfigured number-to-user mapping or a single incorrectly resolved recipient field on the SMS-Magic notification flow — not a fundamental design flaw. The fix for HEAL-33537 (Stephanie's prior issue) was reportedly a targeted correction, not a flow redesign, which confirms this class of problem is resolvable at the mapping/record level. We are not proposing a flow redesign, a new custom object, or a notification architecture change. The minimal fix is: **find the one record or field value that routes `17866404062` to the wrong user, and correct it.** Everything else in this plan is scoped to validating that single change does not regress adjacent configurations. This is the smallest intervention that stops PHI from reaching the wrong staff member.
