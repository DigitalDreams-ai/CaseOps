# Solution Plan — HEAL-33618
## CX Case Function Reporting (Escalation Timestamp + Elapsed Time)

---

## 1. Problem Statement

The customer (reported by Noah Scott) requires a way to report on Case escalation timing. Specifically:

- A **timestamp captured when a Case is escalated from Tier 1 to Tier 2**
- **Elapsed time: Case Opened → Escalated** (how long before escalation occurred)
- **Elapsed time: Escalated → Closed** (how long resolution took after escalation)

Currently, no confirmed mechanism exists in Salesforce to capture this escalation moment as a discrete, reportable data point. The gap means CX leadership cannot measure escalation response performance or identify bottlenecks in Tier 2 handling.

**Affected users:** CX reporting consumers (likely managers/analysts); multiple teams per customer's own impact statement.

**Blocker to solution:** The Tier 1 → Tier 2 escalation trigger mechanism in Salesforce is unknown (Status picklist? Checkbox? Queue reassignment? Entitlement Milestone?). Discovery must precede implementation.

---

## 2. Escalation Decision

**Decision: Support-Resolvable (tentative, confidence Low-Medium)**

**Rationale:**
- Adding a timestamp field, formula fields for elapsed time, a Record-Triggered Flow to stamp the timestamp, and a native Report is standard Salesforce Admin work — no code required under the most likely scenario.
- This class of request (schema + declarative automation + report) does not inherently require Engineering.

**Conditions that would escalate to Engineering:**
- The escalation event is NOT tracked by any existing Status value or field, AND cannot be represented by a declarative Flow trigger (e.g., it is driven by an external system event or a complex multi-object routing condition)
- Business-hours elapsed time is required AND the native `BUSINESSHOURS` formula approach is insufficient (e.g., multiple business-hours schedules, complex SLA logic)
- Existing Apex triggers or processes on the Case object conflict with a new Flow in a way that requires Apex modification

**Pre-condition before any build:** Operator must complete the discovery steps in Section 6 and answer Noah Scott's clarifying questions before any Sandbox work begins.

---

## 3. Proposed Fix (Minimal Viable)

### Why This Fix and Not Something Bigger

> "We could implement a full Entitlement Process with Milestones, SLA warnings, and push notifications — but the customer asked only for reportable timestamps and elapsed-time metrics. That requires exactly three things: (1) a field to hold the escalation moment, (2) formula fields to calculate durations, and (3) a report. An Entitlement Process redesign is a weeks-long project with significant change management risk. A single Flow + two formula fields + one report solves the stated need in a fraction of the time."

> "We are NOT creating a new object, junction object, or custom report type unless the field approach proves insufficient — because native Case fields are directly reportable in standard Case reports without additional configuration."

---

### Step-by-Step Fix (contingent on discovery confirming escalation is tracked via a Case Status value, e.g., `"Escalated to Tier 2"`)

#### Step A — Create Custom Timestamp Field

| Property | Value |
|---|---|
| Object | Case |
| Field Label | `Escalation Timestamp` |
| API Name | `Escalation_Timestamp__c` |
| Data Type | `Date/Time` |
| Required | No (populated by automation) |
| Default Value | None |
| Help Text | `Automatically populated when Case Status is set to Escalated to Tier 2.` |

**FLS:** Read-only for all profiles except System Administrator (field is automation-stamped; manual edits should be prevented).
**Layouts:** Add to Case Detail layout in a "Escalation & SLA" section (or nearest equivalent section). Match placement of any existing SLA/timestamp fields for consistency.
**Record Types:** All Case record types (or the specific record types used by Tier 1/Tier 2 if scoped — confirm during discovery).

---

#### Step B — Create Two Formula Fields for Elapsed Time

**Field 1: Time to Escalation (Hours)**

| Property | Value |
|---|---|
| Object | Case |
| Field Label | `Hours to Escalation` |
| API Name | `Hours_to_Escalation__c` |
| Data Type | `Formula (Number, 2 decimal places)` |
| Formula | `IF(NOT(ISBLANK(Escalation_Timestamp__c)), (Escalation_Timestamp__c - CreatedDate) * 24, NULL)` |
| Blank Field Handling | Treat blank fields as blank |

**Field 2: Hours Escalated to Closed**

| Property | Value |
|---|---|
| Object | Case |
| Field Label | `Hours Escalated to Closed` |
| API Name | `Hours_Escalated_to_Closed__c` |
| Data Type | `Formula (Number, 2 decimal places)` |
| Formula | `IF(AND(NOT(ISBLANK(Escalation_Timestamp__c)), IsClosed), (ClosedDate - Escalation_Timestamp__c) * 24, NULL)` |
| Blank Field Handling | Treat blank fields as blank |

**Note on Business Hours:** If the customer requires business-hours elapsed time, the formula becomes significantly more complex and may require a stored numeric field populated by a Flow using the `BUSINESSHOURS` formula or a scheduled Apex job. Confirm calendar vs. business hours with Noah Scott before building.

**FLS:** Read-only for all profiles.
**Layouts:** Add to same "Escalation & SLA" section as `Escalation_Timestamp__c`.

---

#### Step C — Create Record-Triggered Flow to Stamp Timestamp

| Property | Value |
|---|---|
| Flow Type | Record-Triggered Flow |
| Object | Case |
| Trigger | A record is updated |
| Entry Condition | `Status EQUALS "Escalated to Tier 2"` AND `Escalation_Timestamp__c IS NULL` (prevents re-stamping on subsequent saves) |
| Optimized For | Fast Field Updates |
| Action | Update Triggering Record: set `Escalation_Timestamp__c = {!$Flow.CurrentDateTime}` |
| Flow Label | `Case — Stamp Escalation Timestamp` |
| API Name | `Case_Stamp_Escalation_Timestamp` |

**Why "IS NULL" guard:** Ensures the timestamp is set once (at first escalation) and is not overwritten if the Case is re-saved in Escalated status later.

**Why Record-Triggered Flow and not Process Builder:** Process Builder is being retired by Salesforce. Flow is the current standard.

**Why not Apex trigger:** No code needed. A declarative Flow handles a single-field stamp on a status change with zero complexity.

---

#### Step D — Create Native Report

| Property | Value |
|---|---|
| Report Type | Cases |
| Columns | Case Number, Subject, Created Date, `Escalation_Timestamp__c`, `Hours_to_Escalation__c`, `Hours_Escalated_to_Closed__c`, Status, Owner |
| Filters | `Escalation_Timestamp__c IS NOT NULL` (scopes to escalated cases only) |
| Report Name | `Escalated Cases — Timing Summary` |
| Folder | CX / Case Operations (or existing shared reporting folder) |
| Chart (optional) | Bar chart: average `Hours_to_Escalation__c` by week |

## 4. Affected Components

| Component | Type | Action |
|---|---|---|
| `Escalation_Timestamp__c` | Custom Field (Date/Time) on Case | Create |
| `Hours_to_Escalation__c` | Custom Formula Field (Number) on Case | Create |

**Not affected:** Validation rules, Approval Processes, Entitlement Processes, existing Flows/Triggers (unless conflict discovered during Sandbox testing).

---

## 5. Dependencies

1. **Escalation tracking mechanism confirmed:** The Flow trigger condition must match the exact Status picklist value (or field) used to record escalation. If escalation is tracked by a Checkbox (`IsEscalated`) or a different field, the Flow entry condition changes. **Do not build until Noah Scott confirms the mechanism.**

2. **`ClosedDate` field availability:** The `Hours_Escalated_to_Closed__c` formula uses the standard `ClosedDate` field. Confirm this field is populated in the org (it is a standard Case field but its population depends on how Cases are closed — some orgs use custom closure processes).

3. **Calendar vs. business hours:** Formula approach above uses calendar hours. If business hours required, a different implementation path is needed (stored field + Flow with business-hours logic).

4. **No conflicting automation:** Confirm no existing Flow, Process Builder, or Apex trigger already fires on Case Status change and would conflict with the new Flow. Check in Sandbox after deployment.

5. **Profile/permission set coverage:** Confirm which profiles need to see the new fields on layouts. At minimum: Case Agents, Case Managers, Reporting users.

6. **Report folder permissions:** The target report folder must be shared with CX reporting consumers. Confirm folder access during deployment.

## 6. Sandbox Plan + Risk + Rollback

### Pre-Build Discovery (MUST complete before Sandbox)

- [ ] Operator reviews Case object fields in Production (Setup → Object Manager → Case → Fields) — search for any existing `Escalat*` fields
- [ ] Operator reviews Case Status picklist values in Production — identify the exact value representing Tier 2 escalation
- [ ] Operator reviews active Flows and Process Builders on Case object — document any that fire on Status change
- [ ] Operator sends clarifying questions to Noah Scott: (a) how is escalation recorded today, (b) calendar or business hours, (c) reporting surface preference, (d) any existing partial reports
- [ ] Await Noah Scott response before proceeding

### Sandbox Deployment Steps

1. In **Sandbox** (Full or Partial copy preferred; Developer Sandbox acceptable):
   - Create `Escalation_Timestamp__c` field
   - Create `Hours_to_Escalation__c` formula field
   - Create `Hours_Escalated_to_Closed__c` formula field
   - Set FLS for all 3 fields on relevant profiles
   - Add fields to Case layout in Escalation/SLA section
2. Create and activate `Case_Stamp_Escalation_Timestamp` Flow in Sandbox
3. **Test in Sandbox:**
   - Create a test Case → confirm `Escalation_Timestamp__c` is NULL
   - Update Status to the escalation value → confirm `Escalation_Timestamp__c` is auto-populated
   - Save the Case again without changing Status → confirm timestamp is NOT overwritten (NULL guard works)
   - Close the Case → confirm `Hours_Escalated_to_Closed__c` calculates correctly
   - Verify no existing automation errors surface in debug logs
4. Create report in Sandbox, validate columns and filters
5. Obtain sign-off from Noah Scott or CX stakeholder on Sandbox behavior before Production deployment
6. Deploy to **Production** via Change Set or manual recreation (fields + Flow + layout + report)

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Escalation Status value name in Flow does not match Production picklist value | Medium | High — Flow never fires, timestamp never stamps | Confirm exact picklist value spelling before activating Flow |
| Existing Flow or Apex trigger on Case Status conflicts with new Flow | Low-Medium | Medium — automation errors or double-execution | Review active automation in Sandbox; check debug logs during testing |
| `ClosedDate` not populated in org | Low | Medium — `Hours_Escalated_to_Closed__c` always NULL | Verify in Sandbox on a test closed Case; fallback: use `LastModifiedDate` with Status = Closed filter |
| Business hours required but calendar hours delivered | Medium | Medium — metric is inaccurate for SLA purposes | Clarify with Noah Scott before build; if business hours needed, pause and redesign formula |
| Fields added to wrong layout or missing from a layout variant | Low | Low — fields invisible to some users | Review all active Case layouts during deployment |

### Rollback Steps

If the fix causes issues in Production:

1. **Deactivate Flow** immediately: Setup → Flows → `Case_Stamp_Escalation_Timestamp` → Deactivate. This stops all future automation without data loss.
2. **Remove fields from layouts** if causing confusion: Setup → Object Manager → Case → Page Layouts → remove the 3 fields. Fields and their data are preserved.
3. **Delete formula fields** (`Hours_to_Escalation__c`, `Hours_Escalated_to_Closed__c`) if needed — formula fields hold no stored data, deletion is safe.
4. **Delete or blank `Escalation_Timestamp__c`** — if the field was stamped incorrectly on records, a Data Loader export/import can null out the field on affected records before deletion. Delete the field only after data is cleared.
5. **Delete report** — no data impact, safe to delete at any time.

Full rollback restores Case object to pre-change state with zero data loss to existing Case records.

---

## Minimal Viable Fix Justification Summary

The customer needs three metrics: when escalation happened, how long it took to escalate, and how long it took to close after escalation. The minimal solution is:
- **1 Date/Time field** to record the moment
- **2 formula fields** to calculate durations (zero storage cost, zero maintenance)
- **1 Flow** (4 nodes) to stamp the timestamp declaratively
- **1 Report** to surface the data

This is 3 fields + 1 Flow + 1 Report. It requires no new objects, no Entitlement Process changes, no approval processes, no custom report types, no Apex, and no integration work. Every component is individually reversible. The fix is fully within Salesforce Admin capability and can be built, tested, and deployed in a single sprint once discovery is complete.
