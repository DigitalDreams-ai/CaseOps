# Solution Plan — HEAL-33399

## 1. Problem Statement

All CX Case records are failing on close in Production Salesforce. Every attempt to close a Case results in a flow error, completely blocking the CX team from closing any cases. The failure rate is reported as 100% across all CX Case records. Two confirmed affected Case IDs: `500Ql00000vqcl7IAA` and `500Ql00000w0TS3IAM`. Impact is classified as multiple teams/departments blocked entirely. The error is visible in an attached screenshot (not yet analyzed). The break is suspected to have been introduced by a recent deployment, though this has not been confirmed.

---

## 2. Escalation Decision

**Escalated to Engineering.**

Reason: The root cause is an unidentified broken element inside a Salesforce Flow on the Case object (likely a Record-Triggered Flow or Screen Flow firing on status change to Closed). The investigation record was left as an unfilled template — no Salesforce metadata has been inspected, no specific Flow has been identified, and no root cause has been confirmed. Support cannot safely diagnose or patch a Flow defect affecting all CX cases without:
- Identifying the specific failing Flow by name
- Inspecting its elements for null references, missing required inputs, broken subflow calls, or deactivated actions
- Validating a fix in Sandbox before deploying to Production

All three of these steps require Engineering access and sandbox deployment capability.

## 3. Proposed Fix (Minimal Viable)

Engineering should execute the following targeted investigation and fix sequence. Do NOT redesign or refactor any Flow. The goal is to identify and patch the single broken element causing the failure.

**Step 1 — Analyze the screenshot immediately**
Open the screenshot attached to HEAL-33399. Extract the exact error message text. This will identify the failing Flow by name and the failing element by label or API name. This single step may reduce all remaining investigation time from hours to minutes.

**Step 2 — Identify all active Flows on the Case object that fire on close**
In Salesforce Setup → Flows, filter by:
- Object: Case
- Trigger: Record-Triggered (Before Save or After Save) with entry condition on Status = Closed, OR Status changed
- Also check: Screen Flows invoked from Case page layouts or Quick Actions labeled "Close Case" or similar
- Also check: Process Builder processes on Case (legacy) that may invoke a Flow

Document every active automation found.

**Step 3 — Inspect each candidate Flow for the following failure patterns (in order of likelihood):**
1. **Null reference on a Get Records or Assignment element** — a variable used downstream is not null-checked and the record has no matching data
2. **Missing required input on a subflow or invocable action** — a called action requires a field that is now null, renamed, or removed
3. **Deactivated or deleted referenced component** — an Apex action, invocable method, or subflow that was deactivated or deleted in a recent deployment
4. **Hard-coded record type ID or user ID** — an ID that no longer exists in Production after a sandbox refresh or migration
5. **Required field left blank by the Flow** — a field marked required on a related object that the Flow tries to create/update without populating

**Step 4 — Apply the smallest possible fix**
Once the broken element is identified:
- If null reference: add a null-check Decision element before the failing element and route gracefully
- If missing required input: populate the missing input from available record data
- If deactivated component: reactivate it or replace the reference with the correct active version
- If hard-coded ID: replace with a dynamic lookup or Custom Label/Custom Metadata record
- If required field: populate the field or make it optional if business rules permit

Do NOT restructure the Flow. Do NOT add new Flow elements beyond what is required to fix the single broken path. Do NOT modify any other automation.

**Step 5 — If a recent deployment is identified as the cause**
If Git history or Salesforce deployment logs show a change to this Flow in the last 1–2 deployments, assess whether a rollback to the prior Flow version is faster and safer than a targeted element fix. A rollback is preferred if the prior version was stable and the change was non-critical.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan
1. Retrieve current Production Flow version using Salesforce CLI: `sf project retrieve start --metadata "Flow:"` (insert Flow API name once identified)
2. Open the Flow in Sandbox Flow Builder
3. Apply the minimal element-level fix identified in Step 4 above
4. Save as a new version — do NOT activate yet
5. **Test in Sandbox:** Open two Case records, set Status to Closed, confirm the Flow completes without error
6. Test edge cases: Cases with missing optional fields, Cases with no related records that the Flow may reference
7. Activate the fixed Flow version in Sandbox
8. Re-run close test — confirm zero errors
9. Deploy fixed Flow to Production using: `sf project deploy start --metadata "Flow:"` or via change set
10. **Post-deploy validation in Production:** Close one of the two known-affected Cases (`500Ql00000vqcl7IAA` or `500Ql00000w0TS3IAM`) and confirm success

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Fix addresses wrong Flow element — error persists in Production | Medium | High | Screenshot analysis in Step 1 reduces this risk significantly; validate fully in Sandbox before deploying |
| Fix introduces a new error in a different Flow path | Low | Medium | Test multiple Case close scenarios in Sandbox, not just the happy path |
| Sandbox Flow version differs from Production | Medium | High | Retrieve directly from Production before making changes |
| Another in-flight deployment overwrites the fix | Low | High | Coordinate with release management; deploy fix as soon as validated |
| Rollback required but prior Flow version not retained | Low | High | Before deploying fix, export and save the current broken Flow version as a backup file |

### Rollback Steps

If the deployed fix causes new errors or does not resolve the original issue:

1. **Immediate mitigation (< 5 minutes):** In Salesforce Setup → Flows → find the deployed Flow → click "View All Versions" → activate the previous version. This reverts to the prior state instantly without a deployment.
2. **If the prior version was also broken** (i.e., the Flow was broken before the most recent deployment): deactivate the Flow entirely as a temporary measure to unblock case-close, accepting that any automation the Flow provides will not run. Communicate this trade-off to the CX team immediately.
3. **If deactivation is not safe** (Flow performs a required business operation): restore from the backed-up Flow version exported before the fix deployment using Salesforce CLI metadata deploy.
4. Document the rollback in Jira HEAL-33399 and re-open Engineering investigation.

## Minimal Viable Fix Justification

This plan proposes **patching a single broken element inside an existing Flow** rather than rebuilding, replacing, or redesigning any automation. Reasons:

- **We do not know what is broken yet.** The investigation record was never completed. Proposing a broader fix — such as redesigning the case-close process, creating a new Flow, or refactoring related automation — would be solving a problem that has not been diagnosed. That is over-engineering under uncertainty.
- **The failure mode is 100% consistent across all cases.** A 100% failure rate on a single action (close) points to a deterministic, structural break — a single missing element, null reference, or deactivated component — not a data quality or edge-case issue. A targeted element fix is the correct scope.
- **The prior Flow was presumably working.** If this is a regression, the Flow functioned before a recent deployment. The minimal fix is to restore that working state, not redesign it.
- **Screenshot analysis may make this trivial.** Salesforce Flow error messages identify the Flow name and element label directly. The entire investigation may resolve to: open screenshot, read error, fix one element. Proposing anything larger before reading the error message would be premature.
- **Alternatives rejected:** Full Flow rebuild (unnecessary — scope unknown, risk high, time cost high), creating a new automation to replace the broken one (duplicates logic, creates maintenance burden), disabling the Flow permanently (unacceptable — business logic would be lost). None of these are warranted until the broken element is identified and assessed.
