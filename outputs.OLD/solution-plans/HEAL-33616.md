## 3. Proposed Fix (Minimal Viable — Conditional)

**This section presents the minimal fix for the two most probable root causes. The correct path must be confirmed by completing the investigation.**

### Path A — Data Fix (most likely if error is record-specific)

If inspection of the two Lab Order records reveals missing or invalid field values (e.g., height, weight, DOB, or a required lookup):

1. In Production (read-only), document the exact field(s) that are null or invalid on both records.
2. In Sandbox, reproduce the same field state and confirm the error is triggered.
3. Populate the missing field(s) on both affected Production records directly via the Salesforce UI (if the field is editable by an admin) or via a one-time Data Loader update.
4. Attempt to re-submit the order on each record to confirm the error is resolved.
5. No metadata deployment required.

**Why this is minimal:** We are fixing the two affected records only. We are not modifying any Flow, Validation Rule, or Apex. If the data was always required and the user simply did not provide it, no configuration change is warranted.

### Path B — Validation Rule or Flow Condition Fix (likely if error is systemic)

If both records share valid data but the error persists, and the second affected record appeared after a recent deployment or configuration change:

1. Identify the specific Validation Rule or Flow step that is throwing the error (confirm from screenshot error message).
2. In Sandbox, reproduce the error using matching record data.
3. Apply the smallest possible change to the offending component:
   - **Validation Rule:** Narrow the condition using `ISCHANGED()`, `ISNEW()`, or a null-safe guard so the rule does not fire incorrectly on the Update path.
   - **Flow:** Add a null-safe check (e.g., `{!record.Field__c} != null`) before the failing element, or correct the fault path to surface a user-friendly error rather than a hard fault.
4. Deploy the single changed component (Validation Rule or Flow version) to Production via Change Set or VS Code / Salesforce CLI.
5. Retest both affected records.

**Why this is minimal:** We are changing exactly one metadata component — the specific rule or flow step that is failing — and nothing else. We are not redesigning the Flow, not creating new fields, and not modifying layouts or permissions.

### Path C — Escalate to Engineering

If the error message in the screenshots references an Apex exception (e.g., a stack trace, `System.NullPointerException`, or a named Apex class), this is outside Support scope and must be handed off to Engineering with:
- Both affected record IDs
- The exact error message text
- The name of the Apex class or trigger identified
- The timestamp of the first failure

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

1. **Reproduce:** In Sandbox, create or clone a Lab Order record that matches the field state of the two affected Production records (use the field values documented during Production read-only inspection).
2. **Confirm error:** Attempt the Update Order action and confirm the same error is thrown. This validates that Sandbox is a reliable test environment for this fix.
3. **Apply fix:** Implement the chosen fix (Path A: populate data; Path B: edit Validation Rule or Flow in Sandbox).
4. **Retest:** Re-attempt the Update Order action and confirm the error is gone and the order submits successfully.
5. **Regression test:** Submit a new Lab Order (not previously in error) to confirm the fix does not break the happy path.
6. **Deploy to Production:** For Path A (data fix), apply directly in Production — no deployment needed. For Path B (metadata fix), deploy via Change Set or Salesforce CLI, restricted to the single changed component.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Fix resolves error for these two records but root cause is broader (more records affected) | Medium | Medium | After fix, run a SOQL query on Lab Order records in a similar state to identify any other stuck orders |
| Path B metadata change inadvertently affects other Lab Order flows or rules | Low | High | Restrict Change Set to only the single changed component; regression test the full Lab Order submission flow in Sandbox before deploying |
| Investigation remains incomplete and fix is applied to wrong component | High (currently) | High | Do not proceed to implementation until investigation is completed per Section 2 |
| Second affected record indicates systemic issue; fix is applied but more records break later | Medium | Medium | After fix, monitor Lab Order error logs for 48 hours; set up a list view or report for Lab Orders in error/pending state |

### Rollback Steps

- **Path A (data fix):** If populating the field(s) causes an unintended side effect, revert the field value(s) to their previous state using Data Loader with the original values documented before the change. This is a low-risk, reversible operation.
- **Path B (Validation Rule):** Deactivate the modified Validation Rule immediately in Production (Setup > Object Manager > Lab Order > Validation Rules). This does not require a deployment — it takes effect instantly.
- **Path B (Flow):** In Production, open the Flow in Flow Builder and activate the previous version. Salesforce retains all prior Flow versions; rollback is a one-click operation with no deployment required.
- **Path C (Apex — Engineering):** Rollback is Engineering\'s responsibility; they should follow standard code deployment and rollback procedures (sandboxed hotfix branch, prior Apex class version restore).

## Minimal Viable Fix Justification

This plan deliberately avoids the following over-engineered alternatives:

- **Do not redesign the Lab Order Flow end-to-end.** The error affects two records on one specific path (Update Order). A full Flow redesign is unnecessary and introduces new regression risk.
- **Do not create new fields or objects.** There is no evidence that missing data model components are the cause. The issue is most likely either bad data on two records or a strict condition in an existing component.
- **Do not modify FLS, layouts, or record types.** These are not implicated by any evidence in the investigation. Adding layout or permission changes would broaden the blast radius with no benefit.
- **Do not add a new Validation Rule or approval process.** The fix is to correct or guard an existing condition, not to add new enforcement logic.
- **Do not batch-update all Lab Order records.** Only two records are confirmed affected. A targeted fix to those two records (or to the one metadata component causing the failure) is sufficient until evidence of broader impact is established.

The smallest possible action that closes this ticket is: identify the exact field or condition causing the error, fix it on the two affected records or in the single failing component, and confirm by re-submitting both orders.
