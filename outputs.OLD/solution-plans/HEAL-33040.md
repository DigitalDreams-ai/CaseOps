# Solution Plan — HEAL-33040

## 1. Problem Statement

A patient account (`001Ql00000x8VCkIAM`) created via the **new patient link** entry point was never synced to Shopify. No Shopify Customer ID was written to the account, and no error appeared in the `Shopify Sync Message` field — indicating a **silent failure**. The manual fallback trigger (`Trig Shopify Sync = True`) was attempted by Danielle Cress on 2026-04-13 and also produced no result. Two independent sync paths are broken. Scope may extend to all accounts created via the new patient link during the affected window — total impact is unknown.

---

## 2. Escalation Decision

**Escalate to Engineering.** This is not support-resolvable.

Rationale: Both the automatic creation-time sync and the manual fallback field trigger have failed. This rules out a configuration or data issue on a single record and points to broken Apex trigger or Flow logic that only Engineering can inspect and repair. Support has no safe path to resolve broken code. Danielle Cress already flagged this to the SF team on 2026-04-13 with no confirmed resolution.

---

## 3. Proposed Fix

### Engineering Investigation Scope (Minimal)

Engineering should focus exclusively on the two failing sync paths. Do not refactor or redesign the sync architecture — find the specific breakage and patch it.

**Step 1 — Identify the sync mechanism**
- In the Salesforce org, locate all Apex triggers on the `Account` object. Filter for any trigger referencing Shopify, customer creation, or the `Trig_Shopify_Sync__c` field.
- Locate all active Flows on the `Account` object (Record-Triggered and Scheduled). Filter for any Flow referencing Shopify sync logic or the `Trig_Shopify_Sync__c` field.
- One or both will contain the shared sync logic that is silently failing.

**Step 2 — Reproduce the failure on the affected record**
- Pull Account `001Ql00000x8VCkIAM` in a Sandbox refreshed from Production.
- Confirm `Shopify_Customer_ID__c` is empty and `Shopify_Sync_Message__c` is empty.
- Set `Trig_Shopify_Sync__c = True` and save. Observe: does the trigger or Flow fire? Does it error silently?

**Step 3 — Check entry point divergence**
- Determine whether accounts created via the new patient link entry point differ structurally from accounts created via other paths (e.g., different Record Type, missing required field value, or a field condition in the trigger/Flow that excludes this entry point).
- If a Flow condition or Apex `if` block is gating sync on a field that is not populated by the new patient link flow, that gate is the root cause — relax or correct the condition.

**Step 4 — Check silent failure in sync callout**
- If the trigger/Flow does fire but the Shopify API call fails silently, inspect exception handling in the Apex class responsible for the callout. A caught exception that logs nothing would explain the empty `Shopify_Sync_Message__c`.
- Add or confirm error logging to `Shopify_Sync_Message__c` so failures surface.

**Step 5 — Apply minimal patch**
- Fix the identified condition, exception handler, or missing field value. Do not redesign the sync flow.
- Confirm fix works for both (a) creation-time sync and (b) manual `Trig_Shopify_Sync__c = True` trigger, since both use shared logic.

**Step 6 — Post-deploy remediation**
- After deploying the fix to Production, manually set `Trig_Shopify_Sync__c = True` on Account `001Ql00000x8VCkIAM` to re-trigger sync.
- Run an audit query for all accounts created via the new patient link entry point that have an empty `Shopify_Customer_ID__c` — re-trigger sync for each.

## 5. Dependencies

- **Sandbox refresh:** Fix must be developed and tested in a Sandbox refreshed from Production so the affected account data and org configuration are representative.
- **Named Credential / Connected App:** The Shopify API integration must be active and authenticated in Sandbox for end-to-end testing. Confirm the Named Credential for Shopify is present and valid in Sandbox before testing.
- **Re-trigger after fix:** The fix alone will not retroactively sync already-created accounts. A manual re-trigger step is required post-deploy for all affected accounts.
- **Jira ticket status:** Ticket was auto-closed; it must be reopened and linked to the Engineering fix ticket to ensure traceability and prevent the reporter from assuming the issue was resolved.
- **Do not repeat manual `Trig_Shopify_Sync__c = True` in Production** until the fix is deployed — repeated triggers against broken logic may produce unintended side effects once the fix goes live.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan

1. Refresh Sandbox from Production (or use an existing full-copy Sandbox that contains Account `001Ql00000x8VCkIAM` data).
2. Reproduce the failure in Sandbox: confirm `Shopify_Customer_ID__c` empty, `Trig_Shopify_Sync__c = True` produces no result.
3. Inspect and patch the identified Apex trigger or Flow component.
4. Test path A: create a new Account via the new patient link entry point in Sandbox — confirm `Shopify_Customer_ID__c` is populated post-creation.
5. Test path B: on an existing Account with empty `Shopify_Customer_ID__c`, set `Trig_Shopify_Sync__c = True` — confirm sync fires and `Shopify_Customer_ID__c` is populated.
6. Confirm `Shopify_Sync_Message__c` captures error state when API is unavailable (negative test).
7. Deploy fix to Production via Change Set or preferred deployment method.

### Risk Assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| Fix exposes previously silent errors for other accounts — bulk sync failures surface at once | Medium | Run audit query before re-triggering; batch re-triggers in small groups |
| Entry point condition fix causes double-sync for accounts created via other paths | Low | Verify trigger/Flow conditions against all Account creation entry points in Sandbox before deploy |
| Repeated `Trig_Shopify_Sync__c = True` in Production (already attempted) causes duplicate Shopify customer records post-fix | Low-Medium | Check Shopify for existing customer record before re-triggering; confirm upsert logic in callout class |
| Broader population of unsynced accounts larger than anticipated | Medium | Run audit query immediately; scope remediation effort before re-triggering |

### Rollback Steps

- **If fix is an Apex Trigger change:** Revert to previous Apex Trigger version via source control or re-deploy the prior version from the Sandbox backup. The trigger change is isolated to sync logic and does not affect Account record structure.
- **If fix is a Flow change:** Deactivate the new Flow version and reactivate the prior version in Setup > Flows. No data is altered by rollback.
- **In both cases:** The underlying bug (no sync) was already the pre-fix state — rolling back returns the org to the same broken-but-stable condition. No Account records are modified by the fix itself; only the sync process is affected.
- Flag any Shopify customer records created during the fix window for review if rollback is triggered.

---

## Why This Is The Minimal Viable Fix

The investigation confirms the sync logic itself is broken — not the Account data, not FLS, not a missing field. A full redesign of the Shopify sync architecture is not justified. The minimal fix is: **find the specific condition or exception handler that is causing silent failure and patch it.** This restores both sync paths (creation-time and manual) without touching unrelated components. The only additional action required is a post-deploy re-trigger for affected accounts — which is a data remediation step, not an architectural change. Everything else (audit query, ticket reopening) is operational hygiene, not engineering scope creep.
