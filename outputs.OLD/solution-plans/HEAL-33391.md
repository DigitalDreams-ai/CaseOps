# Solution Plan — HEAL-33391

## 1. Problem Statement

Four members of the Order Processing team — Sarah Phonthaphanh, Kaci Krupnik, Sarah Courchia, and Kristhal Valdez — cannot view all Orders in Salesforce. They lack the object-level Read permission on the Order object and/or visibility into the "All Orders" list view. This blocks their ability to perform order processing work. No system defect is involved; this is a pure permissions/visibility gap.

---

## 2. Escalation Decision

**Support-Resolvable. Confidence: High.**

This is a standard access provisioning request. The scope is clear, the affected users are named, and managerial approval is on record (Sarah Phonthaphanh self-identified as manager and named her team). No code, automation, or schema changes are required. A permission set assignment — and at most a list view sharing rule — fully resolves the issue.

---

## 3. Proposed Fix (Minimal Viable)

### Step 1 — Audit Production (read-only)
- Navigate to Setup → Permission Sets in Production.
- Search for any existing permission set matching "Order Processor", "Order Manager", or "Order Lead" naming conventions.
- Open the candidate permission set → Object Settings → Orders → confirm it carries **Read** (not Edit/Delete) on the Order object.
- If a suitable permission set already exists: proceed to Step 3. If not: proceed to Step 2.

### Step 2 — Create Permission Set in Sandbox (only if no suitable one exists)
- In Sandbox, create a new Permission Set named: `Order_Processor_Read_Access`
- Label: `Order Processor Read Access`
- Object Settings → Orders → enable: **Read** only (do NOT enable Create, Edit, Delete, View All, Modify All unless confirmed as required).
- Field-Level Security: set all standard Order fields to **Read** to match the access level of existing Order-related permission sets.
- Save. Do not assign to profiles.

### Step 3 — Verify "All Orders" List View
- In Sandbox (or Production read-only), navigate to the Orders tab → check whether an "All Orders" list view exists.
- If it exists: confirm its sharing setting is "All Users" or includes the relevant profiles/roles. If it is privately scoped, update sharing to "All Users" or the relevant public group.
- If it does not exist: create a list view named "All Orders" on the Order object with no filter criteria (returns all records) and set visibility to "All Users".

### Step 4 — Assign Permission Set to 4 Users in Production
- Once Sandbox validation is complete (Step 5), in Production:
  - Setup → Permission Sets → select `Order_Processor_Read_Access` (or the identified existing set)
  - Click "Manage Assignments" → Add Assignments
  - Add: Sarah Phonthaphanh, Kaci Krupnik, Sarah Courchia, Kristhal Valdez
  - Save.

### Step 5 — Confirm Access
- Ask Sarah Phonthaphanh to log in and confirm all 4 users can see the Orders tab and the "All Orders" list view.
- Verify that records are visible but Edit/Delete buttons are absent (confirming Read-only access).

### Why this is the minimal viable fix
We are not modifying profiles (which would affect all users on that profile), not creating sharing rules on Order records (not needed — OLS Read already surfaces all records to the user), and not rebuilding list views beyond the single "All Orders" view. A permission set assignment is the smallest targeted change that grants exactly the requested access to exactly the 4 named users without side effects on anyone else.

---

## 4. Affected Components

| Component | Type | Change |
|---|---|---|
| `Order_Processor_Read_Access` | Permission Set | Create (if no existing set qualifies) OR assign existing |
| Order Object Settings (within Permission Set) | OLS | Read enabled |
| Order Field-Level Security (within Permission Set) | FLS | Read on all standard fields |
| "All Orders" List View (Orders object) | List View | Create if absent; update sharing if privately scoped |
| Permission Set Assignments × 4 users | Assignment Record | New assignments in Production |

**Not affected:** Profiles, Flows, Validation Rules, Record Types, Page Layouts, Sharing Rules, Roles.

---

## 5. Dependencies

- **Order object tab must be visible** on the affected users' apps/profiles. If the Orders tab is hidden at the profile level, the permission set alone will not surface it — tab visibility must also be enabled in the permission set (Tab Settings → Orders → Default On).
- **"All Orders" list view** must exist and be shared to all users (or a public group that includes these 4 users). OLS Read is necessary but not sufficient for list view visibility if the list view is privately owned.
- **Manager approval is on record** — Sarah Phonthaphanh has named all 4 users. No additional approval gate is required, but the Jira ticket should be updated with this as the authorization record for audit trail purposes.
- **Sandbox must mirror Production permission set configuration** for validation to be meaningful. Confirm Sandbox was refreshed recently enough that Order object config matches Production.

---

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan
1. Log into Sandbox as admin.
2. If creating a new permission set: create `Order_Processor_Read_Access` as described in Step 2 above.
3. Assign the permission set to a Sandbox test user mapped to the Order Processor role.
4. Log in as the test user → navigate to Orders tab → confirm "All Orders" list view is visible and returns records.
5. Confirm records are Read-only (no Edit/Delete inline or via record detail).
6. If list view was created or modified: confirm it appears and shows expected columns.
7. Document Sandbox validation result before touching Production.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Permission set grants Edit access unintentionally | Low | Medium | Explicitly set OLS to Read only; do not enable Edit/Delete/View All/Modify All |
| "All Orders" list view exposes sensitive order data (e.g., pricing, PII) | Medium | Medium | Review list view columns before sharing broadly; restrict columns if needed |
| Existing permission set already assigned to these users causes conflict | Low | Low | Additive assignment — no conflict; verify in Production user detail |
| Sandbox config does not match Production (stale refresh) | Medium | Low | Cross-check Order object OLS in both orgs before relying on Sandbox validation |
| Three users (non-manager) have not individually confirmed request | Low | Low | Manager approval is sufficient per standard provisioning; noted for audit |

### Rollback Steps
If access must be revoked after deployment:
1. Setup → Permission Sets → `Order_Processor_Read_Access` → Manage Assignments.
2. Remove assignments for all 4 users.
3. Users immediately lose Order Read access — no further changes needed.
4. If the permission set was newly created and causes unintended effects: deactivate it (it cannot be deleted while assigned; remove all assignments first, then delete).
5. If list view sharing was broadened: edit the "All Orders" list view → change visibility back to original setting (e.g., owner-only or specific group).
6. Rollback is fully reversible with no data loss or schema impact.
