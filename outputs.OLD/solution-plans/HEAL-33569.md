# Solution Plan — HEAL-33569

## 1. Problem Statement

User (Marianne Sallo) can see the **Generate Supplement Order** button on the Opportunity object but receives an error when executing it. Access has been approved by manager Danielle. The button is non-functional for this user; all other users with access are presumed unaffected. Root cause is unconfirmed due to unread screenshot attachments, but the symptom profile (visible button, runtime error) points to a permissions gap rather than a broken component.

---

## 2. Escalation Decision

**Decision: Conditionally Support-Resolvable — pending screenshot review.**

The error has not been read from the attached screenshots (`image (99).png`, `Screenshot 2026-05-11 160118.png`). Until the exact error message is extracted, this cannot be fully resolved. However, the symptom pattern — button visible but throws error on click — is consistent with one of two support-resolvable causes:

1. **Missing Permission Set or Profile access** to the underlying Flow, Quick Action, or Apex class driving the button.
2. **FLS or CRUD restriction** on a field or object the process attempts to read or write.

If the screenshot reveals a hard Apex exception unrelated to permissions (e.g., a null pointer, integration timeout, or governor limit), this escalates to Engineering. That determination is a prerequisite gating step.

**Confidence: Low until screenshots are read. Proceed with investigation gate before any change.**

## 3. Proposed Fix (Minimal Viable)

### Gate Step — Must Complete First

1. Open `image (99).png` and `Screenshot 2026-05-11 160118.png` from the Jira attachment. Extract the exact error message text.
2. If the error is **"Insufficient Privileges," "You do not have access," or an FLS/CRUD message** → proceed with Fix A below.
3. If the error is an **Apex exception, flow fault, or integration error** → stop and escalate to Engineering with the error text.

### Fix A — Permission Set Assignment (most likely resolution)

**Assumption:** Another user who can successfully execute the button has been identified as a reference user. Inspect their permissions to confirm the target Permission Set.

**Steps:**

1. In Production (read-only inspection):
   - Navigate to **Setup > Object Manager > Opportunity > Buttons, Links, and Actions**.
   - Locate **Generate Supplement Order** and record the action type (Flow, Quick Action, Apex, LWC).
   - If it is a **Flow**: note the Flow API name. Go to **Setup > Flows**, open the Flow, confirm it is Active.
   - If it is an **Apex class**: note the class name.
   - If it is a **Quick Action** invoking a Flow or Apex: trace to the underlying component.

2. Identify the Permission Set that grants access:
   - Find a working reference user → **Setup > Users > [reference user] > Permission Set Assignments**.
   - Identify the Permission Set that includes the Apex class execution right or Flow access as applicable.
   - Alternatively, search existing Permission Sets for one containing the Apex class or named to match Supplement Order functionality.

3. Verify Marianne Sallo is missing that Permission Set:
   - **Setup > Users > Marianne Sallo > Permission Set Assignments** — confirm the identified Permission Set is absent.

4. In **Sandbox** first (see Section 6):
   - Assign the identified Permission Set to a test user mirroring Marianne's profile.
   - Execute the Generate Supplement Order button.
   - Confirm success with no error.

5. In **Production** (after Sandbox validation):
   - **Setup > Users > Marianne Sallo > Permission Set Assignments > Edit Assignments**.
   - Add the identified Permission Set.
   - Save.
   - Confirm with Marianne that the button executes without error.

### Fix B — FLS/CRUD (if Fix A is not sufficient)

Only pursue if the error message explicitly references a specific field or object permission. In that case:

- Identify the field(s) cited in the error.
- Add Read/Edit FLS for those fields to the existing Permission Set assigned to Marianne (do NOT modify Profile-level FLS unless all users on that profile require access).
- Apply to Sandbox first, validate, then Production.

### Why This Is The Minimal Viable Fix

The button is already visible to Marianne, which means the Page Layout and Record Type configuration is already correct — no layout changes are needed. The component itself is Active and working for other users — no Flow or Apex fix is needed. The only gap is Marianne's individual access. Assigning one existing Permission Set (or adding FLS to an existing Permission Set) is the smallest change that closes that gap without touching any shared configuration. We are explicitly **not** modifying the Profile (which would affect all users on that profile), not creating a new Permission Set (one already exists for this use case), and not redesigning the button or underlying process.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan

1. Log into **Sandbox**.
2. Identify or create a test user with the same Profile as Marianne Sallo.
3. Ensure the test user does **not** have the target Permission Set assigned (mirror the broken state).
4. Navigate to an Opportunity record and confirm the Generate Supplement Order button produces the same error (reproduce the issue in Sandbox).
5. Assign the identified Permission Set to the test user in Sandbox.
6. Re-execute the Generate Supplement Order button.
7. Confirm: (a) no error, (b) expected outcome of the action occurs (supplement order created or process triggered as designed).
8. If Fix B (FLS) is also required, add the FLS change to the Permission Set in Sandbox and re-test.
9. Document test results (pass/fail, screenshots) before moving to Production.

### Risk Assessment

| Risk | Likelihood | Severity | Mitigation |
|---|---|---|---|
| Permission Set grants broader access than intended | Low | Medium | Review all permissions in the target Permission Set before assigning — confirm scope is appropriate for Marianne's role |
| Button invokes external integration; Sandbox test triggers live system | Low-Medium | High | Inspect Flow/Apex for external callouts before Sandbox test; use a test Opportunity record with no real downstream impact |
| Error is Apex exception unrelated to permissions; fix has no effect | Medium | Low | Gated by screenshot review — if error is non-permissions, escalate before proceeding |
| Danielle approval not confirmed; change control violation | Low | Medium | Block Production change until approval is verified in writing |
| FLS change affects other users sharing the Permission Set | Low | Medium | Only applies if Fix B is needed; review all assignees of the Permission Set before modifying FLS |

### Rollback Steps

**If the Permission Set assignment causes unintended behavior after Production deployment:**

1. **Setup > Users > Marianne Sallo > Permission Set Assignments > Edit Assignments**.
2. Remove the assigned Permission Set.
3. Save.
4. Confirm with Marianne that the button returns to its previous state (visible but non-executing — same as before the change).
5. No data loss risk: the Permission Set assignment is additive and non-destructive. Removing it returns the user to the exact prior state.

**If Fix B (FLS) was applied and must be rolled back:**

1. Navigate to the Permission Set in Setup.
2. Locate the modified field(s) under Object Settings.
3. Revert FLS to the prior setting (remove Edit, retain or remove Read as appropriate).
4. Save. No data is affected by FLS rollback.

**Rollback confidence: High.** Both changes (Permission Set assignment and FLS) are fully reversible with no data or configuration side effects.
