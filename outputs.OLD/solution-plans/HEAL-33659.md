# Solution Plan — HEAL-33659

## 1. Problem Statement

A net-new picklist field (`Supplement_Inquiry__c`) was requested on the Case object to support Customer Experience call-handling workflows. The field did not exist prior to this request. Affected users are Customer Support agents who need to capture supplement inquiry data during case intake. The ticket was raised by Noah Scott and escalated due to incomplete deployment confirmation.

---

## 2. Escalation Decision

**Support-Resolvable.** No code, automation, or cross-system dependencies are involved. This is a standard Salesforce admin task: picklist field creation, page layout placement, and FLS update on permission sets. Sean Bingham completed the configuration steps on 2026-05-19, but the target environment (Sandbox vs. Production) was not explicitly confirmed, leaving the ticket in an ambiguous state. The fix is administrative and fully reversible.

## 3. Proposed Fix (Minimal Viable)

### Why This Fix and Not Something Larger

The request is purely additive — a new picklist field with defined values, placed in an existing layout section, with FLS granted to existing permission sets. No object redesign, no flow changes, no validation rules, and no new record types are needed. The "Call Details" section already exists on the Customer Experience page layout, so no layout restructuring is required. We reuse existing permission set targets rather than creating new profiles or permission sets. This is the smallest possible change that delivers the feature.

## 5. Dependencies

- **Permission Sets must pre-exist in Production:** The Customer Support permission sets being updated for FLS must already exist in Production before deployment. Confirm their API names match Sandbox exactly before deploying permission set metadata.
- **Page Layout must pre-exist in Production:** The "Call Details" section on the Customer Experience layout must exist in Production. If it does not, the layout deployment will fail or place the field incorrectly.
- **Picklist value set scoping:** If `Supplement_Inquiry__c` uses a global value set, that value set must be deployed before the field. If it uses a local (field-specific) value set, the field deployment carries the values inline — no separate dependency.
- **No automation dependency:** No flows, triggers, or validation rules reference this field, so no sequencing concerns beyond the above.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Plan
1. Validate all metadata components exist and are correct in Sandbox using Gearset's compare tool.
2. Run a Gearset deployment to Production using "Validate Only" mode first — confirm zero errors before committing.
3. Deploy in this order: Custom Field → Picklist Values → Page Layout → Permission Set FLS.
4. Post-deploy: have a Customer Support agent in Production confirm the field appears on the Case record layout and is editable.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Changes were already made directly in Production | Medium | Low — field may already be live | Verify with SOQL query before deploying; skip if already present |
| Permission set API name mismatch between envs | Low | Medium — FLS not applied | Confirm permission set names match before deploying |
| Picklist values deploy incomplete (35 values) | Low | Low — agents see partial list | Count values post-deploy; re-deploy value set if needed |
| Field placed in wrong layout section | Low | Low — cosmetic, not functional | Manually reorder field on layout post-deploy |

### Rollback Steps

- **Field:** Picklist fields cannot be deleted immediately if they contain data. If no data has been entered (likely, since field is new), delete `Supplement_Inquiry__c` via Setup → Object Manager → Case → Fields. If data exists, deactivate all picklist values and remove field from layout instead.
- **Page Layout:** Remove `Supplement_Inquiry__c` from the "Call Details" section via Layout Editor. No data impact.
- **FLS:** Revoke Read + Edit on `Supplement_Inquiry__c` for Customer Support permission sets via Permission Set editor or re-deploy prior FLS state via Gearset.
- **Rollback window:** All rollback actions are manual and reversible within minutes, with no downstream automation to unwind.
