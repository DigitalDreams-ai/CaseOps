# Solution Plan — HEAL-33562

## 1. Problem Statement

The **Recall** button is missing from the Approval History component on Opportunity records for Tyler Kaplan (RN, Clinical/Nurse profile). Tyler and potentially all Nurse/Clinical users can no longer recall their own submitted approval requests on Opportunities. The OP team is currently performing recalls manually on behalf of affected users as a workaround. The button was previously visible and functional; the regression appears to have occurred months ago. Affected record confirmed: `006Ql00000eMscDIAS`.

---

## 2. Escalation Decision

**Escalated to Engineering / Salesforce Admin.**

Reason: The Recall button visibility is controlled by native Salesforce Approval Process configuration (specifically the "Allow submitters to recall approval requests" setting and who qualifies as the submitter) and/or Permission Set assignments. Support does not have the metadata access required to audit or safely modify Approval Process definitions, Permission Sets, or the automation (Flow/Process Builder) that may be submitting approvals on behalf of users. Additionally, a stakeholder check is required before re-enabling access — the restriction may have been intentional.

---

## 3. Proposed Fix (Minimal Viable)

Engineering should execute the following targeted audit and apply the smallest change that restores recall access:

**Step 1 — Identify the governing Approval Process**
- In Setup > Approval Processes > Opportunity, identify which active process covers the records Tyler submits.
- Check the **"Allow submitters to recall approval requests"** checkbox. If unchecked, this is the fix: enable it and redeploy the process.

**Step 2 — Determine submitter context**
- Check whether the approval is submitted by Tyler directly (UI/button) or by an automated process (Flow, Process Builder, or Apex).
- If automation submits the approval, the "submitter" in Salesforce's context is the running user of the automation (often a system/admin user), not Tyler. This strips her recall rights regardless of the checkbox above.
- If automation is the submitter: update the Flow/Process Builder to use the **"Submit for Approval" action with the record owner or initiating user as the submitter**, not the system context. Alternatively, add Tyler's profile/role to the list of "Initial Submitters" on the Approval Process so she qualifies.

**Step 3 — Check Permission Set assignments**
- Search for any Permission Set that explicitly grants or previously granted approval recall rights (custom or managed package permissions related to approvals).
- Compare Tyler's current Permission Set assignments against a peer user who retains recall access (if one exists). Re-assign any missing Permission Set after stakeholder sign-off.

**Minimal viable outcome:** The single smallest fix is most likely ONE of:
- (A) Re-checking "Allow submitters to recall approval requests" on the Approval Process (if it was unchecked), OR
- (B) Reassigning a removed Permission Set to Tyler and affected Nurse users, OR
- (C) Correcting the submitter context in the automation so Tyler is recognized as submitter.

Apply only the fix that matches the confirmed root cause. Do not change all three unless all three are confirmed broken.

**Why this is minimal:** We are not redesigning the Approval Process, not changing Opportunity page layouts, not creating new automation, and not altering profiles. A single configuration toggle or permission assignment is sufficient to restore the button if the root cause is confirmed. The OP workaround continues until the fix is validated in Sandbox and stakeholder approval is received.

No page layout changes, no new fields, no new objects, no validation rules affected.

---

## 5. Dependencies

- **Stakeholder sign-off required (Courtney + Clinical/OP leadership):** Before any fix is applied, confirm with stakeholders that the recall restriction was NOT intentional. If it was intentional, no fix should proceed.
- **Sandbox parity:** The Approval Process configuration and Permission Sets must exist in Sandbox and must mirror Production for testing to be valid.
- **Automation identification:** If a Flow or Process Builder is submitting the approval, Engineering must identify it before testing — the fix differs depending on whether automation is involved.
- **Broader user scope:** If all Nurse/Clinical users are affected (not just Tyler), the Permission Set reassignment must be applied to all affected users, not just Tyler's record. This should be confirmed before deployment.
- **Active Approval Process:** Any change to an active Approval Process in Salesforce requires the process to be deactivated, edited, and reactivated. Pending approval requests during this window should be identified and handled (they are not automatically cancelled, but the process should be reactivated promptly).

---

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Deployment Steps
1. Confirm Sandbox has a matching Opportunity Approval Process and matching test user with the Nurse/Clinical profile.
2. Reproduce the issue in Sandbox: submit an Opportunity for approval as the test Nurse user and confirm the Recall button is absent.
3. Apply the fix identified in Step 3 above (approval process setting change and/or permission set reassignment).
4. If the Approval Process setting is changed: deactivate the process, make the change, reactivate. Retest recall button visibility.
5. If a Permission Set is reassigned: assign it to the test user in Sandbox. Retest.
6. If automation submitter context is updated: modify the Flow/Process Builder in Sandbox, submit a test approval, confirm Tyler's equivalent test user sees the Recall button.
7. Document which single change resolved the issue before promoting to Production.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|
| Change was intentional — re-enabling breaks a compliance/workflow rule | Medium | High | Stakeholder sign-off before any change |
| Deactivating Approval Process during business hours causes in-flight approvals to be orphaned or delayed | Low | Medium | Schedule change during off-hours; identify any pending approvals first |
| Broader user scope than Tyler — fix applied only to one user while others remain broken | Medium | Low | Query all users with Nurse/Clinical profile and apply fix uniformly |
| Automation submitter fix has unintended side effects on other approval steps | Low | Medium | Test full approval cycle (submit → approve → recall → resubmit) in Sandbox |

### Rollback Steps
- **Approval Process setting:** Deactivate the process, uncheck "Allow submitters to recall," reactivate. Immediate effect.
- **Permission Set reassignment:** Remove the Permission Set from affected users. Immediate effect.
- **Automation change:** Revert the Flow/Process Builder to the prior version using Salesforce Flow version history. Activate the previous version.

All rollback actions are reversible within minutes. No data migration or record changes are involved — this is configuration-only.
