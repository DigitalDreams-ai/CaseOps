# Solution Plan — HEAL-33619

## 1. Problem Statement

Case disposition picklist values (Level 1, Level 2, Level 3) in the Call Details section of the Case object are missing additions and renames that were scoped in prior ticket HEAL-33249. That ticket was closed before its Sandbox changes were deployed to Production. As a result:

- New Level 1 value "Test - At Home Blood" and its dependent Level 2 and Level 3 values are absent from Production.
- Five Level 1 renames and two Level 2 renames are absent from Production.

Affected users: Call center agents and any team relying on Case disposition data entry. Reporter confirms the team is "completely blocked."

---

## 2. Escalation Decision

**Escalated to Engineering.**

Reason: Adding and renaming picklist values on dependent picklist fields (L1 → L2 → L3 hierarchy) requires direct Salesforce metadata changes to CustomField / ValueSet XML. These changes cannot be made by Support in Production directly. A managed deployment from Sandbox to Production via Gearset or a changeset is required. Engineering controls that deployment pipeline.

---

## 3. Proposed Fix (Minimal Viable)

**Why this fix and not a larger one:** The sole broken outcome is that specific picklist values are missing or incorrectly named in Production. No flow logic, validation rules, page layouts (beyond confirming field visibility), or object schema need to change. The fix is purely additive/rename operations on existing picklist field value sets. We do not redesign the dependency structure — we extend it minimally.

### Step-by-Step

**Step 1 — Locate HEAL-33249 Sandbox work**
- Check the Sandbox used for HEAL-33249 for any existing picklist configuration.
- Treat as reference only; re-validate all values before deploying (config may be stale or partial).

**Step 2 — Configure in Sandbox: New Level 1 value**
- On the Case object (or applicable custom object), open the Disposition Level 1 picklist field.
- Add new active value: `Test - At Home Blood`
- Confirm it is added to all applicable Record Types (match existing L1 values' record type availability).

**Step 3 — Configure in Sandbox: New Level 2 values (dependent on "Test - At Home Blood")**
- Confirm with reporter the exact L2 values to add under "Test - At Home Blood" (reference HEAL-33249 scope doc).
- Add each L2 value to the Disposition Level 2 dependent picklist, mapping controlling value = "Test - At Home Blood".
- Note: If "National availability (Dependent)" is itself a controlling value for Level 3, confirm its dependency mapping carefully.

**Step 4 — Configure in Sandbox: New Level 3 values (dependent on applicable Level 2s)**
- Add each L3 value to the Disposition Level 3 dependent picklist, mapping to their respective L2 controlling values.

**Step 5 — Configure in Sandbox: Renames**
- Apply the five Level 1 renames to existing picklist values (use "Replace" / rename in place — do NOT delete and re-add, to preserve existing record data).
- Apply the two Level 2 renames using the same rename-in-place approach.

**Step 6 — Validate dependent picklist mappings**
- In Sandbox, open a Case record and walk through all three levels: confirm new L1 shows correct L2 options, L2 shows correct L3 options.
- Confirm renamed values display correctly on existing Case records.

**Step 7 — Deploy Sandbox → Production via Gearset**
- Include only the affected CustomField metadata components (Disposition Level 1, Level 2, Level 3 fields and their ValueSet/DependentPicklist definitions).
- Do not include unrelated components in the same deployment.

**Step 8 — Post-deploy validation in Production**
- Open a Case record in Production, navigate to Call Details section, confirm all new values appear and renames are reflected.
- Confirm existing records with old value names display correctly after rename.

## 6. Sandbox Plan + Risk + Rollback

### Sandbox Plan

1. Work in the designated Sandbox (same one used for HEAL-33249 if available, otherwise any full/partial Sandbox with Case configuration).
2. Make all picklist additions and renames in Sandbox first.
3. Validate with a QA Case record walk-through across all three disposition levels.
4. Use Gearset to compare Sandbox vs Production — include only the three picklist CustomField components in the deployment package.
5. Deploy to Production during a low-traffic window; notify team lead before deploying.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Renaming existing L1/L2 values corrupts data on existing Case records | Low — Salesforce rename-in-place updates the label without changing the stored API value if done correctly | High | Use Salesforce native rename (not delete + re-add); verify stored API value vs label distinction before deploying |
| New L2/L3 values not mapped to correct controlling value | Medium — dependent picklist config is error-prone | High | Validate in Sandbox with live record test before deploying |
| New values missing from one or more Record Types | Medium | Medium | Check record type picklist inclusions explicitly in Sandbox metadata before deploy |
| Reports or dashboards filtering on old picklist labels break after rename | Low-Medium | Medium | Notify report owners; update report filters post-deploy if needed (out of scope for this fix but flag to stakeholders) |
| HEAL-33249 Sandbox config is partially deployed and conflicts | Low | Medium | Treat HEAL-33249 Sandbox state as read-only reference; re-configure cleanly |

### Rollback Steps

1. **If new values cause issues:** Deactivate the newly added picklist values in Production (makes them unavailable for selection on new records without removing them from existing records). This is reversible with no data loss.
2. **If renames cause issues:** Rename values back to their original labels in Production using the same rename-in-place method. Existing record data is unaffected.
3. **If dependent picklist mappings are wrong:** Correct the controlling value mappings in Sandbox and re-deploy the corrected metadata.
4. **Full rollback via Gearset:** Use Gearset's deployment history to redeploy the pre-change Production metadata snapshot if a clean revert is needed.

> **Minimal viable justification:** This fix touches only picklist value sets on three existing fields. No new objects, fields, flows, or layouts are created. Renames are done in-place to avoid data migration. The deployment package is scoped to the three CustomField components only, minimizing blast radius. Everything else — layouts, flows, record types (beyond value availability) — remains unchanged.
