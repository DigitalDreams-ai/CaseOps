## 5. Rollback Plan

Execute this plan if: the deployed fix causes new errors, the original error persists after deployment, or post-deployment validation fails at any step.

**Rollback objective:** Restore Production to a state where the Flow is active. Note: since the pre-fix state was also broken (Flow was erroring on close), the rollback options are tiered by severity.

---

### Rollback Option A — Activate Previous Flow Version (Fastest, < 5 minutes, No CLI Required)

This is the first action to take if validation fails.

**Manual steps:**
1. Log in to Production Salesforce as a System Administrator
2. Navigate to Setup → Flows
3. Search for `[FLOW_API_NAME]`
4. Click on the Flow
5. Click "View All Versions" (or "View Version History" depending on UI)
6. Locate the version that was active before this deployment (the version number recorded in Step 2c)
7. Click "Activate" on that version
8. Confirm the activation prompt

**Result:** The pre-fix version (which had the original breakage) is restored. This does not fix the original problem but returns Production to the pre-intervention state, preventing any new regressions introduced by a bad fix.

**Post-rollback query to confirm:**

```
sf data query \
  --query "SELECT Id, ApiName, Label, Status, VersionNumber FROM FlowDefinition WHERE ApiName = '[FLOW_API_NAME]'" \
  --target-org prod \
  --json
```

**Confirm:** `VersionNumber` matches the pre-deployment baseline recorded in Step 2c.

---

### Rollback Option B — Redeploy Backup Metadata (If Option A Is Unavailable)

If the previous version cannot be activated via UI (e.g., Salesforce does not retain it), redeploy the backup retrieved in Step 2b:

```
sf project deploy start \
  --metadata "Flow:[FLOW_API_NAME]" \
  --source-dir ./backup/HEAL-33399/pre-fix \
  --target-org prod \
  --json
```

**What it does:** Deploys the exact Flow XML snapshot taken from Production before the fix was applied.

**Expected output:** `"status": 0`, `"success": true`, `numberComponentsDeployed: 1`

**Confirm restored:**

```

  --query "SELECT Id, ApiName, Label, Status, VersionNumber, LastModifiedDate FROM FlowDefinition WHERE ApiName = '[FLOW_API_NAME]'" \
  --target-org prod \
  --json
```

### Rollback Option C — Deactivate Flow Entirely (Emergency Only, Last Resort)

⚠️ **Use only if:** The deployed fix introduces critical new errors AND the pre-fix Flow version also cannot be safely activated AND the CX team and business confirm that temporarily disabling all Flow automation on Case close is preferable to an active error.

This is a **business decision**, not a technical one. Confirm with Sean and CX team lead before executing.

**Manual steps:**
1. Log in to Production Salesforce as a System Administrator
2. Navigate to Setup → Flows → `[FLOW_API_NAME]`
3. Click "Deactivate" on the current active version
4. Confirm deactivation

**Result:** Cases can be closed without a flow error, but any business logic the Flow performs (notifications, field updates, related record creation, etc.) will NOT execute. This is a known business impact trade-off.

**Communicate immediately to CX team:** "Case-close automation is temporarily disabled. Cases can be closed but [describe what the Flow was doing] will not execute until the Flow is repaired and reactivated."

**Confirm deactivation:**

```

  --query "SELECT Id, ApiName, Label, Status, VersionNumber FROM FlowDefinition WHERE ApiName = '[FLOW_API_NAME]'" \
  --target-org prod \
  --json
```

**Confirm:** `Status` = `Inactive` (or no active version returned).
