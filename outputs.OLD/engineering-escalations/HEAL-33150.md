# Engineering Handoff

## Issue

HEAL-33150 - Cx Case Record Response / internal email replies missing from Case activity.

## Root Cause

The affected replies are bypassing Salesforce. In the confirmed Production sample, Salesforce logged the outbound internal approval email from the Case, but the Outlook reply path went back to the agent and/or shared Outlook recipients rather than to a Salesforce Email-to-Case routing address. Because Salesforce never receives that inbound reply, no incoming `EmailMessage` is created on the Case.

This is not supported as a global Customer Experience Email-to-Case/threading defect. Production has Customer Experience cases where inbound replies attach correctly when the reply is delivered to `support@10xhealthsystem.com` or `info@10xhealthsystem.com`.

## Smoking Gun

The exact send path for the affected internal approval email is the Customer Experience Case layout's standard Case email action:

- Layout: `Case-Customer Experience`
- Record action exposed: `Case.SendEmail`
- QuickActionDefinition: `Case.SendEmail`
- Label: `Email`
- Type: `SendEmail`
- Target object: `EmailMessage`
- Target parent field: `Parent`
- Fields exposed on action: `ValidatedFromAddress`, `ToAddress`, `CcAddress`, `BccAddress`, `Subject`, `HtmlBody`
- Only field override: `ToIds = Case.ContactId`
- No field override/default for `ValidatedFromAddress`
- No Reply-To field configured on the action
- No default Salesforce-routed sender enforced by the action

The affected email record proves how that action was used:

- `EmailMessage.Id`: `02sQl00000t0TfnIAE`
- `EmailTemplateId`: `null`
- `EmailRoutingAddressId`: `null`
- `FromAddress`: `ntorres@10xhealthsystem.com`
- `ValidatedFromAddress`: `ntorres@10xhealthsystem.com`
- `ToAddress`: `tjones@10xhealthsystem.com`
- `BccAddress`: `ntorres@10xhealthsystem.com`
- `Headers`: `null`

Absolute conclusion: this was not a Salesforce Email Template problem and not an Email-to-Case routing-address matching problem. The internal approval email was sent from Natasha Torres's personal validated sender, with no Email Template and no Email Routing Address. Replies naturally go back to Natasha / Outlook recipients, not Salesforce.

## Evidence

- Jira screenshots show the Salesforce Case feed has the outbound internal approval email but not the internal approval reply.
- Outlook screenshots show the internal approval reply kept the thread token, but visible delivery is to human/shared Outlook recipients, not a Salesforce Email-to-Case routing address.
- Production query of nearby Cx refund cases found successful inbound replies when addressed to routed mailboxes such as `support@10xhealthsystem.com` or `info@10xhealthsystem.com`.
- Production `EmailServicesAddress` data shows previous notes conflated per-user Email-to-Salesforce addresses with Email-to-Case routing addresses. Most of the 38 active service addresses are `emailtosalesforce` / `*.le.salesforce.com`, not Case routing addresses.
- Pattern check: other internal refund approvals to `tjones@10xhealthsystem.com` with `ValidatedFromAddress = support@10xhealthsystem.com` later produced incoming replies when the reply thread included `info@10xhealthsystem.com` / `refunds@10xhealthsystem.com`. Natasha's affected threads used `ValidatedFromAddress = ntorres@10xhealthsystem.com` and had no incoming approval reply on the Case.

## Proposed Fix

Audit and correct the reply path for Cx/internal approval Case emails:

1. Update or replace the Customer Experience internal approval email path so it does not send from the agent's personal `ValidatedFromAddress`.
2. Change the workflow so required replies are delivered to Salesforce:
   - Use a Salesforce-routed mailbox as Reply-To/From for the internal approval email, or
   - Configure the shared approval/refund mailbox to forward thread-tokened replies into an Email-to-Case routing address.
3. Do not rely on Bcc to the agent or per-user Email-to-Salesforce for reply capture.
4. Validate the fix in Sandbox with an Outlook reply from an internal approver.

## Validation Required

Sandbox:

1. Send an internal approval email from a Cx Case using the current workflow.
2. Reply from Outlook and confirm the current failure mode.
3. Apply the routing/reply-path fix.
4. Reply again and verify an `Incoming = true` `EmailMessage` is created on the original Case.
5. Confirm ordinary customer replies to existing routed addresses still attach correctly.

Production:

- No Production change has been made by this investigation.
- Production update should happen only after Sandbox validation.
- Deployment path depends on implementation: Setup config, mailbox forwarding, email action/template metadata, or admin-managed email routing.

## Notes

Creating a new Cx Email-to-Case routing address by itself will not fix replies that are addressed only to an agent's mailbox. The key requirement is that the reply must be delivered to Salesforce.
