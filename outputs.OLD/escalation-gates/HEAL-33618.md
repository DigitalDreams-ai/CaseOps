# Escalation Gate Decision — HEAL-33618

## Decision
Support-Resolvable

## Confidence
Medium — 65% — Root cause is clearly missing metadata (no escalation timestamp field exists), not a code defect or platform bug. The proposed solution (custom DateTime field, record-triggered Flow, report formula) is within standard Salesforce Admin capability. Confidence is held at Medium rather than High because three blocking unknowns remain unresolved: how escalation is currently signaled in the org, which reporting surface is required, and whether managed packages or external integrations constrain Case status transitions. Any one of these unknowns could reveal complexity requiring Apex or Engineering involvement.

## Key Evidence
- Root cause is a metadata gap, not a broken configuration or platform defect — the Case object simply lacks a field to capture escalation timestamp, which is a standard admin deliverable
- Proposed implementation path (custom DateTime field, record-triggered Flow to stamp on status change, elapsed-time formula in reports) uses native declarative Salesforce tooling with no Apex or Engineering dependency under normal org conditions
- Internal notes confirm no implementation has been attempted yet, meaning there is no failed admin effort or blocked deployment that would suggest the problem exceeds admin scope
- The solution pattern (stamp a DateTime on a record state transition via Flow) is a well-established, low-risk Salesforce admin pattern with no inherent Engineering threshold
- Internal notes explicitly flag the managed package and external CTI integration risk as a conditional concern, not a confirmed blocker — it is a verify-first risk, not a known constraint

## Red Flags
- Escalation trigger mechanism is unconfirmed: if escalation is not a discrete Status picklist value but is instead driven by a managed package, CTI integration, or queue assignment logic, Flow stamping may conflict with that system and Engineering or vendor involvement could become necessary
- Historical data requirement is unresolved: if Noah Scott requires backfilled escalation timestamps on closed cases, this cannot be satisfied by a forward-looking Flow alone and would require a data migration script or Apex batch job, both of which exceed admin scope
- Elapsed-time reporting using two DateTime fields has a known limitation in standard Salesforce Reports (no native cross-field DateTime math); if the required reporting surface is standard Reports rather than CRM Analytics, a workaround or Engineering input on report architecture may be needed
- No Production inspection has been completed — it is possible an escalation-related field or Flow already exists in the org, which could change the implementation plan or reveal a configuration conflict
