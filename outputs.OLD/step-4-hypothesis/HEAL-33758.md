# Step 4 — Problem Hypothesis and Solution

**Issue:** HEAL-33758  
**Date:** 2026-05-21

---

## Problem Hypothesis

**Confirmed facts (from Step 3):**
- Four Lab Order records (LAB-000234000, LAB-000234001, LAB-000233409, LAB-000234287) are blocked by validation error
- Error message: "The product(s) that are on the opportunity are not servicable items and do not require a lab order"
- Three errors are identical; one (LAB-000234287) has related ticket HEAL-31389 requiring patient accommodation (swab kit)
- LAB-000233409 requests lab provider change (Novogenia instead of ASC)
- Error is triggered on Lab Order save, preventing legitimate lab orders from being created

**Symptoms (what the user reported):**
- Validation error blocks Lab Order creation for serviceable products
- No guidance on remediation or workaround
- Related ticket HEAL-31389 status unclear; patient accommodation needs not addressed

**Root cause hypothesis:**
Lab Order validation rule or flow logic determining serviceable status is either incorrectly configured, missing product-type mappings, or failing to account for legitimate lab order scenarios (such as patient accommodations or alternate lab provider selection).

---

## Smallest Viable Fix

**What to fix:**
- **Metadata/code artifact:** Validation rule on `Lab_Order__c` object (likely named something like "Check_Serviceable_Products" or similar) or Flow trigger on Lab Order creation
- **Change scope:** Audit serviceable product logic; either update rule exception/condition to allow these product types OR add missing product configurations OR modify Flow to provide remediation path
- **Why it solves the problem:** Removing the overly strict block allows legitimate lab orders to proceed; updated logic correctly identifies which products require lab orders vs. which do not

## Sandbox Validation Plan

**What to test in Sandbox:**
1. Create Lab Order for LAB-000234000 product, save, confirm no validation error
2. Create Lab Order for LAB-000234287 with swab kit option, confirm accommodation path works
3. Create Lab Order for LAB-000233409 with lab provider selection (Novogenia), confirm provider choice persists

**Expected outcome:**
Lab Order saves without validation error; all four scenario types complete successfully

**Success criteria:**
- All four Lab Order records save without error
- Swab kit option is available for patient-specific needs
- Lab provider selection is functional and persisted

## Rollback Plan

**If Sandbox test fails:**
1. Verify validation rule/flow logic against Production (read-only investigation)
2. If rule is too complex, escalate to Engineering
3. If missing config, add required product mappings and re-test

**If Production deployment fails:**
1. Revert rule/flow changes via Gearset
2. Notify reporter that fix requires further investigation
3. Review data differences between Sandbox and Production

---

## Risks and Constraints

**Implementation risks:**
- Removing validation could allow invalid Lab Orders to be created, affecting downstream processes
- Product serviceable logic may have cross-dependencies with Order generation or Lab vendor assignment

**Constraints:**
- Cannot modify external Lab provider APIs (Novogenia, ASC)
- Must maintain data integrity for existing Lab Orders
- Patient accommodation logic (swab kit) must integrate with existing case workflows

**Mitigation:**
- Audit all Opportunities and Lab Orders in Production before modifying rule
- Test negative cases in Sandbox (non-serviceable products should still error appropriately)
- Verify lab provider selection does not conflict with existing Order fulfillment logic
